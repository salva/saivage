# F33 r1 — Design

## Proposal A — Focused fix: single `seedProject` that writes both files, no default model

**Scope (files touched).**
- [src/store/project.ts](src/store/project.ts#L94-L124): `initProject` is unchanged in signature but its body now delegates the writing of *both* `config.json` and `saivage.json` to a new helper.
- New helper `seedProject(projectRoot, { name, objectives })` lives next to `initProject` in [src/store/project.ts](src/store/project.ts).
- [src/server/cli.ts](src/server/cli.ts#L32-L75): the inline `config` literal disappears; the CLI calls `seedProject(path, { name, objectives })` directly (or `initProject` which delegates to it — see Plan).
- [src/config.ts](src/config.ts#L204-L237): `writeDefaultConfig` is **deleted**. The canonical `saivage.json` seed lives in the new helper.
- [src/types.ts](src/types.ts#L13): `ProjectConfigSchema.provider` is **removed**. Model selection is system-level; per-project model overrides already exist via `model_overrides` ([src/types.ts](src/types.ts#L15)).
- [src/config.ts](src/config.ts#L49): the `.default({ orchestrator: "anthropic/claude-sonnet-4-20250514" })` on `models` is **removed**. `models` becomes `.default({})`. Consumers that read `models.orchestrator` must already handle the unset case (this is F04's territory; see "Cross-links" below).

**What gets added.**
- One canonical seeder: `seedProject(projectRoot, { name, objectives })`. It writes `config.json` (project identity + empty `model_overrides` + the notifications block at its new home — see below) and `saivage.json` (providers/agents/runtime/notifications), then calls `initProjectTree(projectRoot)`.
- The `notifications` block is removed from `ProjectConfigSchema` and lives only on `configSchema` (i.e., only in `saivage.json`). Single declaration, single writer, single reader. The schema defaults for `notifications` ([src/config.ts](src/config.ts#L106-L111)) become the only defaults.

**What gets removed.**
- `writeDefaultConfig` (dead code, plus it disagreed with the CLI).
- `ProjectConfigSchema.provider` (overlaps with `models.orchestrator`).
- The `models.default(...)` literal in `configSchema` (hardcoded model string).
- The `notifications` block on `ProjectConfigSchema`.
- The inline `config` object literal in `cli.ts initProject` action.

**Risk.**
- Medium. Touching `ProjectConfigSchema` invalidates any on-disk `config.json` that carries `provider` or `notifications` fields. Per project guideline #1, that is the intended outcome (no migration shim); existing projects in the workspace (`saivage-v3`, GetRich v2) will have their `config.json` regenerated by the operator. Listed under "Cross-issue ordering" in the plan.
- Low for the `saivage.json` change: schema parsing tolerates the absence of the removed `models.default`, because `.default({})` still produces a valid object.

**What it enables (cross-link).**
- **F02** (agent roster): with `seedProject` as the one place that knows about agents at bootstrap, the roster used in `models` keys becomes a single point that F02's fix can anchor on.
- **F04** (hardcoded models): F33A actively *removes* the hardcoded `anthropic/claude-sonnet-4-20250514` from the orchestrator-schema default, fulfilling part of the operator comment on F04 ("No model should be hard-coded ... the system must just fail to work and report the issue").
- **F32** (config blocks undocumented): consolidating `notifications` into one schema halves what F32 needs to document.

**What it forbids.**
- No "legacy `provider:`" support remains anywhere. Any consumer that read `ctx.config.provider` must read `models.orchestrator` from `SaivageConfig` instead — or, better, route through the existing routing resolver.
- No second writer of `saivage.json` may be reintroduced; `writeDefaultConfig` does not get exhumed.

**Recommendation note.** This is the lowest-risk fix that resolves the literal drift, deletes the dead code, and removes the worst hardcoded model string. It does not address the broader question of "why are there two config files at all", which Proposal B addresses.

## Proposal B — One level up: collapse `config.json` and `saivage.json` into a single seeded file

**Scope (files touched).**
- [src/types.ts](src/types.ts#L11-L45): `ProjectConfigSchema` is removed.
- [src/config.ts](src/config.ts#L34-L113): `configSchema` absorbs `project_name`, `objectives`, `model_overrides`, `routing`, `skills`, and `agents` — i.e., everything currently in `ProjectConfigSchema` except `provider` (deleted as in Proposal A) and `notifications` (already present in `configSchema`).
- [src/store/project.ts](src/store/project.ts): `loadProject` reads only `saivage.json`. `initProject` is renamed `seedProject`; it writes a single canonical `saivage.json` and calls `initProjectTree`.
- [src/server/cli.ts](src/server/cli.ts#L32-L75): `init` calls `seedProject(path, { name, objectives })`.
- Every call site that reads `ctx.config.<X>` for the consolidated fields is updated to read from the unified config (typically already accessed via the same `ProjectContext`, so this is mostly a type-name swap, not a logic change).
- `writeDefaultConfig` deleted as in Proposal A.

**What gets added.**
- A single `SaivageConfigSchema` that holds *both* project identity (`project_name`, `objectives`) and system settings (providers, models, runtime, notifications, mcpServers). Single file: `.saivage/saivage.json`. Single writer: `seedProject`. Single reader: `loadConfig` (which `loadProject` calls).

**What gets removed.**
- `ProjectConfigSchema` and `ProjectConfig` type.
- `config.json` as a file format — the file no longer exists in fresh projects.
- The `models.default(...)` literal in `configSchema`.
- The `notifications` block on `ProjectConfigSchema` (already only-here once `ProjectConfigSchema` is deleted).
- `writeDefaultConfig` (dead).
- `discoverProject` ([src/store/project.ts](src/store/project.ts#L49-L60)) and `resolveProjectRoot` ([src/config.ts](src/config.ts#L118-L135)) currently search for either `config.json` or `saivage.json`; they collapse into one rule ("look for `.saivage/saivage.json`").

**Risk.**
- High in absolute change-volume: every place that touches `ProjectConfig` or reads `loadProject(...).config` has to be updated. But the risk per touch is low because the field names overlap so heavily — most code already reads `ctx.config.project_name`, `ctx.config.objectives`, etc.; only the *schema* changes.
- Operator-facing: existing projects must regenerate `.saivage/saivage.json` and delete `.saivage/config.json`. Per project guideline #1, that is acceptable — no migration shim — but the operator needs explicit notice in the metaplan.

**What it enables.**
- **F02** (agent roster): one schema means one canonical agent roster.
- **F04** (hardcoded models): same removal as Proposal A; in addition, downstream code stops having to know which file holds the model setting.
- **F32** (config blocks undocumented): cuts F32's documentation work roughly in half: one schema, one block per concern.
- New invariant useful elsewhere: `loadProject` no longer has to triangulate two files, which removes a known caching pitfall in `loadConfig` (cached by `saivageDir`, not by individual file path).

**What it forbids.**
- No second file ("project config") reintroduced. No `provider` string at the project level.

**Recommendation note.** This is the architecturally correct destination, but it is a larger change touching many call sites. If F33 ships first, before F02/F04/F32 land, Proposal B forces all of them to revisit migrated code. If F33 ships *after* (or as part of) the metaplan ordering that has F02/F04 in front of it, Proposal B becomes cleaner because those issues have already settled the schemas it needs to merge.

## Recommendation

**Proposal A.**

Rationale:

1. **Scope discipline.** F33 was filed as a localized drift between two specific writers. Proposal A fully fixes that drift, deletes the dead `writeDefaultConfig`, removes the schema-level hardcoded model string, and consolidates `notifications` into one schema — without dragging the entire `ProjectConfigSchema` / `SaivageConfig` split into the change.
2. **Cross-issue ordering.** F02 (agent roster) and F32 (config blocks undocumented) are still in flight in this review batch. Proposal B's heavy refactor of `configSchema` would either pre-empt F02's roster decision or force F02 to chase a moving schema. Proposal A leaves the two-file split intact, so F02 and F32 can settle on their own without coordination.
3. **Architecture-first compliance.** Proposal A still deletes dead code (`writeDefaultConfig`), eliminates the worst hardcoded model string (the orchestrator default), and removes the `provider`/`notifications` duplication — i.e., it does not preserve "old + new" anywhere.
4. **Reversibility.** If, after F02/F04/F32 land, the project decides to escalate to Proposal B's full unification, Proposal A's `seedProject` is already the single bootstrap entry point. Promoting it to Proposal B becomes a follow-up rather than a rewrite.

Proposal B is the better long-term destination and should be filed as a follow-up issue once F02, F04, and F32 are approved — but it is not the right shape for F33's current scope.
