## Summary

The core direction is still right: Proposal B, the schema-driven seed, is the clean architecture fix. It removes the handwritten runtime-config literal in [src/store/project.ts#L136-L162](../../../../src/store/project.ts#L136-L162), brings the write under the same `writeDoc` discipline as project config in [src/store/project.ts#L133](../../../../src/store/project.ts#L133), and avoids compatibility shims.

Round 2 does address most of the four r1 requests on the page: it replaces the weak readDoc-only test with a raw-file comparison, expands the grep audit beyond Playwright, states that full schema-default materialization is intentional, and moves required validation to a local seed-focused CLI check. However, three r2 claims are still materially wrong or overbroad enough to mislead implementation.

## Required changes for r3

1. **Fix the security-sensitive claim that auth-bearing fields never live in saivage.json.** Round 2 says provider account API keys, Telegram bot-token values, and OAuth tokens never lived in saivage.json and instead live in auth-profiles only ([SPEC/v2/review-2026-05-round2/G08/01-analysis-r2.md#L51](./01-analysis-r2.md#L51)). The plan repeats this in the proposed memory update ([SPEC/v2/review-2026-05-round2/G08/03-plan-r2.md#L155](./03-plan-r2.md#L155)). That is not true for the actual schema: provider config allows `apiKey`, `baseUrl`, and `authProfile` in [src/config.ts#L15-L17](../../../../src/config.ts#L15-L17), `providers` is part of SaivageConfig in [src/config.ts#L76](../../../../src/config.ts#L76), and `telegram.botToken` is explicitly a saivage.json field in [src/config.ts#L130-L132](../../../../src/config.ts#L130-L132). Rewrite the analysis, design, and plan to distinguish the new seed's empty default (`providers: {}`, `telegram.botToken: ""`) from operator-owned existing saivage.json files that may contain secrets or auth-adjacent routing data. Remove the proposed memory update or make it preserve the existing operational rule: never assume saivage.json is non-sensitive, and preserve it during reset workflows unless the operator explicitly chooses to regenerate it.

2. **Correct the provider-default and transversal-impact audit.** Round 2 says no production consumer assumes a freshly seeded project has any provider or MCP server present ([SPEC/v2/review-2026-05-round2/G08/01-analysis-r2.md#L66](./01-analysis-r2.md#L66)) and frames the hardcoded localhost endpoints as removed by deleting the seeded provider entries ([SPEC/v2/review-2026-05-round2/G08/02-design-r2.md#L61](./02-design-r2.md#L61)). The seed-file part is true, but the system still has provider defaults outside the seed: the router registers Ollama unconditionally in [src/providers/router.ts#L731-L749](../../../../src/providers/router.ts#L731-L749), constructs it from config or provider defaults in [src/providers/router.ts#L804](../../../../src/providers/router.ts#L804), and Ollama still defaults to localhost in [src/providers/ollama.ts#L20-L36](../../../../src/providers/ollama.ts#L20-L36). Llama.cpp also still carries a localhost default when registered in [src/providers/llamacpp.ts#L10-L19](../../../../src/providers/llamacpp.ts#L10-L19). Those registered-provider choices surface through the HTTP API in [src/server/server.ts#L218-L226](../../../../src/server/server.ts#L218-L226), the CLI models command in [src/server/cli.ts#L291-L296](../../../../src/server/cli.ts#L291-L296), and bootstrap logging in [src/server/bootstrap.ts#L141](../../../../src/server/bootstrap.ts#L141). R3 should state the precise scope: G08 removes persisted provider policy from newly seeded saivage.json, not all provider availability or localhost default policy. Either add a follow-up/cross-link for router/provider defaults or extend the plan to change and test them. Do not leave the current "no production consumer" claim in place.

3. **Make the raw-default test match the claimed review contract.** R2 repeatedly says future schema default edits will fail tests until an expected snapshot is updated ([SPEC/v2/review-2026-05-round2/G08/01-analysis-r2.md#L46](./01-analysis-r2.md#L46), [SPEC/v2/review-2026-05-round2/G08/02-design-r2.md#L53](./02-design-r2.md#L53), [SPEC/v2/review-2026-05-round2/G08/02-design-r2.md#L91](./02-design-r2.md#L91)). But the proposed test compares raw JSON to `SaivageConfigSchema.parse({})` dynamically ([SPEC/v2/review-2026-05-round2/G08/03-plan-r2.md#L57-L61](./03-plan-r2.md#L57-L61)), while the producer is also `SaivageConfigSchema.parse({})` ([SPEC/v2/review-2026-05-round2/G08/03-plan-r2.md#L21-L22](./03-plan-r2.md#L21-L22)). A schema default change will therefore pass automatically; there is no snapshot to update. This test is still useful as a producer-output contract, but it does not review future default edits. R3 must either add an explicit raw default snapshot or field-by-field expected default assertions for the full materialized tree, or remove the snapshot/deliberate-review claim and say plainly that schema default changes are reviewed in the schema diff itself.

## R1 coverage

- R1 change 1, raw on-disk default contract: partially addressed. The raw comparison and empty `providers` / `mcpServers` assertions are good, but the deliberate-schema-default-review claim is not implemented.
- R1 change 2, provider stale-assumption audit: partially addressed. The grep patterns are better, but the audit stops at config-field references and misses router/provider defaults plus API/CLI surfaces.
- R1 change 3, full-default serialization: mostly addressed. The full materialized tree is documented, but the auth-bearing field explanation is wrong and needs correction before this is safe operational guidance.
- R1 change 4, validation target: addressed. The required validation now seeds a local project through the actual CLI path in [src/server/cli.ts#L35-L49](../../../../src/server/cli.ts#L35-L49), compares the raw file, calls `loadConfig(true, seedRoot)`, and makes live harness restart optional/operator-approved.

## Recommendation

Keep Proposal B. Do not revive Proposal A, do not preserve the handwritten literal, and do not add a compatibility seed. The remaining work is to make the documentation and test plan precise enough that the implementation cannot weaken secret-handling guidance, overstate provider-policy cleanup, or rely on a test that does less than it claims.
