# G28 — Review r1

## Verdict

Design B is the right direction: collapsing active plan state and completion history into a single persisted document is cleaner than adding a write-ahead sidecar, and it matches the project rule to remove the old structure instead of building compatibility machinery around it. The writer correctly verifies the root bug: `plan_complete_stage` writes `plan.json`, then `plan-history.json`, then updates the cache, so the two files are individually atomic but not atomically moved together ([src/mcp/plan-server.ts](../../../../src/mcp/plan-server.ts#L245-L252), [src/store/documents.ts](../../../../src/store/documents.ts#L75-L101)).

Changes are still required before approval.

## Required Changes

1. **Specify the full `PlanService` rewrite, not only `plan_complete_stage`.** The plan says to replace the two caches with `doc: PlanDocument | null`, but it only spells out `plan_get`, `plan_get_history`, `plan_complete_stage`, and `plan_commit` ([03-plan-r1.md](03-plan-r1.md#L19-L36)). The current service has five other active-plan mutators and readers that depend on `Plan`/`PlanHistory` split state: `plan_set_stages`, `plan_add_stage`, `plan_remove_stage`, `plan_set_current`, `plan_get_stage`, `plan_get_current_stage`, and `plan_init` ([src/mcp/plan-server.ts](../../../../src/mcp/plan-server.ts#L89-L194), [src/mcp/plan-server.ts](../../../../src/mcp/plan-server.ts#L264-L295)). The next plan must state how every one of these projects active-plan views, preserves `doc.history` on active-plan-only writes, initializes `history: []`, and handles `doc === null`. Without this, Option B can accidentally make the history array disappear on normal stage edits.

2. **Define the new view types and document invariants.** The design removes `PlanSchema`, `Plan`, `PlanHistorySchema`, and `PlanHistory` ([02-design-r1.md](02-design-r1.md#L135-L142)), but the public MCP/HTTP shapes still need typed projections. Add explicit names such as `ActivePlanView` and `PlanHistoryView`, or an equivalent local projection type, so callers are not left with untyped object literals after [src/index.ts](../../../../src/index.ts#L6-L12) stops exporting the old names. Also make the `PlanDocumentSchema` invariant part of the plan: `current_stage_id` must be `null` or an active stage id, active/history ids must not overlap, and malformed combined docs should hard-fail at `init()` as the analysis already recommends ([01-analysis-r1.md](01-analysis-r1.md#L140-L143)).

3. **Expand stale-reference and deletion coverage.** The implementation plan's final sweep only covers `src/` and `web/src/` ([03-plan-r1.md](03-plan-r1.md#L82-L109)), but the old model is referenced in committed source exports, tests, specs, operator docs, prompts, and generated API docs. At minimum, add [src/index.ts](../../../../src/index.ts#L6-L12), [src/agents/conversation-snapshot.test.ts](../../../../src/agents/conversation-snapshot.test.ts#L62), [SPEC/v2/01-DATA-MODEL.md](../../01-DATA-MODEL.md#L97-L102), [SPEC/v2/03-PLAN-MCP-SERVICE.md](../../03-PLAN-MCP-SERVICE.md#L7-L40), [SPEC/v2/05-MCP-SERVICES.md](../../05-MCP-SERVICES.md#L237), [docs/internals/plan-mcp.md](../../../../docs/internals/plan-mcp.md#L32-L40), [docs/internals/on-disk-layout.md](../../../../docs/internals/on-disk-layout.md#L13-L59), and the committed TypeDoc output under `docs/api/` if that tree is expected to stay current. The architecture-first requirement means the old schema/path should be removed from docs too, not only hidden from TypeScript compilation.

4. **Correct the G29 analysis and coordination.** The analysis says G29 makes the inter-write window worse by serialising all plan reads behind the queue ([01-analysis-r1.md](01-analysis-r1.md#L68-L73)). That is backwards for PlanService tool reads: `handleToolCall()` currently queues every tool through `serializeOp` ([src/mcp/plan-server.ts](../../../../src/mcp/plan-server.ts#L347-L357)), which hides in-flight writes from queued MCP reads. The real inconsistent-reader surface is the direct file readers in server/chat/handoff/shutdown paths, which bypass the PlanService cache. Keep the G29 coordination, but restate it precisely: if G29 lets reads bypass the queue before G28 lands, those reads should return a consistent cached snapshot; after G28 lands, the single-document model removes the cross-file disk window for non-PlanService readers.

5. **Tighten G27 coordination.** The plan treats G27 as a trivial `completedStage` field-source conflict ([03-plan-r1.md](03-plan-r1.md#L130-L135)), but G27's issue says the active `Stage` schema lacks any start timestamp and recommends tracking stage start state instead of synthesizing it ([../G27-plan-server-started-at-equals-completed-at.md](../G27-plan-server-started-at-equals-completed-at.md#L14-L31)). If G27 stores `started_at` on active stages, G28's new `PlanDocumentSchema` must include that revised `StageSchema` before history projection is finalized. The writer should specify a real landing order or merge strategy: either land G27 first and embed the new active-stage shape in `PlanDocument`, or land G28 with an explicit placeholder for G27's schema change.

6. **Make the atomicity tests executable and broader.** The proposed "stub `writeDoc` failure" test is directionally useful ([03-plan-r1.md](03-plan-r1.md#L77-L80)), but the plan should say how the imported ESM `writeDoc` will be mocked or injected. It should also add tests that each active-plan mutator preserves the existing `history` array, `plan_get_stage` finds both active and completed stages from one doc, no `.saivage/plan-history.json` is created by seed/init paths, `plan_commit` commits one path, and non-PlanService readers project both `/api/plan` and `/api/plan-history` from the same file. Those are the behavioral promises of Design B, not optional cleanup.

7. **Make live deployment instructions operational enough to be safe.** The plan covers the three affected daemons at a high level ([03-plan-r1.md](03-plan-r1.md#L146-L161)), but it needs exact per-host actions: stop `saivage.service` on `saivage` 10.0.3.111, `diedrico` 10.0.3.113, and `saivage-v3` 10.0.3.112; back up the project-local `.saivage/plan.json` and `.saivage/plan-history.json`; merge into the new single `plan.json` shape without printing secrets; restart; verify both `/api/plan` and `/api/plan-history`; and explicitly do not touch `saivage-v3-getrich-v2` 10.0.3.170. The no-migration-shim rule is fine, but the operator merge shape must be precise enough that an existing deployment does not boot into a schema parse failure.

## Notes

- Option A should remain rejected. It adds `pending-completion.json`, recovery state, and another path in [src/store/project.ts](../../../../src/store/project.ts#L30-L74), while leaving direct file readers exposed to half-applied snapshots. That is more machinery around the old model, not a cleaner model.
- Option B is not over-engineered if the next round treats `PlanDocument` as the single source of truth and deletes the old path everywhere. It becomes over-engineered only if it keeps compatibility projections, migration helpers, or stale docs that imply `plan-history.json` is still authoritative.

VERDICT: CHANGES_REQUESTED
