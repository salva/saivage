# G31 - Review r2

**Reviewer**: GPT-5.5
**Verdict**: CHANGES_REQUESTED

Round 2 fixes the seven concrete r1 issues in substance: the proposed read path uses `bytesRead`, tests now follow the `McpRuntime.callTool` thrown-error surface, the NUL-probe rule is explicit, EOF offsets are coherent, and the G30 dependency is correctly treated as a sequencing gate. I cannot approve yet because the design now promises an exhaustive structured-error contract while leaving ordinary filesystem failures from `stat` and `open` as raw thrown errors.

## Blocking Finding

1. **The structured-error contract is still overstated and can be violated by `stat` / `open` failures.** The r2 design says every error path carries a stable machine-readable `code` and lists the exhaustive set as `FILE_TOO_LARGE`, `LENGTH_TOO_LARGE`, `INVALID_RANGE`, `BINARY_CONTENT`, `NOT_A_FILE`, and `INVALID_ARGUMENT` in [SPEC/v2/review-2026-05-round2/G31/02-design-r2.md](02-design-r2.md#L357-L361). But the handler sketch calls `await stat(fp)` before any filesystem-error mapping in [SPEC/v2/review-2026-05-round2/G31/02-design-r2.md](02-design-r2.md#L194), and later calls `await open(fp, "r")` outside a `try/catch` in [SPEC/v2/review-2026-05-round2/G31/02-design-r2.md](02-design-r2.md#L247). A missing file, permission failure, or deletion between the `stat` snapshot and the later `open` will reject from the handler directly. `McpRuntime.callTool` only serializes structured content after a handler returns `isError: true` in [src/mcp/runtime.ts](../../../../src/mcp/runtime.ts#L188-L193); a raw rejection bypasses that documented code surface. This is not just wording: agents and tests are being taught to rely on embedded codes, while common `read_file` failures would still have no stable machine-readable code. Required change: either map `stat` / `open` failures into structured envelopes and add dedicated tests for the new documented code or codes, or explicitly narrow the design so it no longer claims every error path is covered exhaustively. The architecture-first version is to add structured handling, likely separating not-found from generic read/open failure.

## Required Round-3 Change

Add explicit structured handling for filesystem failures around the `stat` and `open` calls, or remove the exhaustive structured-error promise. If new codes are introduced, list them in the design and add dedicated runtime-level tests that assert them through `McpRuntime.callTool`'s thrown message. At minimum, cover a missing path; a deletion-between-`stat`-and-`open` race is also worth a focused implementation test if the handler structure makes it practical.
