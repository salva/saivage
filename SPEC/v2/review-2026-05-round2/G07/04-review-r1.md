# G07 - Review r1

**Reviewed**: [01-analysis-r1.md](./01-analysis-r1.md), [02-design-r1.md](./02-design-r1.md), [03-plan-r1.md](./03-plan-r1.md)
**Verdict**: CHANGES_REQUESTED
**Required change count**: 7

## Summary

Proposal B is the better architectural direction: the fallback should operate on protocol-valid conversation rounds, not on a raw suffix slice. However, the current design and plan are not ready to approve. The round selector is underspecified in a way that can either keep too little context or keep an over-threshold transcript forever, the fallback counter is not wired into any actual stop condition, and the runtime-state/config/deployment pieces are not grounded in the current code surfaces.

## Required changes

1. **Define a bounded fallback escape path.** The design says only successful summarizations count toward `maxCompactions` while fallback increments `summarizerFallbacks` ([02-design-r1.md](./02-design-r1.md#L70)), and the plan explicitly keeps `compactionCount` unchanged on fallback ([03-plan-r1.md](./03-plan-r1.md#L16-L23)). In the current loop, compaction is gated only by `compactionCount` before each LLM call ([src/agents/base.ts](../../../../src/agents/base.ts#L237-L248)) and again during model repair ([src/agents/base.ts](../../../../src/agents/base.ts#L540-L558)). If the fallback result remains over threshold, especially the single giant `ToolRound` case the design already acknowledges ([02-design-r1.md](./02-design-r1.md#L97)), the agent can repeatedly compact without making an LLM call and without ever tripping `maxCompactions`. Add an explicit consecutive-fallback cap, make `isMaxCompactionsReached` aware of fallback exhaustion, or prove and test that fallback always returns below the pre-call threshold except for a separately handled oversized-atomic-round failure.

2. **Replace the selector stopping rule with a real token-budget algorithm.** `selectKeptRounds` currently says to accumulate tail rounds until the kept count reaches the old 20 percent heuristic or the running token count "drops below" half the threshold ([02-design-r1.md](./02-design-r1.md#L62), [03-plan-r1.md](./03-plan-r1.md#L11)). Tail accumulation makes token count increase, not drop, so this is not implementable as written. It also omits tool schemas from the budget call, while the existing compaction accounting passes `tools` into `router.countTokens` ([src/runtime/compaction.ts](../../../../src/runtime/compaction.ts#L76-L77)) and BaseAgent includes tool schemas in static input tokens ([src/agents/base.ts](../../../../src/agents/base.ts#L204-L208)). Specify the target budget in terms of the next outbound request: system prompt plus tools plus flattened kept rounds. Then add rounds from the tail only while the projected total stays under that target, with one explicit oversized-round fallback case.

3. **Correct the analysis claim about trailing `tool_use` production.** The analysis says compaction can fire before the dispatcher writes the result or during model-repair before an assistant tool-use is paired ([01-analysis-r1.md](./01-analysis-r1.md#L53)). The cited runtime does not support that as a normal producer path: pre-call compaction happens before `callLLM` ([src/agents/base.ts](../../../../src/agents/base.ts#L237-L257)), and after a tool-calling response BaseAgent pushes the assistant tool-use message, runs the dispatcher, then pushes the user tool-result message in the same loop body ([src/agents/base.ts](../../../../src/agents/base.ts#L323-L343)). The model-repair path catches provider errors before any new assistant response is pushed into `messages` ([src/agents/base.ts](../../../../src/agents/base.ts#L540-L558)). Keep dangling assistant halves as a defensive invariant if desired, but revise the analysis so the proven current bug is leading orphan `tool_result` from the fallback slice.

4. **Specify or remove runtime-state exposure for `summarizerFallbacks`.** The design asks to surface the counter through runtime-state ([02-design-r1.md](./02-design-r1.md#L70-L78)), and the plan says to route it through the accessor that already exposes `compactionCount` ([03-plan-r1.md](./03-plan-r1.md#L25)). That accessor does not exist. `RuntimeTracker` snapshots only its internal `AgentState` values ([src/runtime/recovery.ts](../../../../src/runtime/recovery.ts#L331-L399)), `AgentStateSchema` has no compaction fields ([src/types.ts](../../../../src/types.ts#L240-L247)), and BaseAgent's `compactionState` is private ([src/agents/base.ts](../../../../src/agents/base.ts#L146)). Either drop runtime-state exposure from this finding and rely on diagnostics/tests, or define the actual API and schema changes: how BaseAgent publishes compaction status, how RuntimeTracker receives it, what JSON field name is used, and which tests prove the dashboard state is written.

5. **Resolve the fallback summarizer config surface before including it.** Step 8 points to a `SaivageConfig` schema in [src/types.ts](../../../../src/types.ts) ([03-plan-r1.md](./03-plan-r1.md#L27)), but `SaivageConfig` lives in [src/config.ts](../../../../src/config.ts#L62-L194), while [src/types.ts](../../../../src/types.ts#L12-L30) defines project-level `ProjectConfig`. BaseAgent currently hardcodes the summarizer to `ctx.modelSpec` ([src/agents/base.ts](../../../../src/agents/base.ts#L193-L195)). For this round, either remove optional `fallbackSummarizerSpec` from the mandatory proposal, or fully specify the runtime config key, routing validation, model coverage checks, bootstrap plumbing, and tests. Do not leave it as an optional hand-wave in an otherwise implementation-ready plan.

6. **Fix the deployment-validation contradiction.** The design caveat says changes propagate to all three v2-using containers and the plan must restart all three or none ([02-design-r1.md](./02-design-r1.md#L111)), while the plan's validation section says to restart only `saivage-v3` and explicitly not restart `saivage` or `diedrico` ([03-plan-r1.md](./03-plan-r1.md#L44-L51)). Pick one policy and make it consistent with the workspace handoff and operator approval requirements. The live fault-injection step should also avoid mutating provider routing or killing upstream providers unless the operator explicitly authorizes it; prefer a local integration harness or mocked router for the summarizer-failure proof.

7. **State why Proposal A is not acceptable, not merely a smaller B.** Proposal A's forward pass collects all `tool_use.id` values before filtering user messages ([02-design-r1.md](./02-design-r1.md#L20-L21)). That can preserve a `tool_result` whose matching `tool_use` appears later in the kept suffix, which still violates the provider invariant that the tool-use must precede the result. The backward pass has the same non-adjacent-pair weakness. The review should explicitly reject A on order/adjacency correctness grounds, then keep B as the chosen direction with tests for ordering, partial multi-tool matches, duplicate ids, extra results, text-only rounds, and oversized atomic rounds.

## Design A vs B

B should remain the recommended design after revision. A preserves the raw-suffix mental model and tries to clean up after it; worse, the proposed Set-based cleanup is not order-aware. B's round parser matches the real protocol boundary and gives the fallback a structurally valid unit of selection. The acceptance condition should be: B emits only text rounds and complete adjacent tool rounds, never emits a dangling half, and either fits the next outbound request budget or exits through a bounded fallback-failure path.

## Testability requirements

Add direct tests for `parseRounds`, `selectKeptRounds`, and `compactConversation` fallback. The selector tests must assert both protocol validity and budget behavior using `systemPrompt` plus tools, not only message tokens. Add a BaseAgent integration test that forces the summarizer call to throw and asserts that `getConversationSnapshot()` contains the fallback diagnostic and no orphan `tool_result`; diagnostics are exposed through the snapshot surface ([src/agents/base.ts](../../../../src/agents/base.ts#L363-L461)). Add one test for fallback exhaustion so the new counter cannot become an infinite pre-call compaction loop.

## Cross-finding notes

The F13 relationship is correct: typed `orphaned_tool_result` classification remains the consumer-side last line of defense ([src/providers/error.ts](../../../../src/providers/error.ts#L11-L161)). G06 and G29 are plausible latency amplifiers, but G07 should not be ordered behind G06 as a hard dependency. The fallback must be correct under any summarizer failure, regardless of why the summarizer failed.

VERDICT: CHANGES_REQUESTED
