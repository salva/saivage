# G11 - Review r1

Design B is the right architectural direction: delete the free-text restart heuristic and keep Planner restart behind the explicit local slash-command registry. That matches the project rule against preserving obsolete control paths, and it is cleaner than expanding a multilingual regex.

I am requesting changes because r1 misses one real cross-surface impact, proposes a regression test at the wrong layer, and leaves incorrect regex semantics in the analysis.

## Findings

1. Prompt impact is incorrectly treated as optional. The design says the Chat prompt nudge can be deferred at [SPEC/v2/review-2026-05-round2/G11/02-design-r1.md#L74](SPEC/v2/review-2026-05-round2/G11/02-design-r1.md#L74), and later says no prompt changes are required at [SPEC/v2/review-2026-05-round2/G11/02-design-r1.md#L136](SPEC/v2/review-2026-05-round2/G11/02-design-r1.md#L136). The actual prompt still tells Chat that it can restart the Planner at [prompts/chat.md#L7](prompts/chat.md#L7), can request a Planner restart at [prompts/chat.md#L33](prompts/chat.md#L33), should use a deterministic restart path or tell the user to use the slash command at [prompts/chat.md#L43](prompts/chat.md#L43), and should treat free-text Planner restart requests as an interrupt path at [prompts/chat.md#L51](prompts/chat.md#L51). After the code deletion, free text flows from slash-command handling directly into the LLM path at [src/agents/chat.ts#L197-L213](src/agents/chat.ts#L197-L213), while the actual restart implementation remains the local command handler at [src/chat/localCommands.ts#L57-L65](src/chat/localCommands.ts#L57-L65) and [src/chat/localCommands.ts#L137-L158](src/chat/localCommands.ts#L137-L158). Leaving the prompt unchanged makes the LLM contract ambiguous and may cause it to claim it restarted the Planner or create a note for an action that is now intentionally slash-command-only.

2. The proposed regression test does not test the bug. The plan adds cases to [src/chat/localCommands.test.ts](src/chat/localCommands.test.ts) by calling `dispatchLocalCommand` on non-slash text at [SPEC/v2/review-2026-05-round2/G11/03-plan-r1.md#L35-L55](SPEC/v2/review-2026-05-round2/G11/03-plan-r1.md#L35-L55). But `dispatchLocalCommand` already returns `null` for all non-slash content at [src/chat/localCommands.ts#L86-L97](src/chat/localCommands.ts#L86-L97), and the bug sits in `ChatAgent` after `tryHandleCommand` returns and before `injectMessage` at [src/agents/chat.ts#L188-L206](src/agents/chat.ts#L188-L206). The proposed test would still pass if `tryHandleExplicitPlannerRestart` and its call site were accidentally left in place. A ChatAgent-level test is feasible: [src/agents/agents.test.ts#L227-L271](src/agents/agents.test.ts#L227-L271) already drives a real `ChatAgent` through a fake channel and router, and the helper channel exists at [src/agents/agents.test.ts#L626-L655](src/agents/agents.test.ts#L626-L655). The r2 plan should require a test proving free text such as `Why did the planner restart yesterday?` reaches the router and does not call `plannerControl.requestRestart` or publish a restart event.

3. The analysis overstates the current regex behavior. The source pattern is exact-word matching at [src/agents/chat.ts#L352-L355](src/agents/chat.ts#L352-L355), but the analysis says `\b(restart|reset|relaunch)\b` matches suffixes such as `Relaunched` and `Restarting`, and that `\bplanner\b` matches `planners`, at [SPEC/v2/review-2026-05-round2/G11/01-analysis-r1.md#L31-L32](SPEC/v2/review-2026-05-round2/G11/01-analysis-r1.md#L31-L32). Those examples are false for JavaScript word-boundary regexes. Proposal A also says the `u` flag makes `\b` behave correctly on non-ASCII letters at [SPEC/v2/review-2026-05-round2/G11/02-design-r1.md#L23](SPEC/v2/review-2026-05-round2/G11/02-design-r1.md#L23), which is not a sound premise for a multilingual word-boundary detector. Proposal A can remain as the rejected focused alternative, but r2 should not leave technically false claims in the analysis/design.

4. The transversal impact should be widened from source-only to source plus prompt plus tests. The r1 source edit list is correct for [src/agents/chat.ts](src/agents/chat.ts), and the command registry evidence is correct at [src/agents/conventions.ts#L73-L84](src/agents/conventions.ts#L73-L84). However, the plan only treats prompt references as a late optional checklist item at [SPEC/v2/review-2026-05-round2/G11/03-plan-r1.md#L99](SPEC/v2/review-2026-05-round2/G11/03-plan-r1.md#L99), even though [prompts/chat.md#L33-L51](prompts/chat.md#L33-L51) is part of the runtime behavior after deleting the code shortcut.

## Required Changes

1. Keep Proposal B as the recommendation, but update the design and plan so [prompts/chat.md](prompts/chat.md) is edited in the same change: natural-language restart requests must be answered by directing the user to `/restart-planner <reason>`, not by claiming Chat can restart the Planner or by relaying a restart note.

2. Replace the localCommands-only negative regression with a ChatAgent-level regression using the existing fake channel/router harness. The test must fail against the current bug and pass only when the post-command fuzzy restart branch is gone.

3. Correct the regex semantics in the analysis and in rejected Proposal A: remove claims that the current `\b...\b` pattern matches suffixes/plurals, and do not claim the `u` flag makes JavaScript `\b` a Unicode-aware word boundary.

4. Update the transversal-impact and validation sections to include the prompt edit and the ChatAgent regression test as first-class required work, not optional observation after implementation.
