# G26 - Review (round 4, GPT-5.5)

## Findings

1. The narrow-rejection pivot is the right scope, but the proposed schema mechanics do not preserve current unknown-key behavior. Round 3 allowed the design to reject only the legacy key if it preserved current unknown-key behavior ([SPEC/v2/review-2026-05-round2/G26/04-review-r3.md](04-review-r3.md#L5)). Round 4's inventory justifies not using `.strict()` because the known configs carry unrelated `notifications` and `provider` keys ([SPEC/v2/review-2026-05-round2/G26/01-analysis-r4.md](01-analysis-r4.md#L200-L218)), and the analysis even states those other unknown keys remain "silently stripped, exactly as today" ([SPEC/v2/review-2026-05-round2/G26/01-analysis-r4.md](01-analysis-r4.md#L266-L267)). But the actual design and plan use `.passthrough()` ([SPEC/v2/review-2026-05-round2/G26/02-design-r4.md](02-design-r4.md#L59-L63), [SPEC/v2/review-2026-05-round2/G26/03-plan-r4.md](03-plan-r4.md#L129-L156)), and the design says this keeps `notifications`, `provider`, and future fields on the parsed object ([SPEC/v2/review-2026-05-round2/G26/02-design-r4.md](02-design-r4.md#L102-L104)). That is a behavior change from the live plain `z.object` schema ([src/types.ts](../../../../src/types.ts#L12-L29)): `readDoc` returns `schema.parse(data)` ([src/store/documents.ts](../../../../src/store/documents.ts#L20-L23)), `loadProject` stores that parsed result as `project.config` ([src/store/project.ts](../../../../src/store/project.ts#L66-L70)), and `/api/debug/state` returns `runtime.project.config` ([src/server/server.ts](../../../../src/server/server.ts#L476-L498)). So the current `notifications`/`provider` inventory would become newly visible in loaded project config and the debug API, and any future `writeDoc` of project config would serialize the parsed pass-through output ([src/store/documents.ts](../../../../src/store/documents.ts#L75-L81)). Required change: keep the targeted `model_overrides` rejection, but implement it with a raw-key guard that still returns the stripped schema output for all other unknown keys, or explicitly widen the proposal to include a pass-through exposure audit and tests. For G26, preserving strip semantics is the cleaner fix.

## Required Change Coverage

Round-3 blocker 1 is only partially addressed. The new inventory now correctly enumerates the broader-key `.strict()` impact and shows why `.strict()` would couple G26 to unrelated `notifications` and `provider` cleanup ([SPEC/v2/review-2026-05-round2/G26/01-analysis-r4.md](01-analysis-r4.md#L200-L218)). The architectural pivot to targeted rejection is justified by that inventory. The other refreshed resolver anchors also match the live file: source union at [src/routing/resolver.ts](../../../../src/routing/resolver.ts#L66-L77), `resolvePreferredModels` at [src/routing/resolver.ts](../../../../src/routing/resolver.ts#L204-L220), runtime/legacy helpers at [src/routing/resolver.ts](../../../../src/routing/resolver.ts#L247-L262), and `resolveSource` at [src/routing/resolver.ts](../../../../src/routing/resolver.ts#L264-L269).

## What Holds

The core resolver and config direction still holds: delete `model_overrides` from the schema, stop seeding it, remove `resolveLegacyModels`, narrow `ResolvedModelRoute.source` to `"routing" | "runtime-default"`, replace the legacy resolver test, and remove guide references. The hard same-commit constraint for schema plus seeder remains necessary because the seeder currently writes the rejected key ([src/store/project.ts](../../../../src/store/project.ts#L125-L133)).

The round-1 grep-gate concern remains materially addressed in the plan: the schema rejection test builds the legacy key at runtime and the final gates are scoped rather than tree-wide ([SPEC/v2/review-2026-05-round2/G26/03-plan-r4.md](03-plan-r4.md#L393-L452), [SPEC/v2/review-2026-05-round2/G26/03-plan-r4.md](03-plan-r4.md#L507-L533)). While revising Task 1, collapse the self-correcting schema snippets into one final implementation so the design and plan do not show an obsolete [src/types.ts](../../../../src/types.ts#L12-L29) snippet that violates their own grep gate ([SPEC/v2/review-2026-05-round2/G26/02-design-r4.md](02-design-r4.md#L82-L91), [SPEC/v2/review-2026-05-round2/G26/03-plan-r4.md](03-plan-r4.md#L155-L223)).

## Required Changes

1. Replace the `.passthrough()` output contract with an implementation that rejects only the raw `model_overrides` key while preserving current strip semantics for all other unknown top-level keys. Add a positive regression proving an otherwise valid config with `notifications` or another unrelated unknown key succeeds and does not expose that key in `ProjectConfigSchema.parse(...)` output.
2. Update the daemon-impact language to say the known configs still load and their unrelated unknown keys remain stripped from parsed project config, not passed through.
3. Remove the stale literal-key schema snippets from the design/plan or clearly mark them as rejected alternatives, leaving only the runtime-built-key implementation that satisfies the grep gates.
