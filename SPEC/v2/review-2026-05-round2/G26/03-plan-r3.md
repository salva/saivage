## Hard ordering constraint: Task 1 + Task 3 ship together

Per [02-design-r3.md](02-design-r3.md#ordering-constraint-schema-and-seeder),
Task 1 (schema strict-mode + field deletion) and Task 3 (seeder field
deletion) must land in the same git commit / same patch. They are
not separable. If Task 1 lands before Task 3, every project seeded in
the gap fails strict load on the next restart. If Task 3 lands
before Task 1, the change is a no-op until Task 1 lands. Reviewers
must reject any sequencing that splits Task 1 and Task 3 across
commits.

The remaining tasks (resolver body, tests, docs, validation) may be
in the same commit or in follow-on commits within the same PR, as
none of them produces or consumes the `model_overrides` key on disk.

## Grep-gate convention (reviewer concern 1 from round 1)

All grep gates use one of two scopes, never a single workspace-wide
scope that would catch the new schema-rejection test:

1. Production-source scope (excludes test files):
   ```bash
   grep -rn 'model_overrides' src/ --include='*.ts' --exclude='*.test.ts'
   ```
2. Per-file scope (named files only):
   ```bash
   grep -n 'model_overrides' src/routing/resolver.test.ts
   ```

The new schema-rejection test in
[src/types.test.ts](../../../../src/types.test.ts) is the only test
file that intentionally references the legacy key, and it does so via
a runtime-built string. Neither gate scope can fire on it.

The same scoping applies to `"legacy"` token gates: scope to
non-test source paths and add an explicit per-file gate for the
resolver test.

```ts
private resolveRuntimeDefaultModels(role: string): string[] {
  if (role === "supervisor") {
    const models = normalizeModelList(this.runtime.supervisorModel);
    if (models.length) return models;
    throw new MissingModelForRoleError([role], configPath());
  }
  if (role === "security") {
    const models = normalizeModelList(this.runtime.securityModel);
    if (models.length) return models;
    throw new MissingModelForRoleError([role], configPath());
  }
  const key = ROUTING_ROLE_TO_MODEL_KEY[role] ?? role;
  const models = normalizeModelList(this.runtime.models?.[key] ?? this.runtime.models?.default);
  if (models.length) return models;
  throw new MissingModelForRoleError([role], configPath());
}
```

```ts
// Legacy v1 routing key built at runtime so the bareword does not

const LEGACY_KEY = ["model", "overrides"].join("_");

describe("ProjectConfigSchema (G26 strict-mode rejection)", () => {
  it("rejects the pre-v2 legacy routing key at top level", () => {
    const fixture: Record<string, unknown> = {
      project_name: "x",
      objectives: [],
      routing: { roles: {}, profiles: {} },
      skills: { max_per_agent: 5 },

      objectives: [],
      routing: { roles: {}, profiles: {} },
      skills: { max_per_agent: 5 },
    };
    fixture[LEGACY_KEY] = {};

    expect(ProjectConfigSchema.safeParse(fixture).success).toBe(false);
  });
});
```

inventory in
[02-design-r3.md](02-design-r3.md#daemon-impact-concrete-on-disk-inventory)

output is operator-investigated before the restart proceeds.

(not directory-wide), so it cannot return false positives from
unrelated state files. Workspace-mirror paths under
