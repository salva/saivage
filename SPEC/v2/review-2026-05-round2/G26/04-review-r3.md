## Findings

1. The daemon-impact inventory is still not accurate for the proposed strict schema. Round 3 correctly narrowed the inventory to concrete .saivage/config.json paths, but it only checks whether the legacy key is present, while the design and plan explicitly propose `.strict()` so any unknown top-level key fails load, not just `model_overrides` ([SPEC/v2/review-2026-05-round2/G26/02-design-r3.md](02-design-r3.md#L58-L60), [SPEC/v2/review-2026-05-round2/G26/03-plan-r3.md](03-plan-r3.md#L115-L127)). The live project schema currently recognizes only project_name, objectives, model_overrides, routing, skills, and agents at top level ([src/types.ts](../../../../src/types.ts#L13-L29)). A key-name-only audit of the same known configs found existing unknown top-level keys today: saivage-v3, diedrico, and /opt/saivage-v3 carry notifications; getrich carries provider and notifications. Therefore the rows marked clean, loads and the zero-impact conclusion are false for the actual proposed `.strict()` behavior ([SPEC/v2/review-2026-05-round2/G26/01-analysis-r3.md](01-analysis-r3.md#L160-L173), [SPEC/v2/review-2026-05-round2/G26/02-design-r3.md](02-design-r3.md#L405-L418)). Task 7 repeats the same bug by checking only model_overrides and expecting zero hits ([SPEC/v2/review-2026-05-round2/G26/03-plan-r3.md](03-plan-r3.md#L446-L484)). Required change: either make the design intentionally reject only the legacy key while preserving current unknown-key behavior, or keep `.strict()` and replace the inventory plus Task 7 with an actual strict-load/top-level-key audit that enumerates every currently failing config and the operator-gated manual cleanup needed before restart. Do not add an auto-stripper or migration shim.

Round 3 repeatedly says the `resolve()` fallback call is at L108, and even says L108 is correct rather than L110 ([SPEC/v2/review-2026-05-round2/G26/01-analysis-r3.md](01-analysis-r3.md#L205-L207), [SPEC/v2/review-2026-05-round2/G26/02-design-r3.md](02-design-r3.md#L99-L113), [SPEC/v2/review-2026-05-round2/G26/03-plan-r3.md](03-plan-r3.md#L161-L164)). This is smaller than Finding 1, but it fails the line-anchor freshness gate and should be fixed wherever the L108 claim appears.
