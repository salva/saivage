# G29 - Review r1

## Findings

1. The sequencing and safety proof need to be aligned with the approved G28 plan. The reader/writer split itself is mostly correct for the current 11-tool surface: the read cases are `plan_get`, `plan_get_stage`, `plan_get_current_stage`, and `plan_get_history` in [src/mcp/plan-server.ts](../../../../src/mcp/plan-server.ts#L364-L392), while the mutating cases are `plan_set_stages`, `plan_add_stage`, `plan_remove_stage`, `plan_set_current`, `plan_complete_stage`, `plan_init`, and `plan_commit` in [src/mcp/plan-server.ts](../../../../src/mcp/plan-server.ts#L373-L400). That also covers the G27-affected paths. The problem is that round 1 still says G29 may land independently of G27/G28 in [SPEC/v2/review-2026-05-round2/G29/01-analysis-r1.md](01-analysis-r1.md#L67) and says landing before G28 is safe in [SPEC/v2/review-2026-05-round2/G29/02-design-r1.md](02-design-r1.md#L113-L115). The approved G28 coordination says the opposite: G28 lands before G29, and G29 must be re-read to ensure it no longer depends on the old two-cache split in [SPEC/v2/review-2026-05-round2/G28/03-plan-r2.md](../G28/03-plan-r2.md#L294-L299). This is not just administrative. Current live `plan_complete_stage` still has two awaited disk writes before the cache commit in [src/mcp/plan-server.ts](../../../../src/mcp/plan-server.ts#L249-L252), while G28 collapses that to one `PlanDocument` write in [SPEC/v2/review-2026-05-round2/G28/03-plan-r2.md](../G28/03-plan-r2.md#L65-L119). Revise r2 so the proof says in-process MCP reads are safe because they read cache snapshots only, and make G27 -> G28 -> G29 the required implementation sequence unless G28 is explicitly re-approved otherwise.

2. The drift-guard design is not implementable as written and should be tied to the public tool registry. The plan adds `PLAN_WRITER_TOOLS` as an unexported module constant in [SPEC/v2/review-2026-05-round2/G29/03-plan-r1.md](03-plan-r1.md#L35-L44), then asks the test to import it in [SPEC/v2/review-2026-05-round2/G29/03-plan-r1.md](03-plan-r1.md#L80-L81). That will not compile unless the Set is exported or exposed through a deliberate static member. The design also says switch coverage forces Set updates in [SPEC/v2/review-2026-05-round2/G29/02-design-r1.md](02-design-r1.md#L45-L46), but the live switch is keyed by `string`, so TypeScript gives no exhaustiveness guard in [src/mcp/plan-server.ts](../../../../src/mcp/plan-server.ts#L360-L405). R2 should specify a small, explicit surface: export `PLAN_WRITER_TOOLS`, define `PLAN_READER_TOOLS`, and add a test that compares their disjoint union against `PlanService.getToolSchemas().map(({ name }) => name)` from [src/mcp/plan-server.ts](../../../../src/mcp/plan-server.ts#L414-L502). Unknown tools can still fall through to validation error, but every registered schema must be classified.

3. The non-blocking read test is timing-based and can be made deterministic. The plan says to race `plan_get` against a 50 ms timeout in [SPEC/v2/review-2026-05-round2/G29/03-plan-r1.md](03-plan-r1.md#L66-L72). That is unnecessarily flaky for a concurrency invariant already expressible with the existing `deferred` helper in [src/runtime/runtime.test.ts](../../../../src/runtime/runtime.test.ts#L100-L112). R2 should require a microtask-based assertion instead: start a gated `plan_commit`, issue `handleToolCall("plan_get", {})`, let the microtask queue drain, assert the read settled while the commit did not, then resolve the gate and await both promises for cleanup. Keep the post-write read test, but make it use `handleToolCall` for both the writer and reader so it verifies the dispatch boundary changed in [src/mcp/plan-server.ts](../../../../src/mcp/plan-server.ts#L343-L347), not just the direct methods.

4. Several source anchors in the analysis are stale and should be refreshed before approval. For example, the reader-method evidence in [SPEC/v2/review-2026-05-round2/G29/01-analysis-r1.md](01-analysis-r1.md#L25-L28) points at old live-code locations; the current anchors are `plan_get` at [src/mcp/plan-server.ts](../../../../src/mcp/plan-server.ts#L90-L93), `plan_get_stage` at [src/mcp/plan-server.ts](../../../../src/mcp/plan-server.ts#L96-L104), `plan_get_current_stage` at [src/mcp/plan-server.ts](../../../../src/mcp/plan-server.ts#L107-L111), and `plan_get_history` at [src/mcp/plan-server.ts](../../../../src/mcp/plan-server.ts#L265-L270). The cross-document write/cache-commit evidence should point at [src/mcp/plan-server.ts](../../../../src/mcp/plan-server.ts#L249-L252), not the older lines cited in [SPEC/v2/review-2026-05-round2/G29/01-analysis-r1.md](01-analysis-r1.md#L36-L67). The runtime test anchors for the replaced F34 test are current in [src/runtime/runtime.test.ts](../../../../src/runtime/runtime.test.ts#L653-L670).

## What Is Sound

- The proposed writer set includes every existing mutating tool, including the G27 paths `plan_set_stages`, `plan_set_current`, and `plan_complete_stage`.
- The proposed reader set is complete for the current registered schemas and reads cache only, so bypassing the queue does not expose torn in-memory state.
- Keeping `serializeOp` around writers preserves the writer/write ordering invariant already covered by [src/runtime/runtime.test.ts](../../../../src/runtime/runtime.test.ts#L579-L611), and that still composes with G28's single-write `PlanDocument` invariant.

VERDICT: CHANGES_REQUESTED
