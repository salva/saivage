## Blocking findings

1. **Header fast-fail paths still leave the response body alive.** The proposed bounded readers do the right thing once streaming has started: the text reader keeps only the prefix that fits and calls `reader.cancel()` on overflow [02-design-r1.md](02-design-r1.md#L164-L175), while the binary reader refuses the over-limit chunk and cancels [02-design-r1.md](02-design-r1.md#L201-L208). But the `Content-Length` fast-fail path returns immediately after deciding the body is too large [02-design-r1.md](02-design-r1.md#L454-L459), with no `response.body?.cancel()` or drain. That is already a cap-exceeded decision, so it violates the requirement that the socket be drained or aborted as soon as the cap is exceeded. The same early-return leak exists on upstream HTTP errors in `downloadUrl` [02-design-r1.md](02-design-r1.md#L449-L451) and in `fetch_url`/`fetch_page_text` [02-design-r1.md](02-design-r1.md#L355-L363). Require an explicit cancel/drain helper on every early response exit after headers arrive, and add a test where `Content-Length` exceeds `max_bytes` while the server keeps writing unless the client closes.

2. **Mid-body timeouts will be classified as `NETWORK_ERROR`, not `TIMEOUT`.** `fetchWithTimeout` creates an internal `AbortSignal.timeout` and passes it only to `fetch` [02-design-r1.md](02-design-r1.md#L131-L141), but neither bounded reader receives the signal [02-design-r1.md](02-design-r1.md#L151-L154) [02-design-r1.md](02-design-r1.md#L188-L191). The classifier maps only `Error.name === "TimeoutError"` to `TIMEOUT` [02-design-r1.md](02-design-r1.md#L222-L231). Once headers have arrived, a timeout during `reader.read()` will normally surface as an abort/terminated body-read error, so the current design falls through to `NETWORK_ERROR`. G33 r2 already identified this exact class of bug and fixes it by checking the controller state in the body-read catch [../G33/02-design-r2.md](../G33/02-design-r2.md#L26-L67). G34 should do the same in the shared helper: either return the timeout signal/controller from `fetchWithTimeout`, or have `fetchWithTimeout` own both fetch and bounded read so `classifyNetworkError` can distinguish timeout aborts from real network resets. Add a mid-body timeout test, not only a pre-headers stall.

3. **The download handlers cannot emit the structured envelopes promised by the design.** The design records failure classification only as an `attempt.error` string and returns `null` [02-design-r1.md](02-design-r1.md#L437-L440) [02-design-r1.md](02-design-r1.md#L457-L473), and the plan explicitly says the `download_file` and `download_with_fallbacks` case bodies do not change beyond consuming that text [03-plan-r1.md](03-plan-r1.md#L91-L97). In the live handler, that means the final direct-download error is still just `{ error: "Download failed", url, attempts }` [src/mcp/builtins.ts](../../../../src/mcp/builtins.ts#L838-L842), while `DownloadAttempt` has no `code` field [src/mcp/builtins.ts](../../../../src/mcp/builtins.ts#L104-L111). This contradicts the test gates that expect top-level `code: "RESPONSE_TOO_LARGE"`, `code: "UPSTREAM_HTTP_ERROR"`, and `code: "NETWORK_ERROR"` [03-plan-r1.md](03-plan-r1.md#L167-L173). Make `downloadUrl` return a typed success/failure result, or throw/return a classified failure envelope, and propagate a top-level `code` from both download handlers. Attempts should carry `code`, `error`, and relevant `status`/`errno` too. Also implement the claimed `IO_ERROR` branch for local writes; the live file write is still a raw `mkdirSync`/`writeFileSync` sequence [src/mcp/builtins.ts](../../../../src/mcp/builtins.ts#L226-L227), while the design only lists `IO_ERROR` in the table [02-design-r1.md](02-design-r1.md#L480-L489).

4. **`maxFetchChars` parity is left ambiguous, and `fetch_page_text` changes semantics more than the design admits.** The live `fetch_url` path treats `max_chars` as a returned string cap after decoding [src/mcp/builtins.ts](../../../../src/mcp/builtins.ts#L762-L767), and `fetch_page_text` strips HTML before slicing returned text [src/mcp/builtins.ts](../../../../src/mcp/builtins.ts#L793-L799). Round 1 reuses `maxFetchChars` as a raw byte cap [02-design-r1.md](02-design-r1.md#L284-L294), then says exact character parity can be restored later if requested [02-design-r1.md](02-design-r1.md#L296-L300). That is not enough for this finding: the cap should be a memory/bandwidth guard and the returned `max_chars` contract should be explicit. For `fetch_page_text`, capping raw HTML before `stripHtml` can truncate away visible content even when the stripped text would be under the requested character limit [02-design-r1.md](02-design-r1.md#L410-L416). Either preserve `max_chars` as the returned text cap with a separate bounded upstream byte budget, or deliberately rename/document the public contract and add non-ASCII plus markup-heavy tests. The config defaults for the existing caps are correctly identified in the live schema [src/config.ts](../../../../src/config.ts#L137-L145); the missing piece is a precise contract for how those caps map to stream bytes and returned characters.

## Required corrections

- **Helper ownership stance: G34 should own the shared HTTP module.** I agree with G34's direction: the proposed shared HTTP helper module is the natural home for `readBoundedTextBody`, `readBoundedBinaryBody`, `fetchWithTimeout`, and network classification because G34 needs both text and binary readers plus the timeout wrapper [02-design-r1.md](02-design-r1.md#L12-L19). G33 r2 currently claims the opposite, exporting `readBoundedTextBody` from the builtins module and making G34 consume it [../G33/02-design-r2.md](../G33/02-design-r2.md#L193-L208). That design should give. G33 should depend on G34, delete its builtins-local helper/export, and import the shared text reader from G34's shared module. G34's plan should not leave this as a loose out-of-scope note [03-plan-r1.md](03-plan-r1.md#L154-L158); it should make the sequencing dependency explicit.

- **The plan anchors are stale in several places.** Step 5 links to [02-design-r1.md](02-design-r1.md#L184), which is inside the binary reader section, not the `fetch_url` replacement; the actual `fetch_url` section starts at [02-design-r1.md](02-design-r1.md#L321-L324). Step 6 links to [02-design-r1.md](02-design-r1.md#L255), which is the classifier return, not `fetch_page_text`; that section starts at [02-design-r1.md](02-design-r1.md#L410-L416). Step 7 links to [02-design-r1.md](02-design-r1.md#L264), which is the config schema section, not `downloadUrl`; the `downloadUrl` section starts at [02-design-r1.md](02-design-r1.md#L418-L421). The live `downloadUrl` helper also starts at [src/mcp/builtins.ts](../../../../src/mcp/builtins.ts#L162-L172), not [src/mcp/builtins.ts](../../../../src/mcp/builtins.ts#L138-L235) as stated in the plan [03-plan-r1.md](03-plan-r1.md#L86-L97) and files-touched list [03-plan-r1.md](03-plan-r1.md#L141-L146). Fix these before implementation so the same-file edits do not drift into the prompt-injection scan helpers.

- **Same-file coordination is mostly clean once the G33 conflict is resolved.** G30 is orthogonal. G31 provides the structured-envelope precedent but G34 should not depend on `classifyFsError` except for any local filesystem write error handling it actually implements. G32 is spatially separate around `search_files`. G33 is the real coordination point because it currently modifies the same import/export surface and claims the same helper. No G35-specific semantic conflict is visible from the supplied inputs; treat it as standard rebase coordination unless its design also touches the fetch/download cases.

## Summary

The architectural direction is good: a shared HTTP helper module is the right owner, and the text/binary streaming loops are close to the correct cap behavior. Round 2 needs to tighten timeout propagation through body reads, cancel every early response exit, make download failures structurally typed all the way out to the tool result, and settle the `maxFetchChars` byte-vs-character contract with tests.
