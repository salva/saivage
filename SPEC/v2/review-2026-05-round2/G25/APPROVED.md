# G25 — APPROVED

**Chosen proposal**: Proposal A (per [02-design-r4.md](02-design-r4.md)) — distinguish "no candidates" from "candidates filtered to nothing" in `resolvePreferredModels`. Introduce a typed `NoAllowedRouteMatchError` carrying `kind`, `role`, `candidates`, `allowed`, and `configPath` (built from the global `configPath()` helper at the resolver throw site). When the rule-derived candidate set is non-empty but the `allowed_models` intersection is empty, throw the typed error instead of returning `[...allowed]`. Apply the symmetric fix to `allowed_accounts`, treating provider `defaultAccount` as part of the candidate set when no explicit account is supplied, and add the same allowed_accounts-only fallback rejection. `validateModelCoverage` is narrowed to catch only `MissingModelForRoleError` (and incidentally `RoutingProfileCycleError` from G23) and rethrow the new error verbatim. Proposal B (no-typed-error refactor) is rejected because it cannot distinguish the two failure modes.

**Approved by**: GPT-5.5 (copilot) reviewer at round 4 — see [04-review-r4.md](04-review-r4.md).

**Implementation pointer**: [03-plan-r4.md](03-plan-r4.md). Touches [src/routing/resolver.ts](../../../../src/routing/resolver.ts), [src/routing/resolver.test.ts](../../../../src/routing/resolver.test.ts), [src/config-validation.ts](../../../../src/config-validation.ts), and [src/config-validation.test.ts](../../../../src/config-validation.test.ts). The validator-boundary test asserts `configPath` is a non-empty string rather than an exact path, because the typed error captures the resolver's `configPath()` value verbatim and the validator does not thread its argument path into the error payload (avoids over-engineering).

**Sequencing**: Coordinate with G23, G24, and G26 because they all touch [src/routing/resolver.ts](../../../../src/routing/resolver.ts). Recommended landing order is G23 → G24 → G25 → G26 within a single resolver batch.

**Daemon impact**: Configs whose `allowed_models`/`allowed_accounts` constraints already select valid models behave identically. Configs that previously fell open and returned arbitrary allowed entries will now fail loudly at validation or at first route resolution. Any saivage-v3 restart remains operator-gated.
