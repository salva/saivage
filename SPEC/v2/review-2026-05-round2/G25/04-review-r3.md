## Findings

1. The validator-boundary `configPath` assertion is sourced from the wrong contract. Task 6 calls `validateModelCoverage(cfg, routing, "/proj/.saivage/saivage.json")` and then expects the rethrown `NoAllowedRouteMatchError.configPath` to equal that argument at [SPEC/v2/review-2026-05-round2/G25/03-plan-r3.md](03-plan-r3.md#L291-L301). The design also records this in the validator inventory table and says exact equality is valid because the caller passes a fixed string at [SPEC/v2/review-2026-05-round2/G25/02-design-r3.md](02-design-r3.md#L178-L180). But the proposed resolver constructs `NoAllowedRouteMatchError` with `configPath()` at the model and account throw sites, not with the validator argument, at [SPEC/v2/review-2026-05-round2/G25/02-design-r3.md](02-design-r3.md#L83) and [SPEC/v2/review-2026-05-round2/G25/02-design-r3.md](02-design-r3.md#L140). In the live code, `configPath()` derives from `saivageDir(projectRoot)` / environment / current project root at [src/config.ts](../../../../src/config.ts#L217-L225), while `validateModelCoverage` only receives `configPathStr` as its own parameter and currently uses it when creating the aggregate missing-model error at [src/config-validation.ts](../../../../src/config-validation.ts#L41-L69). Since round 3 explicitly preserves verbatim propagation, the validator cannot make the resolver-created error path equal the validator argument unless the design also threads that path into resolver error construction. As written, the planned boundary test will either fail outside a specially prepared environment or pressure the implementation to mutate/re-wrap the resolver error, which would contradict the stated verbatim propagation. Fix the validator test and its inventory row to assert the same non-empty string contract as the resolver tests, or set the project-root environment so `configPath()` deterministically returns `/proj/.saivage/saivage.json` before the resolver throws.
