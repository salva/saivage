## Blocking findings

1. **The G33 handler still uses the pre-G34-r2 fetch helper contract, so the proposed implementation will not compile or preserve the mid-body timeout guarantee after G34 lands.** Round 3 correctly moves helper ownership to G34 and imports from [src/mcp/httpFetch.ts](src/mcp/httpFetch.ts), but the handler sketch assigns `fetchWithTimeout(...)` directly to a `Response` in [SPEC/v2/review-2026-05-round2/G33/02-design-r3.md](SPEC/v2/review-2026-05-round2/G33/02-design-r3.md#L263-L270) and then passes that value to `readBoundedTextBody(response, WEB_SEARCH_MAX_BYTES)` in [SPEC/v2/review-2026-05-round2/G33/02-design-r3.md](SPEC/v2/review-2026-05-round2/G33/02-design-r3.md#L290-L294). G34 r2 no longer exposes that shape: it defines `TimedFetch` with `response`, `signal`, and `timedOut()` in [SPEC/v2/review-2026-05-round2/G34/02-design-r2.md](SPEC/v2/review-2026-05-round2/G34/02-design-r2.md#L62-L85), and its caller pattern destructures that object, passes `signal` into `readBoundedTextBody`, and calls `classifyNetworkError(err, url, { timedOut: timedOut() })` in [SPEC/v2/review-2026-05-round2/G34/02-design-r2.md](SPEC/v2/review-2026-05-round2/G34/02-design-r2.md#L529-L549). G33 r3 even states that no explicit signal is needed because the helper composes its own timeout signal in [SPEC/v2/review-2026-05-round2/G33/02-design-r3.md](SPEC/v2/review-2026-05-round2/G33/02-design-r3.md#L348-L349), which contradicts G34 r2's contract that the caller must carry the returned signal through body reads. Required change: rewrite the G33 handler around `TimedFetch`, import any required G34 types/helpers, pass `signal` to `readBoundedTextBody`, and pass `{ timedOut: timedOut() }` to `classifyNetworkError` in the body-read catch. Otherwise row 14's `TIMEOUT` assertion in [SPEC/v2/review-2026-05-round2/G33/02-design-r3.md](SPEC/v2/review-2026-05-round2/G33/02-design-r3.md#L396-L397) is not a valid guard.

2. **The config insertion plan still targets the pre-G34 config shape even though G33 declares G34 as a hard prerequisite.** G33 r3 says G34 must land first in [SPEC/v2/review-2026-05-round2/G33/03-plan-r3.md](SPEC/v2/review-2026-05-round2/G33/03-plan-r3.md#L17-L17), and the design correctly says G34's `fetchTimeoutMs` should sit between `maxDownloadBytes` and the new web-search group in [SPEC/v2/review-2026-05-round2/G33/02-design-r3.md](SPEC/v2/review-2026-05-round2/G33/02-design-r3.md#L159-L159). But the concrete plan still tells the implementer to append the web-search group immediately after the current `maxDownloadBytes` line in [SPEC/v2/review-2026-05-round2/G33/03-plan-r3.md](SPEC/v2/review-2026-05-round2/G33/03-plan-r3.md#L46-L56). After G34 r2 lands, the config block has renamed `maxFetchChars` to `maxFetchBytes` and added `fetchTimeoutMs` after `maxDownloadBytes` in [SPEC/v2/review-2026-05-round2/G34/02-design-r2.md](SPEC/v2/review-2026-05-round2/G34/02-design-r2.md#L31-L32) and [SPEC/v2/review-2026-05-round2/G34/02-design-r2.md](SPEC/v2/review-2026-05-round2/G34/02-design-r2.md#L255-L257). Required change: make the plan's edit target the post-G34 size-caps group, explicitly placing web-search fields after `fetchTimeoutMs`, and refresh the convention table so it names `maxFetchBytes` rather than the deleted `maxFetchChars` when discussing the post-G34 order.
