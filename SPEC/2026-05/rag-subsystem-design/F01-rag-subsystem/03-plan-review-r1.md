## Per-axis findings

1. Scope and non-goals: mostly aligned, but not approvable as written. The plan correctly scopes the implementation to the focused in-process RAG library under the planned `saivage/src/rag/` tree and makes it opt-in through `rag.enabled` with no loaded module or disk artifact when disabled [03-plan-r1.md](saivage/SPEC/2026-05/rag-subsystem-design/F01-rag-subsystem/03-plan-r1.md#L5). It also explicitly excludes consumers, local embedding, MCP, telemetry, encryption-at-rest, rerankers, hybrid retrieval, cross-dataset query, and background ingest [03-plan-r1.md](saivage/SPEC/2026-05/rag-subsystem-design/F01-rag-subsystem/03-plan-r1.md#L11-L18), matching the approved Proposal A exclusions [02-design-r2.md](saivage/SPEC/2026-05/rag-subsystem-design/F01-rag-subsystem/02-design-r2.md#L787). However, the open-question section still permits substituting `@lancedb/lancedb` for B03 if [sqlite-vec](https://github.com/asg017/sqlite-vec) prebuilds fail [03-plan-r1.md](saivage/SPEC/2026-05/rag-subsystem-design/F01-rag-subsystem/03-plan-r1.md#L366). Proposal A is the approved design [02-design-r2.md](saivage/SPEC/2026-05/rag-subsystem-design/F01-rag-subsystem/02-design-r2.md#L566), and v1 supports one vector store, `sqlite-vec` [02-design-r2.md](saivage/SPEC/2026-05/rag-subsystem-design/F01-rag-subsystem/02-design-r2.md#L20), [02-design-r2.md](saivage/SPEC/2026-05/rag-subsystem-design/F01-rag-subsystem/02-design-r2.md#L640). That fallback path violates the Proposal A-only constraint.

2. Batch ordering and completeness: partially aligned, with a blocking validation gap. The batch order is foundational-first: dependencies, types/config, store/provider/chunkers/security, ingest/query, manager, then e2e/docs [03-plan-r1.md](saivage/SPEC/2026-05/rag-subsystem-design/F01-rag-subsystem/03-plan-r1.md#L43-L53). The table also lists files created, files modified, validation tags, and dependencies for each batch [03-plan-r1.md](saivage/SPEC/2026-05/rag-subsystem-design/F01-rag-subsystem/03-plan-r1.md#L43-L53), and the rollback model says each batch is one commit [03-plan-r1.md](saivage/SPEC/2026-05/rag-subsystem-design/F01-rag-subsystem/03-plan-r1.md#L37). The blocker is B11: the plan states every batch ends with at minimum `T` + `L` + `A` [03-plan-r1.md](saivage/SPEC/2026-05/rag-subsystem-design/F01-rag-subsystem/03-plan-r1.md#L33), but B11 has validation `none (docs only)` in both the summary table and detail section [03-plan-r1.md](saivage/SPEC/2026-05/rag-subsystem-design/F01-rag-subsystem/03-plan-r1.md#L53), [03-plan-r1.md](saivage/SPEC/2026-05/rag-subsystem-design/F01-rag-subsystem/03-plan-r1.md#L339).

3. Files modified outside planned `saivage/src/rag/`: failing. The plan says outside-`saivage/src/rag/` modifications are limited to [saivage/package.json](saivage/package.json) and [saivage/src/config.ts](saivage/src/config.ts) [03-plan-r1.md](saivage/SPEC/2026-05/rag-subsystem-design/F01-rag-subsystem/03-plan-r1.md#L5). It then allows conditional edits to [saivage/tsconfig.json](saivage/tsconfig.json) in B03 [03-plan-r1.md](saivage/SPEC/2026-05/rag-subsystem-design/F01-rag-subsystem/03-plan-r1.md#L45), [03-plan-r1.md](saivage/SPEC/2026-05/rag-subsystem-design/F01-rag-subsystem/03-plan-r1.md#L178), [saivage/eslint.config.js](saivage/eslint.config.js) in B10 [03-plan-r1.md](saivage/SPEC/2026-05/rag-subsystem-design/F01-rag-subsystem/03-plan-r1.md#L52), [03-plan-r1.md](saivage/SPEC/2026-05/rag-subsystem-design/F01-rag-subsystem/03-plan-r1.md#L318), and a possible [saivage/vitest.config.ts](saivage/vitest.config.ts) edit [03-plan-r1.md](saivage/SPEC/2026-05/rag-subsystem-design/F01-rag-subsystem/03-plan-r1.md#L360), [03-plan-r1.md](saivage/SPEC/2026-05/rag-subsystem-design/F01-rag-subsystem/03-plan-r1.md#L373). Those gates violate the stated review constraint. The plan does correctly forbid touching [saivage/src/agents/](saivage/src/agents/), [saivage/src/runtime/](saivage/src/runtime/), [saivage/src/server/](saivage/src/server/), [saivage/src/knowledge/](saivage/src/knowledge/), and [saivage/web/](saivage/web/) [03-plan-r1.md](saivage/SPEC/2026-05/rag-subsystem-design/F01-rag-subsystem/03-plan-r1.md#L11), but the conditional config edits are still out of bounds.

4. Validation commands: mostly realistic, with B11 carrying the same blocker as axis 2. The validation tags map to real commands: `npm run typecheck`, `npm run lint`, `npx vitest run src/rag`, `npx vitest run tests/rag`, and `npm test` [03-plan-r1.md](saivage/SPEC/2026-05/rag-subsystem-design/F01-rag-subsystem/03-plan-r1.md#L26-L30). The package provides `test`, `lint`, and `typecheck` scripts [package.json](saivage/package.json#L17-L21), and Vitest is already a dev dependency [package.json](saivage/package.json#L59). The manual `S` smoke is explicit and tied to B10 [03-plan-r1.md](saivage/SPEC/2026-05/rag-subsystem-design/F01-rag-subsystem/03-plan-r1.md#L31), [03-plan-r1.md](saivage/SPEC/2026-05/rag-subsystem-design/F01-rag-subsystem/03-plan-r1.md#L320). B11 still needs a concrete validation row instead of `none`.

5. Rollback strategy: aligned. The plan requires one commit per batch, uses `git revert <hash>` as the normal rollback path, and keeps the subsystem opt-in via `rag.enabled` so absent/false config loads no module and creates no disk artifact [03-plan-r1.md](saivage/SPEC/2026-05/rag-subsystem-design/F01-rag-subsystem/03-plan-r1.md#L37). This matches the approved design's opt-in lifecycle, where removing the slice or setting `enabled: false` stops manager construction and operators may remove `.saivage/rag/` at any time [02-design-r2.md](saivage/SPEC/2026-05/rag-subsystem-design/F01-rag-subsystem/02-design-r2.md#L774).

6. Acceptance criteria: aligned once the blocking items above are fixed. The criteria are concrete and verifiable: typecheck, lint, full Vitest, isolated planned `saivage/src/rag/` tests, isolated planned `saivage/tests/rag/` tests, manual OpenAI smoke, drift refusal, opt-out inertness, and secret-exclusion behavior [03-plan-r1.md](saivage/SPEC/2026-05/rag-subsystem-design/F01-rag-subsystem/03-plan-r1.md#L379-L387). They are specific enough for an implementer to validate, but they depend on fixing B11's missing validation and the out-of-scope config gates.

7. Architecture and complexity: partially aligned, with two required fixes. The plan keeps one store, one provider, and three chunkers in the main scope [03-plan-r1.md](saivage/SPEC/2026-05/rag-subsystem-design/F01-rag-subsystem/03-plan-r1.md#L5), matching Proposal A's minimal shape [02-design-r2.md](saivage/SPEC/2026-05/rag-subsystem-design/F01-rag-subsystem/02-design-r2.md#L570). Drift refusal is placed at B08 in the batch sequence [03-plan-r1.md](saivage/SPEC/2026-05/rag-subsystem-design/F01-rag-subsystem/03-plan-r1.md#L50), matching the approved requirement that embedding-config drift is refused and rebuild is the recovery path [02-design-r2.md](saivage/SPEC/2026-05/rag-subsystem-design/F01-rag-subsystem/02-design-r2.md#L335), [02-design-r2.md](saivage/SPEC/2026-05/rag-subsystem-design/F01-rag-subsystem/02-design-r2.md#L785). The blockers are the same `@lancedb/lancedb` fallback path [03-plan-r1.md](saivage/SPEC/2026-05/rag-subsystem-design/F01-rag-subsystem/03-plan-r1.md#L366) and the unresolved `releaseFingerprint` derivation [03-plan-r1.md](saivage/SPEC/2026-05/rag-subsystem-design/F01-rag-subsystem/03-plan-r1.md#L193), [03-plan-r1.md](saivage/SPEC/2026-05/rag-subsystem-design/F01-rag-subsystem/03-plan-r1.md#L370). Drift refusal cannot be fully implementer-ready until the fingerprint value is pinned.

8. Autonomous-document rule: aligned. I found no actual references to the review dance, reviewers, prior rounds, or APPROVED markers. The only relevant “round” usage is technical prose about errors round-tripping fields [03-plan-r1.md](saivage/SPEC/2026-05/rag-subsystem-design/F01-rag-subsystem/03-plan-r1.md#L129), which should not be flagged under the literal rule.

9. File-link rule: failing. The detailed batch sections mostly use markdown links, but the batch summary table contains existing repository paths such as [saivage/package.json](saivage/package.json) and [saivage/src/config.ts](saivage/src/config.ts) as inline code rather than links, and planned real-repo paths such as `saivage/src/rag/types.ts`, `saivage/tests/rag/e2e-ingest-query.test.ts`, and `saivage/SPEC/v2/rag/README.md` are also presented as inline code without clear “planned path” framing [03-plan-r1.md](saivage/SPEC/2026-05/rag-subsystem-design/F01-rag-subsystem/03-plan-r1.md#L43-L53). The non-goal list also uses inline-code real directories for the forbidden-touch set [03-plan-r1.md](saivage/SPEC/2026-05/rag-subsystem-design/F01-rag-subsystem/03-plan-r1.md#L11). Illustrative globs and placeholders can stay in inline code, but existing real repo paths should be markdown links, and planned paths should be clearly marked as planned paths.

## Required Changes

1. Remove the `@lancedb/lancedb` substitution path and any other alternative-store path from §8/B01. Under this review, the plan either implements Proposal A with `sqlite-vec` or stops for a new design decision; it should not authorize an alternate implementation inside the batch plan.
2. Resolve design-shaping open questions before approval, especially the exact `releaseFingerprint` derivation, OpenAI embedding input cap, and rebuild staging layout. The plan can include verification commands, but it should not leave implementers to alter batch text midstream.
3. Remove conditional edits to [saivage/tsconfig.json](saivage/tsconfig.json), [saivage/eslint.config.js](saivage/eslint.config.js), and [saivage/vitest.config.ts](saivage/vitest.config.ts). The only allowed outside-`saivage/src/rag/` modifications are [saivage/package.json](saivage/package.json) for engines/deps and [saivage/src/config.ts](saivage/src/config.ts) for the `rag` config slice.
4. Give B11 concrete validation commands, at minimum the same `T`, `L`, `A` baseline the plan declares for every batch, or remove B11 from the implementation batch sequence.
5. Convert real repository paths in the batch summary and forbidden-touch list to markdown links. Keep inline code only for commands, placeholders, and clearly illustrative globs.

VERDICT: CHANGES_REQUESTED
