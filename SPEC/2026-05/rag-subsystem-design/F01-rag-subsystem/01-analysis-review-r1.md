## Axis 3 — Vector Store Claims

The sqlite-vec section mixes exact and approximate-search claims. It says recent releases add an IVF-style approximate option and that brute force is single-digit milliseconds at 100k vectors at [saivage/SPEC/2026-05/rag-subsystem-design/F01-rag-subsystem/01-analysis-r1.md#L340-L341](saivage/SPEC/2026-05/rag-subsystem-design/F01-rag-subsystem/01-analysis-r1.md#L340-L341), then later says to verify whether IVF or ANN indexes are stable at [saivage/SPEC/2026-05/rag-subsystem-design/F01-rag-subsystem/01-analysis-r1.md#L798](saivage/SPEC/2026-05/rag-subsystem-design/F01-rag-subsystem/01-analysis-r1.md#L798). The primary recommendation should not rely on unverified sqlite-vec ANN features; if sqlite-vec is exact scan only, the analysis should say so cleanly.

The Node 20 constraint is under-applied. The sqlite-vec paragraph says it can load into `node:sqlite` at [saivage/SPEC/2026-05/rag-subsystem-design/F01-rag-subsystem/01-analysis-r1.md#L333](saivage/SPEC/2026-05/rag-subsystem-design/F01-rag-subsystem/01-analysis-r1.md#L333), but Saivage v2 targets Node 20 and should not depend on a newer built-in sqlite module path. The viable Node 20 path should be stated as the supported path.

The filtering story is overclaimed. The analysis says filtering on a source column costs nothing when the ANN has built-in filter support at [saivage/SPEC/2026-05/rag-subsystem-design/F01-rag-subsystem/01-analysis-r1.md#L119](saivage/SPEC/2026-05/rag-subsystem-design/F01-rag-subsystem/01-analysis-r1.md#L119), and later says sqlite-vec gives full SQL filtering at [saivage/SPEC/2026-05/rag-subsystem-design/F01-rag-subsystem/01-analysis-r1.md#L338-L339](saivage/SPEC/2026-05/rag-subsystem-design/F01-rag-subsystem/01-analysis-r1.md#L338-L339). The next revision should separate post-filtering, pre-filtering, partition keys, and metadata-table joins for each store; those have different recall and latency implications.

The summary says sqlite-vec and LanceDB both share enough properties including ANN at [saivage/SPEC/2026-05/rag-subsystem-design/F01-rag-subsystem/01-analysis-r1.md#L447-L449](saivage/SPEC/2026-05/rag-subsystem-design/F01-rag-subsystem/01-analysis-r1.md#L447-L449). That is inaccurate if sqlite-vec is being used as exact KNN. The abstraction should be based on the actual common denominator, not a wished-for ANN feature.

The Milvus Lite note at [saivage/SPEC/2026-05/rag-subsystem-design/F01-rag-subsystem/01-analysis-r1.md#L424](saivage/SPEC/2026-05/rag-subsystem-design/F01-rag-subsystem/01-analysis-r1.md#L424) should be checked or removed. Node SDK support against a local embedded Milvus Lite process is not established in the analysis and is not needed for the shortlist.

## Axis 4 — Corpus Size and Latency Estimates

Several latency claims are too confident for an analysis without measurements. The skills lookup budget says hosted query embedding plus ANN search under 150 ms p50 and 400 ms p95 is trivial at [saivage/SPEC/2026-05/rag-subsystem-design/F01-rag-subsystem/01-analysis-r1.md#L33-L34](saivage/SPEC/2026-05/rag-subsystem-design/F01-rag-subsystem/01-analysis-r1.md#L33-L34), but the same document later says hosted query embedding alone is 80-200 ms typical at [saivage/SPEC/2026-05/rag-subsystem-design/F01-rag-subsystem/01-analysis-r1.md#L104-L107](saivage/SPEC/2026-05/rag-subsystem-design/F01-rag-subsystem/01-analysis-r1.md#L104-L107). Provider tail latency can consume the whole p95 budget before the store is touched.

The vector-store timing row at [saivage/SPEC/2026-05/rag-subsystem-design/F01-rag-subsystem/01-analysis-r1.md#L106](saivage/SPEC/2026-05/rag-subsystem-design/F01-rag-subsystem/01-analysis-r1.md#L106) claims 1-15 ms for 10k-100k rows at 1536 dimensions with brute-force sqlite-vec. That may be possible on some hardware, but it is not defensible as a general design input without benchmark data on the target host and selected vector type.

The single-digit-QPS statement at [saivage/SPEC/2026-05/rag-subsystem-design/F01-rag-subsystem/01-analysis-r1.md#L95-L96](saivage/SPEC/2026-05/rag-subsystem-design/F01-rag-subsystem/01-analysis-r1.md#L95-L96) also conflates vector-store QPS with hosted embedding-provider QPS. A burst of 50 queries still means 50 hosted embedding calls unless query embeddings are cached or batched.

The ingest throughput section is internally inconsistent. It calculates 5M hosted tokens at a 1M TPM limit as roughly 5 minutes at [saivage/SPEC/2026-05/rag-subsystem-design/F01-rag-subsystem/01-analysis-r1.md#L544-L546](saivage/SPEC/2026-05/rag-subsystem-design/F01-rag-subsystem/01-analysis-r1.md#L544-L546), but then says 1 percent incremental ingest is sub-second in both hosted and local modes at [saivage/SPEC/2026-05/rag-subsystem-design/F01-rag-subsystem/01-analysis-r1.md#L552](saivage/SPEC/2026-05/rag-subsystem-design/F01-rag-subsystem/01-analysis-r1.md#L552). One percent of 5M tokens is still about 50k tokens, which cannot be guaranteed sub-second through hosted embeddings.

## Axis 5 — Chunking and Ingestion Strategy

The tree-sitter Node 20 ESM risk is acknowledged at [saivage/SPEC/2026-05/rag-subsystem-design/F01-rag-subsystem/01-analysis-r1.md#L482](saivage/SPEC/2026-05/rag-subsystem-design/F01-rag-subsystem/01-analysis-r1.md#L482), which is good. The next revision should make the fallback choice crisper: if native tree-sitter is not clean under Node 20 ESM, the design should prefer the portable path or defer AST chunking rather than leaving a hard install problem to implementation.

The code chunking section sends mixed signals on exact-symbol lookup. It says symbol metadata can answer function-name queries without round-tripping to ripgrep at [saivage/SPEC/2026-05/rag-subsystem-design/F01-rag-subsystem/01-analysis-r1.md#L478-L479](saivage/SPEC/2026-05/rag-subsystem-design/F01-rag-subsystem/01-analysis-r1.md#L478-L479), then correctly says exact-symbol search belongs to ripgrep or the language server at [saivage/SPEC/2026-05/rag-subsystem-design/F01-rag-subsystem/01-analysis-r1.md#L484](saivage/SPEC/2026-05/rag-subsystem-design/F01-rag-subsystem/01-analysis-r1.md#L484). Remove the first implication or recast it as metadata display only.

The idempotency algorithm is under-specified for chunk-boundary changes. It upserts by dataset, path, chunk index, and content hash at [saivage/SPEC/2026-05/rag-subsystem-design/F01-rag-subsystem/01-analysis-r1.md#L523-L527](saivage/SPEC/2026-05/rag-subsystem-design/F01-rag-subsystem/01-analysis-r1.md#L523-L527), but does not explicitly delete stale chunks from the old version of a changed file when chunk counts or boundaries shift. Deletion handling must cover changed files, not only files absent from the source tree.

Dimension-change handling is strong and aligned with the no-backward-compatibility rule at [saivage/SPEC/2026-05/rag-subsystem-design/F01-rag-subsystem/01-analysis-r1.md#L531-L540](saivage/SPEC/2026-05/rag-subsystem-design/F01-rag-subsystem/01-analysis-r1.md#L531-L540). Keep that part.
