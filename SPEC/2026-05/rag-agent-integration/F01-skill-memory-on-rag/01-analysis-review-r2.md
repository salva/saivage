# F01 — Skill and Memory on RAG: Analysis Review

The analysis is directionally aligned with the topic: it treats RAG as the intended skill/memory backend, keeps the existing MCP tool names, rejects a flat-file fallback, separates skill and memory datasets, and catalogs most of the current knowledge modules. It also correctly identifies several current-source facts, including the `project|stage|session` knowledge scopes, bundled skill UUID instability, the `text-embedding-3-small` config constraint, and the `metadataOverlay.scope === "memory"` source inference defect.

It is not yet approvable. Several core mechanics depend on behavior that the fixed RAG subsystem does not provide, so the proposed facade would lose chunks, misclassify records, or fail to reconstruct exact skill/memory bodies.

## Required Changes

1. Correct the records-ingest model. The document repeatedly treats `Dataset.ingest({ kind: "records", items: [...] })` as an incremental upsert/delete-by-path primitive, but current `runIngest` treats the supplied records as the complete seen path set for that ingest call. It builds record inputs in [src/rag/pipeline.ts](src/rag/pipeline.ts#L108), deletes prior chunks for each changed path in [src/rag/pipeline.ts](src/rag/pipeline.ts#L256), and then deletes every prior `file_state` path not present in the current input in [src/rag/pipeline.ts](src/rag/pipeline.ts#L276-L288). A one-record create/update/delete will therefore remove chunks for unrelated skills or memories. The analysis must redesign all CRUD flows around complete collection snapshots, or name this as an implementation blocker under the fixed RAG API. The empty-text deletion strategy in the lifecycle and resolved-questions sections is not sufficient as written.

2. Fix the canonical `source` model. The schema table says skills and memories can supply `ChunkMetadata.source`, and the resolved-questions section says the facade can pass `metadata.source` through records input. That is factually wrong against current source. `ChunkMetadataInput` includes `source` in [src/rag/types.ts](src/rag/types.ts#L84), but `buildRecordItems` drops it and passes only a limited overlay, while `runIngest` sets `source` from `metadataOverlay.scope === "memory" ? "memory" : inferSource(path)` in [src/rag/pipeline.ts](src/rag/pipeline.ts#L197). Since real knowledge scopes are `project|stage|session` in [src/knowledge/types.ts](src/knowledge/types.ts#L41) and the proposed synthetic paths end in `.md`, skill and memory chunks will be indexed as `doc`, not `skill`/`memory`. The analysis must choose one canonical identity/source/path model that works without changing `src/rag/`; a follow-up to honor `metadataOverlay.source` is a RAG implementation change and violates the fixed-RAG constraint for this dance.

3. Provide a feasible exact-read and eager-loading design using only the public RAG surface. Existing `read_skill`, `get_memory`, eager boot injection, and survivor reinjection need exact record bodies, not semantically similar chunks. `Dataset.query` always embeds a query string and asks the vector store for top-K nearest hits in [src/rag/query/pipeline.ts](src/rag/query/pipeline.ts#L23-L30); there is no public fetch-by-path or list-chunks-by-filter operation. The proposed helper that runs a `pathGlob` chunk query and concatenates by `chunkIndex` is not reliable or available through the public API. The analysis must either define a storage model that preserves exact bodies while still making RAG the primary backend, or explicitly state that fixed RAG lacks the exact-read primitive required to preserve the MCP result contract.

4. Repair the sidecar schema and identity rules. The sidecar is doing important record resolution work, but the schema is not strong enough for the document's own model. `rag_path` needs a uniqueness guarantee and lookup index because hydration is by path. The skill-name uniqueness constraint uses nullable `scope_ref`, which SQLite will not enforce for project-scope duplicate names unless normalized or implemented with partial indexes. Memory topic collision rules are not represented at all. The stable builtin id form `builtin:<name>` conflicts with the current UUID-only `SkillRecordSchema` in [src/knowledge/types.ts](src/knowledge/types.ts#L105-L117), and `body_path` is still required in [src/knowledge/types.ts](src/knowledge/types.ts#L114) despite the inventory saying it becomes optional or obsolete. The analysis must define the post-refactor public record type and sidecar keys coherently.

5. Clarify lock ordering, transaction boundaries, and recovery. The lifecycle section says it begins a SQLite transaction, calls `dataset.ingest`, then commits, but the partial-write section discusses step 6 throwing after step 5 has committed. Those cannot both be true. Holding a SQLite write transaction open across embedding/provider work and RAG ingest is also a risky cross-resource lock design. The analysis must specify a consistent ordering for runtime lock, scope chain lock, sidecar transaction, sidecar audit append, RAG ingest lock, and recovery. Remove or fully define the invented `AUDIT_LATE` mechanism; it is not grounded in the current file inventory and reads like over-engineering.

6. Correct current-state permissions and tool-filter facts. The analysis overstates agent access to knowledge read/search tools. The current read-only filter contains only `list_skills` and `read_skill` among knowledge tools in [src/agents/tool-filters.ts](src/agents/tool-filters.ts#L9-L12), while `search_skills`, `list_memories`, `get_memory`, and `search_memories` are not in `READ_ONLY_TOOLS`. The permissions summary also omits planner memory supersede/archive/delete rights and inspector skill/memory destructive rights from [src/knowledge/permissions.ts](src/knowledge/permissions.ts). Since the topic explicitly asks which agents call which MCP tools and with what patterns, the analysis must separate MCP ACL permissions from roster tool-filter exposure and state both accurately.

7. Complete the file inventory. The backout table lists the main implementation files, but the test inventory is incomplete. Current `src/knowledge/` also contains `integration.test.ts`, `regression.test.ts`, `lifecycle.archive.test.ts`, and `concurrency.test.ts` in addition to the tests already listed. The topic asks for every file under `src/knowledge/` and `src/mcp/knowledge*.ts` to be classified, so the analysis must include all current files and state keep/rewrite/delete/rename for each.

8. Tighten first-run seeding and backout semantics. The document refuses to auto-migrate old `records/*.json`, but then relies on a manual `saivage knowledge seed` command and deletes the legacy trees after successful seed. It does not specify what authoritative source the seed command reads, how that avoids becoming a compatibility shim, or how it performs the complete-record RAG ingest required by current `runIngest`. The analysis must define the first-run source of truth, the seed transaction/checkpoint behavior, and the exact failure codes when the sidecar exists but RAG chunks are empty or stale.

9. Keep the public MCP result shapes precise. The analysis says search returns records with body snippets, scores, and lineage fields, but the current search helpers return only `{ id, score, snippet }` hits through the MCP handlers. The migration can preserve or intentionally expand shapes only if the topic's no-degradation constraint is respected and any surface change is explicitly justified.

## Style and Coverage Notes

The document is mostly self-contained and uses repo-root-relative links. It covers the required headings at a high level, but the incomplete mechanics above affect the topic's required questions on source indexing, identity, lifecycle, first-run seeding, error codes, and file inventory. The next revision should remove the RAG follow-up that changes source handling, make one canonical model for source/path/id/body retrieval, and keep the design architecture-first without preserving legacy JSON/index machinery as a hidden fallback.
