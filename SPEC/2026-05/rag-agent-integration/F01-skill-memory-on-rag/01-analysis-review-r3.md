# Review: F01 Skill/Memory on RAG Analysis

## Findings

1. The analysis does not fully cover the topic's required current-state questions. It describes the target sidecar/RAG model well, but the required analysis must also explain what the current `src/knowledge/` layer stores, how its JSON tree and `index.json` files are used, where skill bodies live today, and which agents can call which MCP tools today. The current implementation stores skill records plus separate `body_path` markdown files, memory records with inline `body`, per-scope `records/*.json`, `index.json`, and `audit.jsonl` under `.saivage/{skills,memory}/...` ([src/knowledge/types.ts](src/knowledge/types.ts), [src/knowledge/lifecycle.ts](src/knowledge/lifecycle.ts), [src/mcp/knowledgeSkills.ts](src/mcp/knowledgeSkills.ts), [src/mcp/knowledgeMemory.ts](src/mcp/knowledgeMemory.ts)). The document needs a concrete current-state section instead of jumping directly to the replacement layout.

2. The source and metadata mapping is partly inaccurate. It correctly observes that `buildRecordItems` drops `ChunkMetadataInput.source`, but the sample ingest metadata passes `createdAt: row.created_at` even though `ChunkMetadata.createdAt` is typed as `number` ([src/rag/types.ts](src/rag/types.ts)). It also says `id` is ignored but `record_id` propagates; the RAG pipeline has no `record_id` metadata field, and `recordId` is only an internal unused `InputItem` property after `buildRecordItems` ([src/rag/pipeline.ts](src/rag/pipeline.ts)). The design must explicitly rely on `metadata.path`/`rag_path` as the sole join key, and it must coerce timestamps to the current metadata types.

3. The sidecar schema drops current record fields without a gap analysis. The topic requires mapping current skill and memory record fields onto `ChunkMetadata`, sidecar columns, or deliberate deletion. The proposed `record` table keeps `record_json` and `body`, but it omits explicit treatment of `author_agent`, `origin`, `description`, `triggers`, `target_agents`, `topic`, `keys`, `source_ref`, `expires_at`, `ttl_ms`, `supersedes`, `superseded_by`, `relates_to`, and `survive_compaction` from the current schemas ([src/knowledge/types.ts](src/knowledge/types.ts)). Some of these can live inside `record_json`, but the analysis must say which are indexed/queryable, which are sidecar-only, and which are removed under the architecture-first rule.

4. Permission coverage is factually wrong. The analysis says every existing role has read access to both kinds in every scope, but `data_agent` has `read-memory`, `list-memory`, and `search-memory` set to `-` in the current matrix ([src/knowledge/permissions.ts](src/knowledge/permissions.ts)). It also says workers have project/stage memory writes, but coder/researcher writes are `Y*` and `checkScope` restricts them to the current stage only. This matters because any added tool-filter exposure must still line up with the handler ACL and must not imply broader memory access than the current contract.

5. The `READ_ONLY_TOOLS` delta is under-justified and potentially contract-changing. The current read-only set contains `read_file`, `list_dir`, `search_files`, `git_status`, `git_log`, `git_diff`, `list_skills`, and `read_skill` ([src/agents/tool-filters.ts](src/agents/tool-filters.ts)). Adding `search_skills`, `list_memories`, `get_memory`, and `search_memories` changes Planner/Reviewer/Inspector/Chat tool availability, while Worker already receives most non-excluded tools. If this is intended, the analysis must state the before/after per filter kind and explain why it is not an agent-facing contract degradation or accidental expansion, especially for `data_agent` where the handler ACL still denies memory access.

6. First-run config persistence depends on an invented helper. The analysis says `initKnowledgeStore` writes missing datasets through a `saveSaivageConfig` helper introduced by F02, but the current config module exposes `loadConfig`, path helpers, and schemas only; there is no config-save helper ([src/config.ts](src/config.ts)). The F02 analysis explicitly calls out that runtime registration requires mutating the manager's dataset array or recreating the manager, not using an existing save helper. F01 must either specify a real new config writer owned by this work or make first-run protected dataset registration follow the fixed manager/config constraints without pretending the helper exists.

7. The startup ordering is incomplete. `registerBuiltinServices` currently registers knowledge MCP handlers before runtime lock acquisition and before there is any RAG manager object in scope ([src/server/bootstrap.ts](src/server/bootstrap.ts), [src/mcp/builtins.ts](src/mcp/builtins.ts)). The analysis introduces `initKnowledgeStore(projectRoot)` and `runtime.knowledge.<kind>.mutex` but does not say where those live, how the handlers receive the `RagManager`, or how startup ordering changes so knowledge initialization can fail fast when `config.rag.enabled === false` before tools are used.

8. Locking and ingest deferral are internally inconsistent. The write path says `pending_reingest` is written after ingest failure even though SQLite has already committed and no transaction is open; the crash-recovery section then says a crash between SQLite commit and RAG ingest is recovered by a marker written by the previous handler or by `rag_sync` divergence. A crash in that window cannot have written a failure marker. The design can rely on `rag_sync` divergence, but then `pending_reingest` is only an error marker, not the primary recovery mechanism. The analysis should define exactly when `pending_reingest` rows are inserted, cleared, and read, and avoid implying they cover pre-ingest crashes.

9. Search result compatibility is not demonstrated. Current `searchSkills` and `searchMemories` return `SearchHit[]` with exactly `{ id, score, snippet }` ([src/knowledge/lifecycle.ts](src/knowledge/lifecycle.ts)). The analysis proposes adding `scope`, `scopeRef`, and `lifecycleStatus`, then later claims the shape is identical and also introduces an internal degraded response envelope. Because the topic forbids public MCP API changes, the document must choose one: preserve the exact current `hits` element shape externally, or explicitly call out and justify a surface change.

10. The RAG query filter plan is incomplete for current MCP arguments. Current search tools accept only `query`, optional `scope`, and optional `limit`; list/read/get have different exact-return semantics ([src/mcp/knowledgeSkills.ts](src/mcp/knowledgeSkills.ts), [src/mcp/knowledgeMemory.ts](src/mcp/knowledgeMemory.ts)). The analysis describes caller-supplied `scope`/`scopeRef`/`role` filters for search even though the public schemas do not expose `scopeRef` or `role`. It needs to distinguish internal hydration filters from public input schemas and preserve the current argument shapes.

11. First-run seeding is underspecified against the fixed RAG manager surface. The analysis says the loader ensures datasets exist in `config.rag.datasets[]`, seeds builtins, and triggers snapshot ingest. It does not specify how `manager.register("knowledge.skills", ...)` is called, how provider stamp drift is handled for protected datasets, whether both protected datasets are registered before seeding, or how absent provider credentials turn into the exact fail-fast error required by the topic. This is a required failure-mode topic.

12. File inventory is not complete enough for implementation planning. The topic requires listing every file under `src/knowledge/` and `src/mcp/knowledge*.ts`; the document covers the main implementation and test files that currently appear likely, but it does not prove the inventory from the actual tree and misses adjacent registration/bootstrap touchpoints it already depends on, such as [src/mcp/builtins.ts](src/mcp/builtins.ts) and [src/server/bootstrap.ts](src/server/bootstrap.ts). Since no new file is added is asserted while the design also invents helpers and mutexes, the inventory must be reconciled with the proposed module layout.

13. Some architecture-first choices are solid, but the document still preserves ambiguous old behavior by omission. Removing the JSON tree, making SQLite the record-of-truth, treating RAG as a vector index, and full-snapshot reingest align with the project rule. However, leaving the legacy tree undeleted by runtime while saying the previous tree is gone is ambiguous. If runtime never reads it, that is acceptable, but the analysis should say old code paths are removed and stale files are inert operator artifacts, not part of the supported layout.

14. Style is close but not fully self-contained. The links are repo-root-relative, which is good. The document does not include reviewer/round references. It does, however, rely on F02/F03 for protected datasets, config saving, and future ACL rows without restating the necessary local assumptions. For an rN document that must be self-contained, those dependencies should be summarized as local facts or clearly marked as external prerequisites.

## Required Changes

1. Add a verified current-state section covering current on-disk layout, record schemas, indexing/search behavior, eager loading, MCP tool schemas/result shapes, and per-filter/per-ACL agent access.
2. Correct the factual errors around `ChunkMetadataInput`, `runIngest`, join keys, timestamp metadata types, current `READ_ONLY_TOOLS`, current `SkillRecord`/`MemoryRecord` schemas, and the permission matrix.
3. Replace the invented `saveSaivageConfig` dependency with a concrete config/manager registration strategy that respects the fixed RAG public API and the F02 runtime-registration constraints.
4. Specify exact startup wiring: where knowledge initialization runs, how handlers receive the `RagManager` and sidecars, how runtime mutexes are owned, and where RAG-disabled startup fails.
5. Tighten the sidecar schema and field-mapping analysis so every current record field is accounted for as metadata, indexed sidecar data, canonical JSON-only data, or deliberately removed.
6. Make search/list/read/get compatibility exact: preserve current external input schemas and result shapes, or explicitly mark and justify any surface change.
7. Rework ingest deferral and crash recovery so `pending_reingest`, `rag_sync`, and full-snapshot reingest have one consistent ordering story.
8. Complete the file inventory and implementation plan, including bootstrap/registration touchpoints and validation commands/tests needed for the new RAG-backed path.
9. Keep the architecture-first stance, but remove ambiguity about stale legacy files: supported code must not read, write, migrate, or shim the old JSON-tree format.

VERDICT: CHANGES_REQUESTED
