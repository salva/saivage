# F02 Collection Tools Analysis Review R5

Reviewed `01-analysis-r5.md` against the current `src/rag/`, MCP runtime, bootstrap, dispatcher, and tool-filter source. The analysis is directionally aligned on the seven-tool surface, public RAG API preservation, manager construction basics, runtime error envelope behavior, watcher unavailable mapping, and the filter-vs-handler authorization split, but it still has source-level correctness gaps that would lead to unsafe behavior or non-compiling implementation guidance.

Positive confirmations:

- The source supports the envelope correction: `McpRuntime.callTool` rejects unavailable in-process services before the handler runs, and wraps arbitrary `result.content` with `JSON.stringify` when `isError` is true (`src/mcp/runtime.ts#L153-L191`). F02 can use a typed `{ error: { code, message, details? } }` content shape for its own handler without claiming that every handler already does.
- The filter is presentation-only: `BaseAgent.getToolSchemas()` filters `mcpRuntime.getAllTools()` (`src/agents/base.ts#L662-L668`), while `Dispatcher.executeLocalTool()` resolves the called tool from the unfiltered runtime catalog and calls `mcpRuntime.callTool()` (`src/runtime/dispatcher.ts#L156-L192`). Handler-side `ctx.role` enforcement is therefore required.
- `createRagManager` is async, `providerOptions` is optional, and `RagManager.get()` resolves datasets from the captured `opts.datasets` array (`src/rag/manager.ts#L34-L39`, `src/rag/manager.ts#L87-L118`). There is no `watcherLogger` on `RagManagerOptions` or `DatasetOpenOptions` (`src/rag/dataset.ts#L59-L61`), although the lower-level `WatcherControllerArgs` has a private/test `log` seam (`src/rag/watcher/controller.ts#L49`).
- `WatcherUnavailableError` is exported from the public RAG barrel and is raised from the `chokidar.watch()` arm path (`src/rag/index.ts#L1-L14`, `src/rag/watcher/controller.ts#L126-L136`). Mapping that to `RAG_WATCHER_UNAVAILABLE` is distinct from the async `ENOSPC` event path.
- The reconcile wording about deletion-only sweeps is now correct: `updates = changedPaths.concat(removedPaths)` prevents the early return, but the later ingest loop filters only `result.changedPaths` and skips roots with no changed paths (`src/rag/watcher/controller.ts#L81-L96`).
- The tool count is internally consistent: exactly `rag_list`, `rag_stats`, `rag_query`, `rag_register`, `rag_ingest`, `rag_drop`, and `rag_admin`; no `rag_add`, no `delete_record`, and no `RAG_PROTECTED_SOURCE` are proposed.

Required changes:

1. Remove caller-supplied `include` / `exclude` from `rag_ingest`, or rewrite the analysis so they cannot narrow the snapshot. The analysis says caller globs are accepted and "cannot break snapshot semantics" because the root still comes from `dataset.config.sources[0]` (`01-analysis-r5.md#L132-L140`, `01-analysis-r5.md#L360-L369`). That is false against `runIngest`: the fs walk uses the supplied include/exclude (`src/rag/pipeline.ts#L83-L88`), `seenPaths` is built only from that walked set (`src/rag/pipeline.ts#L178-L184`), and every prior file_state path not seen is deleted (`src/rag/pipeline.ts#L276-L285`). A caller can pass a narrow include under the registered root and purge every other file in the collection. `rag_ingest` should use the configured `sources[0].include` / `sources[0].exclude` exactly for destructive snapshot convergence.

2. Fix the realpath containment design so it actually closes symlink and prefix escapes without changing the public RAG API. The analysis specifies `realpath.startsWith(realpath(ctx.projectRoot))` (`01-analysis-r5.md#L172-L180`), which accepts sibling prefixes such as `<project>-evil`; use a `path.relative`/exact-or-contained check like the existing `assertInside` pattern in `src/mcp/builtins.ts#L64-L71`. More importantly, the proposed handler post-validation cannot observe the actual `WalkedFile[]` used by `manager.ingest`: the public manager API accepts only `IngestInput` and returns `IngestReport` (`src/rag/manager.ts#L47-L47`, `src/rag/manager.ts#L154-L156`), `Dataset.ingest()` immediately delegates to `runIngest()` (`src/rag/dataset.ts#L101-L103`), and the walker runs inside `runIngest()` (`src/rag/pipeline.ts#L83-L88`). A separate handler pre-walk would be a TOCTOU check and would not validate what gets embedded. The analysis needs to require containment in private RAG internals, such as hardening `walk()` / `loadFsItems()` while preserving exported signatures, or define another implementation path that validates the exact walked files before any read/embed/store work.

3. Correct the deletion-convergence wording. `rag_admin reconcile` is correctly described as not purging deletion-only results, but the analysis overstates that `rag_ingest` is the only deletion convergence path (`01-analysis-r5.md#L142-L153`, `01-analysis-r5.md#L370-L373`). A live watcher `unlink` event enters `processBatch()` (`src/rag/watcher/controller.ts#L161-L164`), groups the event by source root, and calls full-root fs ingest (`src/rag/watcher/controller.ts#L184-L225`), which then purges missing file_state entries via the same snapshot deletion path (`src/rag/pipeline.ts#L276-L285`). Say `rag_ingest` is the manual/operator deletion convergence path; live watcher unlink also converges deletions, while reconcile does not.

4. Make the operator bypass source-grounded and compile-safe. `ToolCallContext` has no `operator` field today (`src/mcp/toolContext.ts#L17-L34`), while the analysis says the CLI passes a private `ctx.operator === true` flag (`01-analysis-r5.md#L261-L269`) and the handler snippet rejects all mutating calls solely by `ctx.role` and empty `RAG_ADMIN_ROLES` (`01-analysis-r5.md#L248-L255`). Define the internal context extension and predicate explicitly, for example `isRuntimeOperatorContext(ctx)`, and show the authorization condition as `isMutating && !isRuntimeOperatorContext(ctx) && !adminRoles.has(ctx.role)`. Also state that the flag is never parsed from tool args and is only set by the CLI/server code path that constructs the runtime context.

5. Fix the service-construction dataset type. The analysis correctly states `RagManagerOptions.datasets` is `ReadonlyArray<Omit<DatasetConfig, "projectId">>` (`01-analysis-r5.md#L83-L88`), but the bootstrap pseudocode declares `const ragDatasets: DatasetConfig[] = [...config.rag.datasets]` (`01-analysis-r5.md#L440-L447`). Current config dataset entries do not include `projectId` (`src/config.ts#L217-L260`), and `DatasetConfig` requires it (`src/rag/types.ts#L39-L54`). Use `Array<Omit<DatasetConfig, "projectId">>` or a dedicated local `RuntimeRagDatasetConfig` alias so F02 does not imply adding `projectId` to config or changing the public RAG types.

VERDICT: CHANGES_REQUESTED
