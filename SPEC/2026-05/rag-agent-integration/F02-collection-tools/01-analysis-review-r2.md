# F02 Collection Tools Analysis Review

## Summary

The analysis is directionally good on the single `rag` service, flat tool names, protected skill/memory datasets, and the current MCP `registerInProcess` and `ToolCallContext` shapes. It also correctly identifies that `Dataset.reconcile()` returns `Promise<void>` and that `IngestInput` uses `items`, matching [src/rag/dataset.ts](src/rag/dataset.ts#L166), [src/rag/types.ts](src/rag/types.ts#L143), and [src/rag/types.ts](src/rag/types.ts#L150).

The document is not ready to approve because the registration/persistence design is not implementable against the current manager contract, the register/watch contract does not satisfy the automatic-indexing requirement, the delete-record mechanism is an abstraction leak with incorrect semantics, and the authorization/error/concurrency stories still contain contradictions.

## Findings

1. Runtime registration and persistence contradict the current `RagManager` contract.

   The analysis says runtime-only `rag_register` materialises a collection for the current process and that `persist: true` writes it into config afterward [SPEC/2026-05/rag-agent-integration/F02-collection-tools/01-analysis-r2.md](SPEC/2026-05/rag-agent-integration/F02-collection-tools/01-analysis-r2.md#L64) [SPEC/2026-05/rag-agent-integration/F02-collection-tools/01-analysis-r2.md](SPEC/2026-05/rag-agent-integration/F02-collection-tools/01-analysis-r2.md#L275). In the current code, `RagManager.register()` opens and caches the dataset, but `get()`, `ingest()`, `query()`, `stats()`, and `drop()` all resolve the dataset id from the original `opts.datasets` array [src/rag/manager.ts](src/rag/manager.ts#L117) [src/rag/manager.ts](src/rag/manager.ts#L154) [src/rag/manager.ts](src/rag/manager.ts#L169). A dataset registered only through `manager.register()` can appear in `rag_list` via the registry, but subsequent `manager.ingest(id, ...)`, `manager.query(id, ...)`, or `manager.drop(id)` can still throw `DatasetNotFoundError`. This breaks the central promise of runtime-created collections. It also means `rag_list` can show registry entries that are not operational after restart, because `list()` reads the registry while `get()` reads configured datasets [src/rag/manager.ts](src/rag/manager.ts#L126) [src/rag/manager.ts](src/rag/manager.ts#L127).

   The persistence half is also unsupported as written. The document names an "existing atomic config-write helper" [SPEC/2026-05/rag-agent-integration/F02-collection-tools/01-analysis-r2.md](SPEC/2026-05/rag-agent-integration/F02-collection-tools/01-analysis-r2.md#L277), but [src/config.ts](src/config.ts#L301) and [src/config.ts](src/config.ts#L335) expose path/load helpers, not a save helper. The proposed `RAG_PERSIST_FAILED` behavior intentionally leaves manager state updated when config persistence fails [SPEC/2026-05/rag-agent-integration/F02-collection-tools/01-analysis-r2.md](SPEC/2026-05/rag-agent-integration/F02-collection-tools/01-analysis-r2.md#L343), which creates a split-brain runtime/config/registry state. That is not architecture-first.

2. The canonical tool list and schemas are internally inconsistent.

   The operations table says collection drop is multiplexed under `rag_admin`, while the canonical list includes a separate `rag_drop` and later explains why drop is separate [SPEC/2026-05/rag-agent-integration/F02-collection-tools/01-analysis-r2.md](SPEC/2026-05/rag-agent-integration/F02-collection-tools/01-analysis-r2.md#L86) [SPEC/2026-05/rag-agent-integration/F02-collection-tools/01-analysis-r2.md](SPEC/2026-05/rag-agent-integration/F02-collection-tools/01-analysis-r2.md#L90) [SPEC/2026-05/rag-agent-integration/F02-collection-tools/01-analysis-r2.md](SPEC/2026-05/rag-agent-integration/F02-collection-tools/01-analysis-r2.md#L93). F03 consumes the separate-tool list [SPEC/2026-05/rag-agent-integration/F03-librarian-agent/01-analysis-r2.md](SPEC/2026-05/rag-agent-integration/F03-librarian-agent/01-analysis-r2.md#L84), so F02 needs one canonical list used by the operation decision table, schemas, role filters, F03 grants, tests, and documentation. If `rag_drop` exists, remove all claims that drop is an `rag_admin` action.

   The `rag_query` schema also says truncated hit text can be recovered as full text "via `rag_admin`" [SPEC/2026-05/rag-agent-integration/F02-collection-tools/01-analysis-r2.md](SPEC/2026-05/rag-agent-integration/F02-collection-tools/01-analysis-r2.md#L208), but no `rag_admin` action reads full text [SPEC/2026-05/rag-agent-integration/F02-collection-tools/01-analysis-r2.md](SPEC/2026-05/rag-agent-integration/F02-collection-tools/01-analysis-r2.md#L297). Either remove that promise or add a deliberately permissioned action and account for it in the tool count and permission table.

3. `delete_record` is not correctly specified.

   The proposed implementation ingests an empty records item and asserts that this deletes all chunks at `record_path` [SPEC/2026-05/rag-agent-integration/F02-collection-tools/01-analysis-r2.md](SPEC/2026-05/rag-agent-integration/F02-collection-tools/01-analysis-r2.md#L315). The current pipeline deletes old chunks for changed paths, but then writes a new file-state entry for that path, with no explicit tombstone semantics and no result proving that a record was deleted [src/rag/pipeline.ts](src/rag/pipeline.ts#L256) [src/rag/pipeline.ts](src/rag/pipeline.ts#L281). The direct deletion primitives live on the store seam, not on `RagManager` or `Dataset` methods [src/rag/store/index.ts](src/rag/store/index.ts#L17). The analysis must choose a clean design: either declare single-record deletion unsupported by the F02 public tool layer, or intentionally use an existing public store seam through `Dataset` and delete both chunks and file state. The current empty-ingest trick is too implicit and leaves inconsistent state.

4. Register-with-watch does not satisfy the topic's automatic indexing requirement.

   The topic requires `rag.register` with `sources` and `watch` mode to arm the watcher or polling path. The analysis includes `sources` and `watch` fields [SPEC/2026-05/rag-agent-integration/F02-collection-tools/01-analysis-r2.md](SPEC/2026-05/rag-agent-integration/F02-collection-tools/01-analysis-r2.md#L265) [SPEC/2026-05/rag-agent-integration/F02-collection-tools/01-analysis-r2.md](SPEC/2026-05/rag-agent-integration/F02-collection-tools/01-analysis-r2.md#L266), but says `rag_register` only calls `manager.register(...)` [SPEC/2026-05/rag-agent-integration/F02-collection-tools/01-analysis-r2.md](SPEC/2026-05/rag-agent-integration/F02-collection-tools/01-analysis-r2.md#L273). `Dataset.open()` does not arm a watcher; watcher control is explicit through `dataset.watch()` [src/rag/dataset.ts](src/rag/dataset.ts#L89) [src/rag/dataset.ts](src/rag/dataset.ts#L149). If F02 keeps watcher arming as a later `rag_admin action: "watch_arm"`, it no longer meets the stated automatic-insertion requirement. The analysis must specify whether `rag_register` with `watch !== false` calls `dataset.watch()`, whether it runs an initial reconcile, and what gets persisted when arming fails.

5. Operator authorization is not a real contract.

   The analysis says handlers check `ctx.role === undefined || ctx.role === "operator"` as defense in depth [SPEC/2026-05/rag-agent-integration/F02-collection-tools/01-analysis-r2.md](SPEC/2026-05/rag-agent-integration/F02-collection-tools/01-analysis-r2.md#L133). `ToolCallContext.role` is required and typed as `AgentRole`, and the roster has no `operator` role [src/mcp/toolContext.ts](src/mcp/toolContext.ts#L17) [src/mcp/toolContext.ts](src/mcp/toolContext.ts#L19) [src/agents/roster.ts](src/agents/roster.ts#L352). `applyToolFilter` is name-only and only sees agent-visible tools [src/agents/tool-filters.ts](src/agents/tool-filters.ts#L42). F02 needs a concrete operator path: for example, an explicit no-context operator invocation contract for the CLI, a dedicated non-agent context type, or no operator write path until F03 grants Librarian access. As written, the document simultaneously says operators can use the same tools and that direct typed invocation by an operator is impossible.

6. Error and failure handling is incomplete and partly inaccurate.

   The document covers the public RAG error classes in [src/rag/errors.ts](src/rag/errors.ts#L16), but it conflates handler result shape with the content envelope by writing `{ error: ..., isError: true }` as the envelope [SPEC/2026-05/rag-agent-integration/F02-collection-tools/01-analysis-r2.md](SPEC/2026-05/rag-agent-integration/F02-collection-tools/01-analysis-r2.md#L323). Existing handlers return `{ content: { error: ... }, isError: true }`, and `McpRuntime.callTool()` then throws a JSON-wrapped `Error` [src/mcp/runtime.ts](src/mcp/runtime.ts#L178). The analysis also needs to distinguish true `RagError` mappings from F02-owned validation/authorization/config codes.

   The `RAG_SECRET_DROPPED` mapping is also not supported as described. The analysis says single-record `rag_add` surfaces `SecretDroppedError` as `RAG_SECRET_DROPPED` [SPEC/2026-05/rag-agent-integration/F02-collection-tools/01-analysis-r2.md](SPEC/2026-05/rag-agent-integration/F02-collection-tools/01-analysis-r2.md#L341), but the current ingest pipeline calls `scanChunk()`, increments `chunksDroppedSecrets`, and continues [src/rag/pipeline.ts](src/rag/pipeline.ts#L213) [src/rag/pipeline.ts](src/rag/pipeline.ts#L214). If `rag_add` should fail on secret-shaped content, the tool handler must pre-scan or translate an all-dropped report into an error; otherwise F03's secret-incident behavior should key off `IngestReport.chunksDroppedSecrets` for both single and bulk ingests.

   The topic explicitly asks for flood and watcher-unavailable behavior. Watcher floods are currently logged and dropped asynchronously, not returned as tool errors [src/rag/watcher/controller.ts](src/rag/watcher/controller.ts#L188). Async chokidar `ENOSPC` errors after `arm()` are also logged rather than thrown [src/rag/watcher/controller.ts](src/rag/watcher/controller.ts#L142). F02 must say what an agent sees for these cases: immediate `watch_arm` failure, later telemetry/log event, a status field in `rag_stats`, or no caller-visible signal. The current taxonomy omits that distinction.

7. The path/secret guard story uses the wrong canonical surface.

   The topic requires every ingest path to run through `shouldSkipPath` plus `scanChunk`. The analysis repeatedly names `isBlockedPath` for explicit record metadata and directory roots [SPEC/2026-05/rag-agent-integration/F02-collection-tools/01-analysis-r2.md](SPEC/2026-05/rag-agent-integration/F02-collection-tools/01-analysis-r2.md#L231) [SPEC/2026-05/rag-agent-integration/F02-collection-tools/01-analysis-r2.md](SPEC/2026-05/rag-agent-integration/F02-collection-tools/01-analysis-r2.md#L250). RAG's guard module is [src/rag/security/secrets.ts](src/rag/security/secrets.ts#L59), which delegates to the app-wide blocked-path scanner and adds RAG-specific blocked globs. Directory ingest reaches it through the walker, but `rag_add` path metadata and any pre-walk source-root checks must use `shouldSkipPath`, not only `isBlockedPath`; content safety relies on `scanChunk` [src/rag/security/secrets.ts](src/rag/security/secrets.ts#L71).

8. Concurrency is under-specified for the mutating control plane.

   The ingest lock story is mostly correct for `rag_add`, `rag_ingest`, and reconcile [src/rag/lock.ts](src/rag/lock.ts#L28). The document then dismisses `register` versus `drop` races as operator-only and serialised at the CLI level [SPEC/2026-05/rag-agent-integration/F02-collection-tools/01-analysis-r2.md](SPEC/2026-05/rag-agent-integration/F02-collection-tools/01-analysis-r2.md#L379), but no such serialization exists in the MCP runtime, and F03 grants the future Librarian the same mutating tools. The design needs a service-level mutex or explicit conflict behavior for register/drop/persist operations, including registry writes and config writes.

9. Bootstrapping, file layout, and telemetry omit required dependencies.

   The file list adds `src/mcp/rag.ts` and registers it in builtins [SPEC/2026-05/rag-agent-integration/F02-collection-tools/01-analysis-r2.md](SPEC/2026-05/rag-agent-integration/F02-collection-tools/01-analysis-r2.md#L404), but `registerBuiltinServices()` currently receives only MCP and security config, not the full `rag` config or provider options needed to create a `RagManager` [src/mcp/builtins.ts](src/mcp/builtins.ts#L1912). The analysis should specify where the singleton RAG service is constructed, how it receives project root/project id/provider options, how disabled RAG is represented, and how tests inject a fake manager without changing public RAG APIs.

   The logging section covers synchronous tool invocations [SPEC/2026-05/rag-agent-integration/F02-collection-tools/01-analysis-r2.md](SPEC/2026-05/rag-agent-integration/F02-collection-tools/01-analysis-r2.md#L394), but the topic also asks what is recorded for automatic re-ingest and failed ingest. The current watcher default logs to `console.warn`, not the shared `log` buffer, unless a logger is injected [src/rag/watcher/controller.ts](src/rag/watcher/controller.ts#L60). F02 must specify how watcher-driven re-ingest, floods, async watcher errors, and failed background ingests reach `src/log.ts` without exposing query text or record content.

10. Style is mostly compliant, but the document's self-consistency check overclaims.

   The document is self-contained and uses repo-root-relative links. It does not rely on review-process back references. The final internal consistency check claims every section uses one canonical tool list and that persistence semantics are mutually consistent [SPEC/2026-05/rag-agent-integration/F02-collection-tools/01-analysis-r2.md](SPEC/2026-05/rag-agent-integration/F02-collection-tools/01-analysis-r2.md#L414), but the findings above contradict both claims. That section should be rewritten after the architecture is made coherent.

## Required Changes

1. Resolve runtime registration and persistence against the current `RagManager` API without changing `src/rag/`: specify whether the tool service maintains its own dynamic dataset map, persists config before registering, recreates the manager after config writes, or drops runtime-only registration. The chosen design must make `rag_register` followed by `rag_ingest`, `rag_query`, `rag_stats`, and `rag_drop` work predictably.

2. Replace the split-brain `persist: true` behavior with atomic tool-level semantics. If persistence fails, the runtime manager, registry, and config must not intentionally diverge. Also name the actual config-write implementation to add, since no helper exists today.

3. Fix the canonical tool list so the operation table, schemas, permissions, F03 whitelist, file list, tests, and docs all agree on exactly eight or fewer tools. In particular, decide whether drop is `rag_drop` or an `rag_admin` action, not both, and remove schema promises such as full-text recovery via `rag_admin` unless the action actually exists.

4. Rework `delete_record` into a clean, explicit decision. Either remove it from the tool surface/admin actions as unsupported under the fixed public RAG surface, or specify a correct existing-seam implementation that deletes chunks and file state and reports the actual deletion count.

5. Specify automatic indexing on register. `rag_register` with `sources` and `watch !== false` must either arm the watcher/polling path and define initial reconcile/error semantics, or the document must justify why the topic's automatic insertion requirement is deferred elsewhere.

6. Rewrite authorization around the real `ToolCallContext` and roster. Do not rely on a non-existent `operator` role or an optional `role` inside a typed context. Define the CLI/operator invocation contract and the future Librarian grants in the same canonical permission table.

7. Correct the path and secret guard requirements to use `shouldSkipPath` and `scanChunk` for every explicit, directory, and watcher-driven ingest path. Keep `isBlockedPath` only as an implementation detail inside the RAG guard.

8. Complete the error taxonomy and failure surface. Separate handler return shape from content envelopes, map each `RagError` subclass exactly once, define whether single-record secret drops are errors or reports, and define caller-visible behavior for disabled RAG, watcher-disabled generic errors, watcher-unavailable, watcher floods, async watcher errors, provider failures, lock contention, and persistence failures.

9. Add the missing concurrency and lifecycle design for register/drop/persist/watch operations. The design needs a service-level lock or conflict response for control-plane mutations and must state how it interacts with the existing per-dataset ingest lock.

10. Expand the implementation/file plan to include RAG service construction and dependency injection from bootstrap/builtins, config persistence helpers, focused tests for runtime register followed by ingest/query/drop, protected-dataset mutation denial, watcher arm/disarm/reconcile and register-with-watch, watcher telemetry into `log`, secret-drop behavior, and persistence failure rollback.
