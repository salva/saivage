# F01 - Markdown Table Rendering - Analysis Review r1

## Findings

1. Section 6's test-coverage claim is materially wrong. The analysis says "No file under `web/` defines `describe(...)`, `it(...)`, or imports `vitest`", but a fresh precise grep finds existing web Vitest tests: `web/src/composables/useWebSocket.test.ts`, `web/src/composables/useAuthState.test.ts`, `web/src/components/agents/timeline.test.ts`, and `web/src/components/agents/round-id.test.ts`. The markdown-specific conclusion is still true: grepping test files for `renderMarkdown`, `FormattedContent.vue`, `ChatWindow.vue`, or `markdown` finds only the unrelated RAG chunker references in `tests/rag/e2e-drift.test.ts` and `tests/rag/e2e-ingest-query.test.ts`. Fix: rewrite Section 6 to say the web package already has Vitest tests, but none cover the markdown renderer or its consuming components; include the fresh grep commands/results instead of the generated-bundle claim.

2. The dependency landscape overstates where markdown/parser/sanitizer packages exist. It is correct that `web/package.json` has no direct markdown or sanitizer dependency, and exact source-import grep found no `markdown-it`, `marked`, `micromark`, `dompurify`, or `sanitize-html` imports in `web/src`, `src`, or `docs`. However, `package-lock.json` already contains transitive `markdown-it`, `marked`, `micromark`, GFM micromark/mdast packages, `markdown-table`, and `dompurify`. `markdown-it` is not only a VitePress transitive; `typedoc` also depends on it. `marked` and `dompurify` appear through `mermaid`, and GFM micromark/mdast packages appear through `telegramify-markdown`. Fix: distinguish direct web runtime dependencies from root/dev/backend transitive lockfile packages, and do not claim `markdown-it` enters only through VitePress.

3. Several cited line numbers are stale or point at the wrong symbol. Current line-number checks show `package.json#L43` is `proper-lockfile`, while `telegramify-markdown` is on line 45; `package.json#L66` is `typescript`, while `typedoc-plugin-markdown` is on line 64; `package.json#L70` is `vitest`, while `vitepress` is on line 68. The root Vitest/happy-dom setup citation should point to the test scripts on lines 17-18, `happy-dom` on line 59, and `vitest` on line 70, not lines 18/57/72. `FormattedContent.vue#L82` is not the `white-space` declaration; `.formatted-text { white-space: pre-wrap; }` is on lines 83-84. `web/package.json#L11-L15` should be `web/package.json#L12-L16` for the runtime dependencies. Fix: refresh every `package.json`, `web/package.json`, and `FormattedContent.vue` line citation before approval.

4. The unsupported-syntax list is directionally right, but two entries describe the actual renderer behavior incorrectly. Task-list lines like `- [ ] item` and `- [x] item` do match the generic unordered-list rule in `web/src/utils/markdown.ts` lines 19-22, so they render as a normal bullet with `[ ]` or `[x]` in the text; they are unsupported as checkboxes, but not left fully raw. Indented nested bullets do not get their leading whitespace "baked into the bullet text"; because the regexes are anchored to `^[-*] ` and `^(\d+)\. ` with no leading whitespace allowance, nested lines remain raw unchanged. Fix: reword these entries to describe the partial task-list transformation and the raw rendering of indented nested list lines.

5. Section 4 overstates the table parser limitation by saying alignment and header/body split are "unrecoverable." The observed fact is narrower: the current `renderMarkdown` implementation has no pipe-row rule and no existing multi-line table state. A future block-level pre-pass or parser replacement could recover the header row, separator row, and alignment markers from the original text. Fix: replace "unrecoverable" with a code-derived constraint such as "requires multi-line parsing/buffering before or around the current line-by-line replacements."

6. Section 8 smuggles in a design policy that is not derived from the observed code. The "No backward compatibility shims" bullet cites user memory/preferences and then prescribes removing the regex pipeline wholesale if the renderer is replaced; that is a project/process preference, not a factual constraint from `web/src/utils/markdown.ts`, `ChatWindow.vue`, or `FormattedContent.vue`. The same bullet also says class names and DOM shape can change freely, which is a design conclusion and should not be in the analysis constraints. Fix: remove this bullet from the factual analysis or move it to a separate proposal-policy section; keep only constraints derived from code, such as preserving the `v-html` XSS posture, JSON-vs-text routing, current syntax behavior, and the two scoped CSS consumers.

7. The bundle-size bullets in Section 7 are unverified external facts, and one is likely misleading for proposal work. The analysis gives current-size estimates for `marked`, `markdown-it`, and `micromark`, but these numbers are not verified against the repository, lockfile, or a reproducible measurement command. It also says `markdown-it` needs optional plugins for GFM tables, while the repository evidence here only establishes whether the package is declared or present transitively, not its table feature surface. Fix: remove bundle-size estimates from the factual analysis or move them to the proposal with sources or a repeatable measurement method.

8. Open question 2 is too broad for this analysis and repeats unverified data. It asks whether the backend could pre-render markdown and includes the same size estimates, even though the analysis scope is the web renderer and the cited backend markdown packages were already ruled out of the web UI path. Fix: make the question answerable as a proposal choice: "Should the web renderer remain no-dependency or add an explicit web dependency, and what sanitizer/HTML-disabled configuration preserves the current XSS model?"

Verified checks with no requested change: the escape-first/XSS model is accurately described for the current renderer (`&`, `<`, and `>` are escaped before literal renderer-owned tags are emitted, and captured text is not interpolated into attributes), and the `white-space: pre-wrap` behavior exists in both consumers: `.msg-content` in `ChatWindow.vue` and `.formatted-text` in `FormattedContent.vue`.
