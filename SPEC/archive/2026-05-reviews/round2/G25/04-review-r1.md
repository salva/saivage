## Findings

1. The new typed error would be masked during boot-time coverage validation. The design says production callers let resolver failures propagate and the new error remains distinct from `MissingModelForRoleError` at [SPEC/v2/review-2026-05-round2/G25/01-analysis-r1.md#L27-L31](01-analysis-r1.md#L27-L31) and [SPEC/v2/review-2026-05-round2/G25/02-design-r1.md#L108-L109](02-design-r1.md#L108-L109). That is not true for the main startup path: `bootstrap()` constructs the resolver and immediately calls `validateModelCoverage` at [src/server/bootstrap.ts#L130-L136](../../../../src/server/bootstrap.ts#L130-L136), and `validateModelCoverage` currently catches every `routing.resolve(role)` error and later throws `MissingModelForRoleError` at [src/config-validation.ts#L47-L69](../../../../src/config-validation.ts#L47-L69). If G25 adds `NoAllowedRouteMatchError` but leaves that catch-all intact, an allow-list mismatch for a required role will be reported as a generic missing-model coverage failure, losing the `kind`, `role`, `candidates`, and `allowed` payload the design relies on. The plan only appends the new class to [src/config-validation.ts#L11-L21](../../../../src/config-validation.ts#L11-L21) via [SPEC/v2/review-2026-05-round2/G25/03-plan-r1.md#L15-L39](03-plan-r1.md#L15-L39); it must also narrow the validator catch behavior.

2. The proposed tests prove the class is thrown, but not the contract that matters. The plan's model and account failure cases use only `toThrow(NoAllowedRouteMatchError)` at [SPEC/v2/review-2026-05-round2/G25/03-plan-r1.md#L84-L162](03-plan-r1.md#L84-L162). That misses the typed payload required by the design shape at [SPEC/v2/review-2026-05-round2/G25/02-design-r1.md#L7-L28](02-design-r1.md#L7-L28): `kind`, `role`, `candidates`, `allowed`, and `configPath`. It also does not isolate the most important account wrinkle from the critique prompt. The explicit-account failure test includes a provider default outside the allow-list, but it would still fail even if the implementation forgot to treat `defaultAccount` as a candidate. Add a case where there is no explicit account, the provider has `defaultAccount: "user-a"`, and `allowed_accounts` contains only `"github-copilot.user-b"`; that must throw `NoAllowedRouteMatchError` with `kind: "account"`, `candidates: ["github-copilot.user-a"]`, and `allowed: ["github-copilot.user-b"]`.

3. The account no-candidate fallback should be explicitly tested if the writer keeps it. The design introduces the accounts analogue of F04 by returning `unique(allowed)` when `allowed_accounts` is configured and there are no explicit/default candidates at [SPEC/v2/review-2026-05-round2/G25/03-plan-r1.md#L72-L73](03-plan-r1.md#L72-L73). Current tests have no `allowed_accounts`-only case at all; the only account assertions are profile resolution and auth-profile pinning at [src/routing/resolver.test.ts#L30-L99](../../../../src/routing/resolver.test.ts#L30-L99). Add a small regression test for this intended no-candidate branch, or explicitly drop that branch from the design. Leaving it untested makes it too easy to regress the very distinction this issue is about.

## Required Changes

1. Update [src/config-validation.ts#L47-L69](../../../../src/config-validation.ts#L47-L69) so `validateModelCoverage` does not collapse `NoAllowedRouteMatchError` or unrelated resolver bugs into `MissingModelForRoleError`. The cleanest shape is to catch and aggregate only `MissingModelForRoleError`, then rethrow every other error.

2. Add a config-validation regression test showing that a required role whose candidates are filtered out by `allowed_models` surfaces `NoAllowedRouteMatchError` through `validateModelCoverage` instead of being reported as missing model coverage.

3. Strengthen the resolver failure tests in [SPEC/v2/review-2026-05-round2/G25/03-plan-r1.md#L84-L162](03-plan-r1.md#L84-L162) to inspect the thrown error object and assert `kind`, `role`, `candidates`, `allowed`, and `configPath`, not only the class.

4. Add an account-specific default-candidate test: with no explicit account, a provider `defaultAccount` outside `allowed_accounts` must throw `NoAllowedRouteMatchError` rather than returning the bare allow-list.

5. Add or remove the intended `allowed_accounts`-only no-candidate fallback test. If the design keeps the fallback, test it; if the design decides accounts should be stricter than models, revise the account proposal and acceptance criteria so the asymmetry is deliberate.
