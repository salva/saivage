# G34 — Review r2

**Reviewer**: GPT-5.5

## Blocking findings

1. **`fetchWithTimeout` has no valid success-path timer cleanup.** The design creates a real `setTimeout` and deliberately keeps it alive after `fetch()` resolves so it can cover the body read [SPEC/v2/review-2026-05-round2/G34/02-design-r2.md](02-design-r2.md#L88-L102), but the only success-path cleanup is `response.body?.["finally"]?.(() => clearTimeout(timer))` [SPEC/v2/review-2026-05-round2/G34/02-design-r2.md](02-design-r2.md#L97-L102). A WHATWG `ReadableStream` does not expose `finally`, and TypeScript should reject that indexed access unless the implementation widens the type unsafely. More importantly, neither `discardBody` nor `readBounded*` receives any cleanup callback, so every successful fetch/read can leave a live timer until `mcp.fetchTimeoutMs` expires. That keeps the event loop open in tests and can flip `timedOut`/abort a completed response later. Make cleanup explicit: return a `dispose()`/`clearTimeout()` function in `TimedFetch`, or have `fetchWithTimeout` own the read/discard operation. Every caller path that consumes, cancels, or rejects the body must clear the timer in `finally`, and the helper tests should assert a successful short response does not wait for the timeout.

2. **Mid-body timeout can still be misreported as a successful partial body.** Round 2 threads a timeout signal into the bounded reader, which is the right direction, but the reader's abort listener calls `reader.cancel(signal?.reason)` [SPEC/v2/review-2026-05-round2/G34/02-design-r2.md](02-design-r2.md#L143-L148), while the loop treats `done` as normal EOF and returns the accumulated prefix [SPEC/v2/review-2026-05-round2/G34/02-design-r2.md](02-design-r2.md#L153-L175). If cancellation resolves a pending `reader.read()` with `{ done: true }`, the handler never reaches the catch that calls `classifyNetworkError(..., { timedOut: timedOut() })` [SPEC/v2/review-2026-05-round2/G34/02-design-r2.md](02-design-r2.md#L543-L550), so a slow/stalled upstream can produce `isError: false` with partial data instead of `code: "TIMEOUT"`. The bounded reader needs to throw on `signal.aborted` before returning, including after a `done` result caused by the abort. The mid-body timeout tests should cover the exact handler result, not only `timedOut() === true` inside the helper.

3. **The UTF-8 truncation contract is based on incorrect `TextDecoder` behavior.** The design says one-shot `new TextDecoder("utf-8", { fatal: false }).decode(...)` drops a partial UTF-8 sequence at the tail [SPEC/v2/review-2026-05-round2/G34/02-design-r2.md](02-design-r2.md#L127-L132), and the test expects no replacement character when a 3-byte rune is cut by `max_bytes` [SPEC/v2/review-2026-05-round2/G34/02-design-r2.md](02-design-r2.md#L803-L807). With `fatal: false` and a final decode, incomplete trailing bytes are replaced with U+FFFD, not silently dropped. Either trim an incomplete terminal UTF-8 sequence before decoding, or decode with streaming semantics that intentionally leave the incomplete tail buffered and never flush it. As written, the proposed test gate cannot pass for a cap that lands inside a multibyte character.

## Required corrections

- The `discardBody` fast-fail direction is correct: `downloadUrl` now cancels on both `!response.ok` and over-limit `Content-Length` [SPEC/v2/review-2026-05-round2/G34/02-design-r2.md](02-design-r2.md#L379-L393), and `fetch_url` cancels the HTTP-error body [SPEC/v2/review-2026-05-round2/G34/02-design-r2.md](02-design-r2.md#L529-L541). Keep that shape, but wire timeout cleanup into the same early-exit paths.
- The `DownloadOutcome` union closes the round-1 structured-download gap in principle [SPEC/v2/review-2026-05-round2/G34/02-design-r2.md](02-design-r2.md#L328-L335), and the handlers propagate `outcome.failure` to the top-level envelope [SPEC/v2/review-2026-05-round2/G34/02-design-r2.md](02-design-r2.md#L642-L660). Preserve that direction.
- The no-shim rename is now explicit: config moves from live `maxFetchChars` [src/config.ts](../../../../src/config.ts#L143) to `maxFetchBytes` [SPEC/v2/review-2026-05-round2/G34/02-design-r2.md](02-design-r2.md#L253-L268), and tool args move from live `args.max_chars` [src/mcp/builtins.ts](../../../../src/mcp/builtins.ts#L764-L797) to `args.max_bytes` [SPEC/v2/review-2026-05-round2/G34/02-design-r2.md](02-design-r2.md#L515). That satisfies the architecture-first/no-compat requirement.
- G33 direction is fixed: the plan now makes G34 land before the G33 helper swap [SPEC/v2/review-2026-05-round2/G34/03-plan-r2.md](03-plan-r2.md#L16-L18), matching the round-1 review's ownership decision.
- One small anchor cleanup remains: the design says the `fetch_url` tool schema update is in [SPEC/v2/review-2026-05-round2/G34/03-plan-r2.md](03-plan-r2.md#L40), but that line is the config rename step, not the dataTools schema instruction. Fix that while revising the timeout helper.

## Summary

Round 2 resolves the main architectural direction: G34 owns the shared helper, G33 depends on it, download failures become structured, and the byte-cap rename is deliberately breaking. I cannot approve yet because the proposed timeout helper can leak timers, can convert a timeout into a normal partial read, and relies on false UTF-8 decoder semantics.
