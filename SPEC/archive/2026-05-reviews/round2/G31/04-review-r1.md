# G31 — Review r1

**Reviewer**: GPT-5.5
**Verdict**: CHANGES_REQUESTED

The direction is right: Proposal A puts the cap in `mcp.maxFileReadBytes`, follows the existing config-to-module cap pattern, rejects a premature stash/policy module, and keeps G31 scoped to `read_file`. That is the correct architecture for this finding. I cannot approve r1 yet because the proposed mechanics and tests still have several concrete mismatches with the runtime contract and with the stated binary/window semantics.

## Findings

1. **Windowed reads ignore `bytesRead`, so the returned envelope can lie and the binary probe can fire on zero-filled bytes.** The proposed handler allocates `toRead`, calls `handle.read`, then discards the returned byte count [02-design-r1.md](02-design-r1.md#L266-L270). The risk section says the handler reports the true byte count after a concurrent truncation [02-design-r1.md](02-design-r1.md#L400-L403), but the code actually returns `length: toRead` and computes `truncated` from `toRead` [02-design-r1.md](02-design-r1.md#L293-L300). In Node, `Buffer.alloc(toRead)` is zero-filled; if the file shrinks between `stat` and `read`, unread tail bytes remain `0`, making the NUL probe report `BINARY_CONTENT` for an ordinary truncated text file. Required change: capture `{ bytesRead }`, slice the buffer to `bytesRead`, decode only that slice, return `length: bytesRead`, and make the race discussion match the implementation.

2. **The new error tests do not match `McpRuntime.callTool`.** The plan's G31 tests await returned payloads for `FILE_TOO_LARGE`, `LENGTH_TOO_LARGE`, `INVALID_RANGE`, and `BINARY_CONTENT` [03-plan-r1.md](03-plan-r1.md#L158-L204). But `McpRuntime.callTool` throws whenever an in-process handler returns `isError: true`, and only returns `result.content` on success [src/mcp/runtime.ts](../../../../src/mcp/runtime.ts#L188-L193). The existing happy-path test is written against that unwrapped success contract [src/mcp/builtins.test.ts](../../../../src/mcp/builtins.test.ts#L71-L72). Required change: either assert `rejects.toThrow` with the serialized code/context in the message, or add a narrow lower-level test harness for the filesystem handler. The current seven-test plan will fail as written.

3. **Structured error-code coverage is incomplete.** The design claims every error path returns a stable machine-readable `code` [02-design-r1.md](02-design-r1.md#L337-L341), but malformed `offset` / `length` values go through `parseNonNegativeInt`, which throws a raw `Error` [02-design-r1.md](02-design-r1.md#L194-L195), [02-design-r1.md](02-design-r1.md#L312-L319). The plan even expects the invalid-offset case to throw a plain string [03-plan-r1.md](03-plan-r1.md#L220). That leaves input-validation failures outside the envelope the design says agents can rely on. Required change: define a structured code such as `INVALID_ARGUMENT` for non-number, non-finite, negative, or non-integer window arguments, and make the tests assert that code through the runtime's thrown error message.

4. **The 4 KiB NUL-byte probe is underspecified for windowed reads.** The design says the handler performs a fast binary check on the first 4 KiB [02-design-r1.md](02-design-r1.md#L17-L24), but the implementation probes the first 4 KiB of the returned buffer [02-design-r1.md](02-design-r1.md#L275-L282). Those are different once `offset` is non-zero: a binary file with a NUL in the file header can pass if the requested window starts later, while a text file that contains a NUL outside the requested window could be treated differently depending on which interpretation is intended. Required change: choose and document one rule. If the rule is “file-head heuristic,” read/probe the file head independently of the requested window and add a non-zero-offset regression test. If the rule is “returned-window heuristic,” rewrite the design prose and error text so it no longer claims a file-head probe.

5. **Offset semantics contradict the schema.** The proposed schema says `offset` must be less than the file size [02-design-r1.md](02-design-r1.md#L167), but the handler only rejects `effectiveOffset > st.size` [02-design-r1.md](02-design-r1.md#L225-L235). That means `offset === size` returns an empty success. Either behavior is defensible, but r1 needs one contract. Required change: decide whether EOF windows are valid and align the schema text, `INVALID_RANGE` branch, and tests.

6. **Plan anchors and pre-flight need to be reworked around the actual G30 state.** The analysis correctly says G30 is approved but not landed in live source [01-analysis-r1.md](01-analysis-r1.md#L121-L133). The live file still imports sync fs helpers [src/mcp/builtins.ts](../../../../src/mcp/builtins.ts#L18-L25) and still implements `read_file` with `readFileSync` [src/mcp/builtins.ts](../../../../src/mcp/builtins.ts#L274-L276), while the approved G30 note names scanner infrastructure that is not present in the live tree yet [../G30/APPROVED.md](../G30/APPROVED.md#L7). The plan is right to stop if G30 has not landed [03-plan-r1.md](03-plan-r1.md#L12-L16), but it should explicitly say to re-anchor after G30 lands. Also, the helper insertion anchor is off in live source: `parseOptionalTimeoutMs` is currently at [src/mcp/builtins.ts](../../../../src/mcp/builtins.ts#L375), not the line cited by the plan [03-plan-r1.md](03-plan-r1.md#L70-L76). Required change: refresh the implementation anchors after G30, and do not present current line numbers as stable.

7. **The verification checklist contains at least one false expected result.** The plan says `grep -c "MAX_FILE_READ_BYTES"` should return 3 [03-plan-r1.md](03-plan-r1.md#L110), but the proposed handler references the symbol in multiple error branches, max-byte fields, and range calculations [02-design-r1.md](02-design-r1.md#L197-L207), [02-design-r1.md](02-design-r1.md#L241-L258). That exact count cannot hold. Required change: replace that check with targeted greps for declaration, register-time assignment, and the two main behavior codes, or drop the exact count.

8. **The import/dead-code note is inaccurate post-G30.** The plan says G31 will reuse `readFile`, `stat`, and `open` from G30 [03-plan-r1.md](03-plan-r1.md#L99-L101), and the design repeats that claim [02-design-r1.md](02-design-r1.md#L322-L323), [02-design-r1.md](02-design-r1.md#L369-L373). The proposed replacement handler uses `stat` and `open`, but not `readFile` [02-design-r1.md](02-design-r1.md#L213-L270). If G30 only introduced `readFile` for the old async whole-file body, G31 must remove that now-unused import rather than preserve dead code.

## What Looks Good

- `mcp.maxFileReadBytes` belongs in the config schema beside the existing MCP caps [src/config.ts](../../../../src/config.ts#L137-L144), and wiring it through `registerBuiltinServices` mirrors the current cap pattern [src/mcp/builtins.ts](../../../../src/mcp/builtins.ts#L1071-L1080).
- Rejecting Proposal B is correct. A new read-policy module plus auto-stash would overreach for this local builtin and would blur the dispatcher/tool boundary.
- Same-file coordination with G32–G35 is mostly acceptable as described: G31 touches the schema/handler/config-cap corridor, while the neighboring findings own search, fetch, and environment filtering ranges [01-analysis-r1.md](01-analysis-r1.md#L140-L148). The important caveat is that G30 must land first and then G31 must re-anchor.

## Required Round-2 Changes

1. Fix the windowed-read algorithm to use `bytesRead` and adjust the race section accordingly.
2. Make every new read-window validation failure structured, including malformed offset/length inputs.
3. Resolve and test the binary-probe semantics for non-zero offsets.
4. Align offset-at-EOF behavior across schema, implementation, and tests.
5. Rewrite the seven-test plan so error cases follow `McpRuntime.callTool`'s thrown-error contract, and create files in tests that do not intentionally depend on validation-before-stat ordering.
6. Refresh line anchors after G30 lands; keep G31 sequenced after G30 and ensure the G30 no-sync-fs regression remains green.
7. Correct the terminal verification checklist and remove any unused `readFile` import after the handler rewrite.

VERDICT: CHANGES_REQUESTED
