# G31 — APPROVED

**Chosen proposal**: Proposal A (per [02-design-r4.md](02-design-r4.md)) — replace the unbounded `readFileSync(fp, "utf-8")` with an async, capped, windowed read built on `node:fs/promises`. Configurable `mcp.maxFileReadBytes` (default 200_000, parity with `maxFetchChars`), optional `offset` and `length` window, structured error envelope shape `{ content, offset, length, size_bytes, truncated }` for success, and a structured error contract (`INVALID_ARGUMENT`, `FILE_TOO_LARGE`, `LENGTH_TOO_LARGE`, `INVALID_RANGE`, `BINARY_CONTENT`, `NOT_A_FILE`, `NOT_FOUND`, `PERMISSION_DENIED`, `IO_ERROR`). A 4 KiB NUL-byte probe always reads `[0, min(4096, totalSize))` from the file head independently of `offset` so binary content cannot be smuggled past the probe via a non-zero offset. An exported `classifyFsError` helper maps Node `ErrnoException` codes (ENOENT/ENOTDIR -> NOT_FOUND, EACCES/EPERM -> PERMISSION_DENIED, EISDIR -> NOT_A_FILE, default -> IO_ERROR) and is unit-tested per branch. `handle.close()` rejections are wrapped through the same classifier with a new `"close"` context and only surface when no primary read failure was recorded. Proposal B (streaming + auto-stash via `runtime/stash.ts`) is rejected as over-engineering that masks the failure signal.
