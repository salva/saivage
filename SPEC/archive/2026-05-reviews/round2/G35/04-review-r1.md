## Blocking findings

1. **`PASSWORD_PROMPT` is still classified as secret.** The required predicate contract includes rejecting `PASSWORD_PROMPT`, but the proposed `SECRET_ENV_NAME_PATTERNS` match `PASSWORD` as an underscore-bounded lexeme and the proposed `ENV_CONFIG_POINTER_SUFFIXES` do not include `_PROMPT` [02-design-r1.md](02-design-r1.md#L55-L85). The result is `true`, so a legitimate prompt/config field would still be scrubbed. The corpora do not catch this because `PASSWORD_PROMPT` is absent from the false-positive list [02-design-r1.md](02-design-r1.md#L93-L124), and the implementation plan only says to copy those corpora into [src/security/secrets.test.ts](../../../../src/security/secrets.test.ts#L1) [03-plan-r1.md](03-plan-r1.md#L50-L61). Add an explicit exemption strategy for prompt/config field names, then gate it with `PASSWORD_PROMPT` in the false-positive corpus.

2. **The force-prefix layer preserves the same broad false-positive shape for provider namespaces.** The design says config-pointer suffixes are preserved only after the force-prefix layer, and force-prefix wins for all `ANTHROPIC_`, `OPENAI_`, `GITHUB_`, `GH_`, and `TELEGRAM_` names [02-design-r1.md](02-design-r1.md#L44-L85). That means values such as `OPENAI_BASE_URL`, `ANTHROPIC_BASE_URL`, and `GITHUB_API_BASE_URL_TEMPLATE` are deliberately classified as secret, with the false-negative corpus requiring that behavior [02-design-r1.md](02-design-r1.md#L126-L152). This contradicts the analysis claim that config-pointer suffixes like URLs, paths, and endpoints are the false-positive class being fixed [01-analysis-r1.md](01-analysis-r1.md#L21-L31), and it leaves common non-secret CI/provider variables under `GITHUB_` and `GH_` scrubbed for the same reason the current live regex block is too broad [src/mcp/builtins.ts](../../../../src/mcp/builtins.ts#L400-L432). Narrow the force set to known credential names, or apply the config-pointer exemption before broad provider namespace stripping except for exact high-risk names. If the project intentionally wants provider config pointers hidden from shell children, the design needs to state that as a separate policy and test both sides explicitly.

3. **The round-1 corpora are incomplete and internally ambiguous.** The positive corpus does not include `SLACK_TOKEN`, even though the review contract calls it out; it happens to match the generic token lexeme, but the regression suite should pin it directly [02-design-r1.md](02-design-r1.md#L126-L152). The false-positive corpus also presents `GITHUB_API_BASE_URL_TEMPLATE` inside the false-positive block and then says the actual false-positive list ends at `USER_PROFILE_URL` [02-design-r1.md](02-design-r1.md#L93-L124). The plan then tells the implementer to copy the lists verbatim from that section [03-plan-r1.md](03-plan-r1.md#L50-L61), which is not a deterministic instruction. Round 2 should split the corpora into unambiguous arrays, include the required examples (`RESET_PASSWORD_URL`, `MY_SECRETARY`, `PASSWORD_PROMPT`, `ANTHROPIC_API_KEY`, `OPENAI_API_KEY`, `GITHUB_TOKEN`, `SLACK_TOKEN`), and add a few provider-prefix non-secret cases if the broad force layer is narrowed.

## Notes

- The underscore-bounded lexeme strategy is directionally right for the original substring bug: `MY_SECRETARY`, `TOKENIZER`, and `PASSWORDLESS_MODE` no longer match because the credential word is not bounded by start/end or `_` [02-design-r1.md](02-design-r1.md#L55-L63). That is a good core, but it needs the missing prompt/config exemptions above.
- The plan's live-code anchor is correct: the obsolete scrubber sits in the standalone env-filter region of [src/mcp/builtins.ts](../../../../src/mcp/builtins.ts#L400-L432), and the spawn site delegates through `filterShellEnv` immediately afterward [src/mcp/builtins.ts](../../../../src/mcp/builtins.ts#L451). Deleting `SECRET_ENV_PATTERNS` and replacing the loop with `isSecretEnvName(key)` is the right call-site shape [03-plan-r1.md](03-plan-r1.md#L31-L44).
- Co-locating the new predicate in [src/security/secrets.ts](../../../../src/security/secrets.ts#L67-L79) is reasonable; the existing security module currently owns content-shaped secret detection and blocked sensitive paths [src/security/secrets.ts](../../../../src/security/secrets.ts#L24-L79), with tests in [src/security/secrets.test.ts](../../../../src/security/secrets.test.ts#L1-L132). The existing `src/security` surface is limited to [src/security/secrets.ts](../../../../src/security/secrets.ts#L1) and [src/security/prompt-injection-cop.ts](../../../../src/security/prompt-injection-cop.ts#L1), so no new module is needed.
- Sequencing is mostly sound.
