## Blocking Findings

1. The new predicate compiler regresses the G35 r2 hyphen-key contract. G35 r2 explicitly preserved `API[_-]?KEY` and `ACCESS[_-]?KEY` so `SOME_API-KEY` stayed classified as secret [SPEC/v2/review-2026-05-round2/G35/02-design-r2.md](../G35/02-design-r2.md#L91), and the false-negative corpus pins `SOME_API-KEY` [SPEC/v2/review-2026-05-round2/G35/02-design-r2.md](../G35/02-design-r2.md#L193). G35b replaces that with a literal escaped lexeme compiled as `(?:^|_)${escaped}S?(?:$|_)` [SPEC/v2/review-2026-05-round2/G35b/02-design-r1.md](02-design-r1.md#L109-L110). With default lexeme `API_KEY`, that pattern does not match `SOME_API-KEY`; the hyphen between `API` and `KEY` is no longer accepted. This contradicts the plan's requirement to copy the G35 r2 false-negative corpus into [src/security/secrets.test.ts](../../../../src/security/secrets.test.ts) [SPEC/v2/review-2026-05-round2/G35b/03-plan-r1.md](03-plan-r1.md#L252-L255) and means the proposed implementation would either fail its own corpus or quietly drop a prior safety guarantee. Fix the compiler so configured multi-word lexemes can preserve the intended separator tolerance, then pin `SOME_API-KEY` and an `ACCESS-KEY` variant in the G35b corpus.

2. The config-mutation test proves additive plumbing, but not the full replacement semantics required by the new principle. The design rejects additive-only overrides because operators must be able to remove a bad default [SPEC/v2/review-2026-05-round2/G35b/02-design-r1.md](02-design-r1.md#L544-L564), yet the end-to-end test writes `credentialLexemes: [...DEFAULT_CREDENTIAL_LEXEMES, "PII"]` and only checks that `PII_DATA` is scrubbed [SPEC/v2/review-2026-05-round2/G35b/03-plan-r1.md](03-plan-r1.md#L311-L341). An additive-only implementation would pass that test. Add a config-level replacement assertion, for example `credentialLexemes: ["PII"]` should scrub `PII_DATA` while preserving `ANTHROPIC_API_KEY`, and `configPointerSuffixes: []` should make `RESET_PASSWORD_URL` scrubbed again. That directly proves the predicate is rebuilt from the merged config object and that the schema fields are full operator-controlled lists, not hidden defaults plus extras.

## Notes

- The main config plumbing shape is otherwise right: `security.envScrubber.credentialLexemes` and `security.envScrubber.configPointerSuffixes` are added under the Zod `security` block with defaults [SPEC/v2/review-2026-05-round2/G35b/02-design-r1.md](02-design-r1.md#L160-L177), `registerBuiltinServices` takes `config.security` and rebuilds the predicate from that resolved slice [SPEC/v2/review-2026-05-round2/G35b/02-design-r1.md](02-design-r1.md#L271-L285), and all live call sites are accounted for in the plan [SPEC/v2/review-2026-05-round2/G35b/03-plan-r1.md](03-plan-r1.md#L35-L40).
- I am not treating the default-array placement as a separate blocker, but the prose should be tightened. The lists do not literally live in every operator JSON document loaded by [src/config.ts](../../../../src/config.ts); they live as Zod-backed defaults imported from [src/security/secrets.ts](../../../../src/security/secrets.ts). That is acceptable if the contract is "schema defaults plus full replacement override", but the docs should avoid saying the values live in the config file unless the implementation also materializes them into .saivage/saivage.json.
- The schema direction is good, but add the missing negative pin for an empty string element. The proposed regexes reject empty strings, lowercase lexemes, and suffixes without a leading underscore [SPEC/v2/review-2026-05-round2/G35b/02-design-r1.md](02-design-r1.md#L168-L175), while the plan tests empty array, lowercase lexeme, and suffix without underscore [SPEC/v2/review-2026-05-round2/G35b/03-plan-r1.md](03-plan-r1.md#L282-L290). Add `credentialLexemes: [""]` or `configPointerSuffixes: [""]` so a future implementation cannot accidentally satisfy only the array-level `.min(1)`.
- Architecture-first / no-shim is satisfied in shape: [src/mcp/builtins.ts](../../../../src/mcp/builtins.ts) loses `SECRET_ENV_PATTERNS`, `filterShellEnv` delegates to the rebuilt predicate, and the old two-argument `registerBuiltinServices` call shape is removed in the same plan [SPEC/v2/review-2026-05-round2/G35b/03-plan-r1.md](03-plan-r1.md#L365-L368), [SPEC/v2/review-2026-05-round2/G35b/03-plan-r1.md](03-plan-r1.md#L389-L402).
- No regression is visible for the symmetric `allowed_accounts` fix: G35b does not touch [src/routing/resolver.ts](../../../../src/routing/resolver.ts) or the G25 resolver/config-validation surfaces. Keep that separation explicit when revising the plan.
