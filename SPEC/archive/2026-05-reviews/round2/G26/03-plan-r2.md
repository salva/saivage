  `model_overrides` token, with a separate explicit gate for the

  literal token is absent from the test source. This is belt-and-

  grep scope, the test still passes its own gate.

  rewritten so no gate runs against a file whose cleanup has not yet
  happened. The Task 3 seeder gate no longer runs over
  `src/routing/resolver.test.ts` before Task 4 cleans it.
- Daemon impact section now describes the strict-load failure for
  every seeded project on disk and the operator action required to
  remove the key (no shim, no auto-stripper, no warn-and-strip).

## Grep-gate convention (reviewer concern 1)

All grep gates use one of two scopes, never a single workspace-wide
scope that would catch the new schema-rejection test:

1. Production-source scope (excludes test files):
   ```bash
   grep -rn 'model_overrides' src/ --include='*.ts' --exclude='*.test.ts'
   ```
2. Per-file scope (named files only):
   ```bash
   grep -n 'model_overrides' src/routing/resolver.test.ts
   ```

The new schema-rejection test in
[src/types.test.ts](../../../../src/types.test.ts) is the only test
file that intentionally references the legacy key, and it does so via
a runtime-built string. Neither gate scope can fire on it.

The same scoping applies to `"legacy"` token gates: scope to
non-test source paths and add an explicit per-file gate for the
resolver test.

```ts
private resolveRuntimeDefaultModels(role: string): string[] {
  if (role === "supervisor") {
    const models = normalizeModelList(this.runtime.supervisorModel);
    if (models.length) return models;
    throw new MissingModelForRoleError([role], configPath());
  }
  if (role === "security") {
    const models = normalizeModelList(this.runtime.securityModel);
    if (models.length) return models;
    throw new MissingModelForRoleError([role], configPath());
  }
  const key = ROUTING_ROLE_TO_MODEL_KEY[role] ?? role;
  const models = normalizeModelList(this.runtime.models?.[key] ?? this.runtime.models?.default);
  if (models.length) return models;
  throw new MissingModelForRoleError([role], configPath());
}
```

```ts
describe("ProjectConfigSchema (G26 strict-mode rejection)", () => {

      routing: { roles: {}, profiles: {} },
      skills: { max_per_agent: 5 },

    fixture[LEGACY_KEY] = {};

    expect(ProjectConfigSchema.safeParse(fixture).success).toBe(false);
```

### Task 7 — Operator-gated post-deploy check

Per the workspace handoff, restarting saivage-v3 (and any other
Saivage harness) is operator-gated. G26 does not restart any daemon.
This task documents the check the operator should run before
restarting any container after G26 lands, given the daemon-impact
discussion in [01-analysis-r2.md](01-analysis-r2.md) and
[02-design-r2.md](02-design-r2.md):

```bash
ssh root@10.0.3.111 'grep -rln model_overrides /work/getrich/.saivage/ 2>/dev/null || true'
ssh root@10.0.3.112 'grep -rln model_overrides /work/saivage-v3/.saivage/ 2>/dev/null || true'
ssh root@10.0.3.113 'grep -rln model_overrides /work/diedrico/.saivage/ 2>/dev/null || true'
ssh root@10.0.3.170 'grep -rln model_overrides /work/getrich-v2/.saivage/ 2>/dev/null || true'
```

For every container that returns a file path, the operator must
edit that `config.json` and remove the `model_overrides` key by
hand before restarting `saivage.service` (or
`saivage-v3-getrich.service` on `10.0.3.170`). Per workspace rule
(no backward compatibility, no migration shim), the plan does not
ship an auto-stripper or a one-shot migration tool. The expected
failure mode if the operator restarts without removing the key is a
typed Zod `unrecognized_keys` error at startup pointing at
`model_overrides`; the daemon refuses to load the project until
the key is removed.

This is operator action, not part of the writer plan, and is gated
on the workspace handoff. The plan does not perform the edits.
