## Grep-gate convention (reviewer concern 1 from round 1)

All grep gates use one of two scopes, never a single workspace-wide
scope that would catch the new schema-rejection test:

1. Production-source scope (excludes test files):
   ```bash
   grep -rn 'model_overrides' src/ --include='*.ts' --exclude='*.test.ts'
   ```
2. Per-file scope (named files only):
   ```bash
   grep -n 'model_overrides' src/routing/resolver.test.ts
   ```

The new schema-rejection test in
[src/types.test.ts](../../../../src/types.test.ts) is the only test
file that intentionally references the legacy key, and it does so
via a runtime-built string. Neither gate scope can fire on it.

The same scoping applies to `"legacy"` token gates: scope to
non-test source paths and add an explicit per-file gate for the
resolver test.
