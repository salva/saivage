# G33 - Review r2

**Reviewer**: GPT-5.5 (Copilot)

**Reviewed docs**: analysis r2, design r2, plan r2, round-1 review, and neighboring G31/G32/G34 designs.

## Blocking findings

1. **Helper ownership now conflicts with G34, so the merge order and helper API are no longer single-valued.** G33 r2 explicitly decides that G33 owns `readBoundedTextBody`, exports it from builtins, and has G34 consume that export later in [SPEC/v2/review-2026-05-round2/G33/02-design-r2.md](02-design-r2.md#L193-L216). The G33 plan repeats that by adding an exported helper in [SPEC/v2/review-2026-05-round2/G33/03-plan-r2.md](03-plan-r2.md#L137-L142) and gating on exactly one export in [SPEC/v2/review-2026-05-round2/G33/03-plan-r2.md](03-plan-r2.md#L248-L249). G34 r1 says the opposite: it creates a new shared helper module, exports `readBoundedTextBody` from that module, and has G33 import the shared helper in [SPEC/v2/review-2026-05-round2/G34/02-design-r1.md](../G34/02-design-r1.md#L10-L26). It also specifies the G33 import swap in [SPEC/v2/review-2026-05-round2/G34/02-design-r1.md](../G34/02-design-r1.md#L501-L511) and a different order, `G31 -> G34 -> G33 r2`, in [SPEC/v2/review-2026-05-round2/G34/02-design-r1.md](../G34/02-design-r1.md#L565-L568). These plans cannot both land. Required change: reconcile ownership before approval. Given G34 now has four text/binary fetch consumers plus the timeout wrapper, prefer making G34 the owner of the shared helper module and updating G33 r2 to import it; otherwise G34 r1 must yield and be rewritten to consume G33's export.

2. **The handler-level in-process server tests still lack a concrete `web_search` endpoint seam.** The r2 handler still constructs DuckDuckGo directly in [SPEC/v2/review-2026-05-round2/G33/02-design-r2.md](02-design-r2.md#L520-L522), matching the current hardcoded production path in [src/mcp/builtins.ts](../../../../src/mcp/builtins.ts#L741-L743). Yet test rows 10-16 require a local server for success, 503, drifted markup, pre-header timeout, mid-body timeout, oversize, and closed-port failure in [SPEC/v2/review-2026-05-round2/G33/02-design-r2.md](02-design-r2.md#L696-L702). The design says the DDG hostname is replaced through an existing environment-derived test seam in [SPEC/v2/review-2026-05-round2/G33/02-design-r2.md](02-design-r2.md#L707-L715), while the plan defers the exact seam to a `PROJECT_ROOT`-style env var or test-only exported constant in [SPEC/v2/review-2026-05-round2/G33/03-plan-r2.md](03-plan-r2.md#L220-L235). The existing `withTextServer` helper only returns a URL for tools that already accept one in [src/mcp/builtins.test.ts](../../../../src/mcp/builtins.test.ts#L12-L23); it does not redirect a hardcoded `web_search` upstream. Required change: specify and anchor the actual seam, including reset behavior between tests. For example, G33 can add a private module-level search-base URL overridden only from tests, or adopt the G34 `fetchWithTimeout`/shared-helper path with an injected fetch/search URL. Without that, the min-valid cap tests are well chosen but not implementable against the registered handler.

3. **The `uddg` single-decode fixture contradicts itself, so the regression assertion is ambiguous.** The round-2 fix correctly removes the second `decodeURIComponent` call in [SPEC/v2/review-2026-05-round2/G33/02-design-r2.md](02-design-r2.md#L393-L407). But the expected fixture contract is inconsistent: section 2.4 says the nested target contains `%252B`, `%252F`, and `%2526` and should extract exactly `https://example.com/path?ref=a%2Bb%2Fc%26d` in [SPEC/v2/review-2026-05-round2/G33/02-design-r2.md](02-design-r2.md#L176-L181). The later fixture uses an `href` with `%252526d` and expects `https://example.com/path?ref=a%2Bb%2Fc%2526d` in [SPEC/v2/review-2026-05-round2/G33/02-design-r2.md](02-design-r2.md#L664-L670) and [SPEC/v2/review-2026-05-round2/G33/02-design-r2.md](02-design-r2.md#L688-L688). Those are different targets by one encoding layer around the ampersand. Required change: choose the intended target and align the prose, fixture HTML, test table, and plan. If the goal is to prove a single decode of a target URL containing encoded `+`, `/`, and `&`, the final expected URL should preserve `%2B`, `%2F`, and `%26` exactly.
