# G33 - Review r1

**Reviewer**: GPT-5.5 (Copilot)

**Reviewed docs**: analysis r1, design r1, plan r1.

## Blocking findings

1. **Body-read timeouts are still classified as network errors, so the timeout envelope and timeout test do not hold.** The handler only emits `TIMEOUT` inside the initial `fetch` catch after arming the controller in [SPEC/v2/review-2026-05-round2/G33/02-design-r1.md](02-design-r1.md#L329-L342). Once `fetch` has resolved, the body is consumed by `readBoundedTextBody` in [SPEC/v2/review-2026-05-round2/G33/02-design-r1.md](02-design-r1.md#L358-L365), and any reader rejection is unconditionally returned as `NETWORK_ERROR`. That contradicts the test that explicitly mocks a body that never closes and expects `TIMEOUT` in [SPEC/v2/review-2026-05-round2/G33/02-design-r1.md](02-design-r1.md#L473). Under Node fetch, aborting the signal while the response body is being read rejects the body read path, so this round would misreport the timeout after headers arrive. Required change: classify aborts during both initial fetch and body streaming as `TIMEOUT`, either by checking `controller.signal.aborted` in the body-read catch or by having `readBoundedTextBody` return/throw a typed abort signal.

2. **The proposed cap tests cannot use the proposed config values.** The config schema sets `webSearchTimeoutMs` to `.min(1_000)` and `webSearchMaxBytes` to `.min(64 * 1024)` in [SPEC/v2/review-2026-05-round2/G33/02-design-r1.md](02-design-r1.md#L152-L154), but the test matrix asks for `webSearchTimeoutMs: 50` and `webSearchMaxBytes: 1024` in [SPEC/v2/review-2026-05-round2/G33/02-design-r1.md](02-design-r1.md#L473-L474). If the tests go through `loadConfig`, those fixtures fail schema validation before the handler runs; if they bypass `loadConfig`, they no longer test the real registration/config path. Required change: either use min-valid values in the tests, or explicitly create a narrowly typed test-only config object after schema validation and say why that bypass is acceptable. Prefer min-valid values plus fake timers or a synthetic stream that resolves quickly after abort.

3. **The pinned parser tests call a private helper that the plan never exports.** The test surface calls `extractDdgResults` directly for the parser smoke and drift cases in [SPEC/v2/review-2026-05-round2/G33/02-design-r1.md](02-design-r1.md#L467-L471). The implementation plan only inserts file-private helpers near `stripHtml` in [SPEC/v2/review-2026-05-round2/G33/03-plan-r1.md](03-plan-r1.md#L76-L81), and today the test file imports only `registerBuiltinServices` from [src/mcp/builtins.ts](../../../../src/mcp/builtins.ts#L1071-L1079) via [src/mcp/builtins.test.ts](../../../../src/mcp/builtins.test.ts#L7). In ESM, those tests cannot reach a non-exported helper. Required change: either export `extractDdgResults` with the same explicit testing-boundary discipline G31 uses for `classifyFsError` in [SPEC/v2/review-2026-05-round2/G31/02-design-r4.md](../G31/02-design-r4.md#L41-L48), or make every fixture test go through the registered handler with a mocked `fetch` response. If `vi.spyOn` remains the mocking strategy, the plan also needs to update the current Vitest import in [src/mcp/builtins.test.ts](../../../../src/mcp/builtins.test.ts#L1).

4. **The `uddg` decoding repeats the current double-decoding bug and the fixture test is too weak to catch it.** The live handler already calls `URLSearchParams.get("uddg")` and then `decodeURIComponent` in [src/mcp/builtins.ts](../../../../src/mcp/builtins.ts#L747-L752). Round 1 copies that same pattern in [SPEC/v2/review-2026-05-round2/G33/02-design-r1.md](02-design-r1.md#L219-L223). `URLSearchParams.get` returns a decoded string, so decoding again corrupts target URLs that legitimately contain encoded percent escapes, and can skip otherwise valid results if the decoded target contains a literal malformed percent sequence. The proposed test only covers the trivial `https%3A%2F%2Fexample.com` case in [SPEC/v2/review-2026-05-round2/G33/02-design-r1.md](02-design-r1.md#L468). Required change: treat the `uddg` value returned by `searchParams.get` as the decoded target, validate it with `new URL(uddg)`, and add a pinned fixture case that preserves a nested escape such as `%252B` or `%252F` in the target query.

5. **Same-file coordination with G34 and helper ownership is contradictory.** The design says G33 will ship a private `readBoundedTextBody`, then G34 will land second and replace both copies in [SPEC/v2/review-2026-05-round2/G33/02-design-r1.md](02-design-r1.md#L504-L505). The same section then recommends `G30 -> G31 -> G32 -> G34 -> G33 -> G35`, with G33 consuming G34's shared helper in [SPEC/v2/review-2026-05-round2/G33/02-design-r1.md](02-design-r1.md#L517-L519). The plan still tells G33 to add its own helper in [SPEC/v2/review-2026-05-round2/G33/03-plan-r1.md](03-plan-r1.md#L76-L81), and the risk table still says the duplicate helper ships in G33 in [SPEC/v2/review-2026-05-round2/G33/03-plan-r1.md](03-plan-r1.md#L208). Required change: pick one merge order and one owner for the bounded body helper. If G34 lands first, G33 should reuse the shared helper and delete the private-copy plan. If G33 lands first, the recommended merge order must say so and G34 must own the later dedup.

6. **Plan anchors assume helper state that is not in the current code.** Round 1 says `parseNonNegativeInt` already exists and is reused by G31/G32 in [SPEC/v2/review-2026-05-round2/G33/02-design-r1.md](02-design-r1.md#L418-L421), but the live `web_search` path still parses `max_results` inline with `Number(...)` in [src/mcp/builtins.ts](../../../../src/mcp/builtins.ts#L737-L756), and the current file has no `parseNonNegativeInt` definition. This matters because G32's error-envelope parity depends on shared parsing semantics and stable `INVALID_ARGUMENT` handling in [SPEC/v2/review-2026-05-round2/G32/02-design-r2.md](../G32/02-design-r2.md#L454-L456). Required change: make G33 explicitly depend on the G31/G32 helper landing first, with the exact approved anchor, or include the helper addition in G33's own plan if G33 can land independently.

## Required corrections

- Fix the dependency justification. The design claims `node-html-parser` has zero transitive dependencies and is roughly 50 KB in [SPEC/v2/review-2026-05-round2/G33/02-design-r1.md](02-design-r1.md#L68-L68) and [SPEC/v2/review-2026-05-round2/G33/02-design-r1.md](02-design-r1.md#L125-L135). `npm view node-html-parser@6.1.13` reports direct dependencies on `he` and `css-select` and an unpacked size of 165,463 bytes. The dependency may still be acceptable, but the design and [package.json](../../../../package.json#L27-L39) plan need an honest footprint/rationale and a lockfile audit note.
- Strengthen the DOM-parser fixture assertions. The design claims the parser handles the `a.result__a` anchor and snippet fallback including non-anchor `.result__snippet` nodes in [SPEC/v2/review-2026-05-round2/G33/02-design-r1.md](02-design-r1.md#L211-L231), but the fixture matrix only proves a happy path and a renamed-class drift path in [SPEC/v2/review-2026-05-round2/G33/02-design-r1.md](02-design-r1.md#L467-L471). Add fixture rows for class attribute reordering, multiple classes, snippet as `<div class="result__snippet">`, missing snippet with a kept result, and malformed per-entry links incrementing `skipped` instead of failing the whole call.
- Update the tool schema details together with the handler. The live schema still advertises `max_results` as default 8 and max 20 in [src/mcp/builtins.ts](../../../../src/mcp/builtins.ts#L658-L664), while the new config default is 20 and configurable up to 50 in [SPEC/v2/review-2026-05-round2/G33/02-design-r1.md](02-design-r1.md#L152-L154). The plan only changes the outer description in [SPEC/v2/review-2026-05-round2/G33/03-plan-r1.md](03-plan-r1.md#L88-L95); it should also update the `max_results` property description and, ideally, schema metadata for the configured ceiling.

## What is solid

Proposal A is the right architectural direction: keep one real upstream, remove regex-over-HTML from the structural parser, add bounded network behavior, and return the G31/G32-style `{ content: { error, code, ...context }, isError: true }` envelope. Rejecting a provider registry until there is a second concrete provider is also consistent with the project guideline against speculative shims.
