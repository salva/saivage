The actual prompt has a fifth restart directive at [prompts/chat.md#L73](prompts/chat.md#L73): "Restart cautiously: Only restart the Planner when the user explicitly asks to restart it." After deleting the code fast path, free text falls through to the LLM path after the current intercept block at [src/agents/chat.ts#L197-L206](src/agents/chat.ts#L197-L206), so leaving this guideline in place keeps the model contract ambiguous in exactly the way r1 asked r2 to remove. The proposed negative grep at [SPEC/v2/review-2026-05-round2/G11/03-plan-r2.md#L76](SPEC/v2/review-2026-05-round2/G11/03-plan-r2.md#L76) and the validation table at [SPEC/v2/review-2026-05-round2/G11/03-plan-r2.md#L213](SPEC/v2/review-2026-05-round2/G11/03-plan-r2.md#L213) would also pass while this stale instruction remains.

2. The regex semantics section still contains incorrect JavaScript claims. R2 says the live code uses neither `i` nor `u` at [SPEC/v2/review-2026-05-round2/G11/01-analysis-r2.md#L40](SPEC/v2/review-2026-05-round2/G11/01-analysis-r2.md#L40), but the source uses the `i` flag in both regexes at [src/agents/chat.ts#L353-L354](src/agents/chat.ts#L353-L354). More importantly, R2 says `\bplanner\b` does not match `planner's` and describes the apostrophe as a word character at [SPEC/v2/review-2026-05-round2/G11/01-analysis-r2.md#L43](SPEC/v2/review-2026-05-round2/G11/01-analysis-r2.md#L43). In JavaScript, apostrophe is not a word character, so there is a word boundary between `r` and `'`; `\bplanner\b` does match the `planner` prefix in `planner's`. R1 required the regex discussion to stop carrying false semantics, and R2 explicitly claims that correction was made at [SPEC/v2/review-2026-05-round2/G11/01-analysis-r2.md#L13](SPEC/v2/review-2026-05-round2/G11/01-analysis-r2.md#L13), so this needs another pass.

3. The cross-finding grep would not catch the current bug. The plan's joint G09/G11 check at [SPEC/v2/review-2026-05-round2/G11/03-plan-r2.md#L232](SPEC/v2/review-2026-05-round2/G11/03-plan-r2.md#L232) is `git grep -nE "\b/(\b\(restart\|reset\|relaunch\)\b|/i" src/`. That pattern is shaped around a word-boundary before a slash and would not match the current live regex at [src/agents/chat.ts#L353-L354](src/agents/chat.ts#L353-L354), so it can return no matches even before G11 is fixed. This is a validation issue rather than a design blocker, but the plan should replace it with a literal or properly escaped search that actually finds the existing branch, for example one covering `tryHandleExplicitPlannerRestart`, `restartPlanner(this.localCommandContext`, and the restart/reset/relaunch regex literal.
