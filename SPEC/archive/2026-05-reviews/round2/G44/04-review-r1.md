# G44 - Review r1

## Findings

1. Medium - The design and plan still document `sendEvent` as registration-glue-only, but the live Chat agent calls that optional extension directly. The mistaken claim appears in [SPEC/v2/review-2026-05-round2/G44/02-design-r1.md](SPEC/v2/review-2026-05-round2/G44/02-design-r1.md#L23-L29) and is made more explicit in [SPEC/v2/review-2026-05-round2/G44/03-plan-r1.md](SPEC/v2/review-2026-05-round2/G44/03-plan-r1.md#L48-L56). Live code disagrees: `ChatAgent` emits `thinking` through `sendEvent` in [src/agents/chat.ts](src/agents/chat.ts#L208-L210) and sends non-Telegram assistant responses through `sendEvent` in [src/agents/chat.ts](src/agents/chat.ts#L389-L392). Telegram also describes `sendEvent` as handling typed events from ChatAgent in [src/channels/telegram.ts](src/channels/telegram.ts#L371-L378). Keep the important distinction that `sendEvent` is not part of the four-member `ChatChannel` interface in [src/channels/types.ts](src/channels/types.ts#L5-L16), but rewrite the round 1 docs to say it is an optional concrete-channel extension used by ChatAgent and, for WebSocket session setup, by route glue in [src/server/server.ts](src/server/server.ts#L692-L693).

2. Low - The post-edit checks do not fully verify the stale claims that G44 is supposed to remove. The plan bans `chat-chunk`, `start(runtime)`, and `stop()` during the rewrite in [SPEC/v2/review-2026-05-round2/G44/03-plan-r1.md](SPEC/v2/review-2026-05-round2/G44/03-plan-r1.md#L85-L92), but the sanity and dist greps only cover removed file paths plus `publish(event)` in [SPEC/v2/review-2026-05-round2/G44/03-plan-r1.md](SPEC/v2/review-2026-05-round2/G44/03-plan-r1.md#L117-L157). That leaves the current stale `chat-chunk` envelope in [docs/internals/channels.md](docs/internals/channels.md#L40-L45) without an automated source-doc or dist-doc check. Add `chat-chunk|interface Channel|start\(runtime|stop\(\)` to the source and dist verification so a partial rewrite cannot pass while still contradicting `WebSocketChannel.send()` and `sendEvent()` in [src/channels/websocket.ts](src/channels/websocket.ts#L35-L43).

3. Low - While the plan touches the state prose, its verification is too imprecise for the current chat log layout. The live path includes a channel subdirectory: `ChatAgent` builds the directory as .saivage/tmp/chats/<channel> in [src/agents/chat.ts](src/agents/chat.ts#L98-L104) and writes <sessionId>.json underneath it in [src/agents/chat.ts](src/agents/chat.ts#L398-L400). The plan asks implementers to verify a .saivage/tmp/chats/<sessionId>.json claim with a broad tmp/chats grep in [SPEC/v2/review-2026-05-round2/G44/03-plan-r1.md](SPEC/v2/review-2026-05-round2/G44/03-plan-r1.md#L78-L83), and the adjacent live doc still has the flat-path form in [docs/internals/agent-chat.md](docs/internals/agent-chat.md#L55-L59). Since G44 already edits [docs/internals/agent-chat.md](docs/internals/agent-chat.md#L31-L40), either fix that adjacent session path or explicitly keep it out of scope so the channels rewrite remains minimal without carrying a nearby known drift.

## What Is Solid

The rewrite scope is otherwise appropriately small: two source docs plus rebuilt VitePress output, with the broader docs-drift lint left to G40. The docs build command is real in [package.json](package.json#L23-L24), and the plan requires running it before acceptance.

Strengthen post-edit verification to catch every stale channel-interface and WebSocket-envelope string, especially `chat-chunk`. Resolve or explicitly exclude the adjacent chat-log path drift while editing [docs/internals/agent-chat.md](docs/internals/agent-chat.md#L55-L59).
