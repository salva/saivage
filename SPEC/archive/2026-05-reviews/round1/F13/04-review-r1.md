## Findings

### Analysis

The analysis accurately identifies the core failure mode: [src/agents/base.ts](src/agents/base.ts#L515-L547) classifies `router.chat` failures by regexing `err.message`, while [src/providers/router.ts](src/providers/router.ts#L407-L418) has a separate, narrower context-overflow predicate. The provider throw-site survey is also directionally correct.

One factual correction matters for the plan: `consecutive invalid tool calls` is thrown by [src/runtime/dispatcher.ts](src/runtime/dispatcher.ts#L177) and propagated from [src/runtime/dispatcher.ts](src/runtime/dispatcher.ts#L212), not thrown by `BaseAgent.callLLM`. The current `NON_RETRYABLE_RE` entry for that text is therefore not evidence that the callLLM provider-error classifier currently handles that path.

### Design

Proposal B is the right architecture for this issue: the provider boundary should normalize SDK/native failures into a typed error, and BaseAgent should switch on the typed kind instead of string fragments.

The design does not yet specify how `orphaned_tool_result` is produced after deleting [src/agents/base.ts](src/agents/base.ts#L873). This is a required existing repair path: [src/agents/base.ts](src/agents/base.ts#L515-L531) treats orphaned tool-result errors the same as context overflow and compacts immediately. The proposed classifier currently says Anthropic/OpenAI `invalid_request_error` or HTTP 400 become `non_retryable` in [SPEC/v2/review-2026-05/F13/03-plan-r1.md](SPEC/v2/review-2026-05/F13/03-plan-r1.md#L13-L14), which would likely turn several tool-result shape errors into fatal errors before the compact-and-retry path can run.

The design also overstates its prohibition on message inspection: [SPEC/v2/review-2026-05/F13/02-design-r1.md](SPEC/v2/review-2026-05/F13/02-design-r1.md#L122) says no regex against `err.message` anywhere in `src/agents/` or `src/providers/`, but [SPEC/v2/review-2026-05/F13/03-plan-r1.md](SPEC/v2/review-2026-05/F13/03-plan-r1.md#L13-L16) already requires message/status parsing for SDK and raw errors. The intended rule should be narrower: no agent/router regex classifier; provider-boundary classification may inspect provider-native error fields and, where unavoidable, provider message text in one central helper.

### Plan

The final router wrap edit is not executable as written. [SPEC/v2/review-2026-05/F13/03-plan-r1.md](SPEC/v2/review-2026-05/F13/03-plan-r1.md#L31) proposes `providerName` in the wrap after all providers fail, but the current `providerName` binding is scoped inside the candidate loop at [src/providers/router.ts](src/providers/router.ts#L290), while the final wrap occurs at [src/providers/router.ts](src/providers/router.ts#L356-L360). The plan needs either a tracked `lastProviderName`, `lastError.providerName`, or no provider name on the aggregate wrapper.

The router short-circuit rule is incomplete for repair-class errors. [SPEC/v2/review-2026-05/F13/03-plan-r1.md](SPEC/v2/review-2026-05/F13/03-plan-r1.md#L29-L31) says `context_overflow` short-circuits failover, but omits `orphaned_tool_result`. Since orphaned tool-result errors are generated by Saivage's message shape rather than by a single vendor outage, the router should not exhaust the failover chain before handing that typed error back to BaseAgent for compaction.

The internal-sentinel step needs correction. [SPEC/v2/review-2026-05/F13/03-plan-r1.md](SPEC/v2/review-2026-05/F13/03-plan-r1.md#L55) says BaseAgent throws both cancellation and consecutive-invalid sentinels, but cancellation is thrown at [src/agents/base.ts](src/agents/base.ts#L489) and [src/agents/base.ts](src/agents/base.ts#L861), while consecutive invalid calls are in [src/runtime/dispatcher.ts](src/runtime/dispatcher.ts#L177). The plan should either remove consecutive-invalid handling from F13 as outside provider error classification, or explicitly include the dispatcher change and explain why that cross-module change belongs in this issue.

## Required changes

1. Revise the design and plan so `orphaned_tool_result` has a concrete producer after the BaseAgent regexes are deleted. It should be classified before generic `invalid_request_error` / HTTP 400 mapping, short-circuit router failover like `context_overflow`, and have tests at the provider classifier, router, and BaseAgent levels.
2. Fix the router aggregate-wrapper plan so it does not reference out-of-scope `providerName`, and state exactly how the last provider/account context is preserved or intentionally omitted.
3. Correct the internal-sentinel discussion for `consecutive invalid tool calls`: it is a dispatcher error, not a BaseAgent callLLM/provider error. Either remove it from the ProviderError conversion plan or add the dispatcher file to scope with a clear reason and tests.
4. Reword the design's `err.message` prohibition so it does not contradict the necessary centralized provider-boundary classifier. The unacceptable pattern is scattered BaseAgent/router regex classification, not all message inspection inside `classifyProviderError`.

## Strengths

The recommended direction is clean and appropriately architecture-first. It deletes the brittle BaseAgent regex switch, removes the duplicated router context predicate, uses a narrow union for behaviorally meaningful error kinds, and captures `retry-after` metadata that the current code discards.
