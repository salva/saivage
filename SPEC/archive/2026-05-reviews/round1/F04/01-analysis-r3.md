Resolution: the analysis now treats `rule.allowedModels?.length > 0` as a routing-derived source, identical to `rule.model` / `rule.preferredModels` / `rule.profile`. The classification change applies before the `"hardcoded-default"` branch is removed.

## Problem restated

A model identifier string is encoded directly in source code in ten places. Five are operational defaults that take effect when the project does not configure a model; two are last-ditch fallbacks deep in the routing/resolver stack; one is an inline `??` literal inside the resolver's `resolve()` method that is currently dead but still embeds a model name; one is a hardcoded seed value written into every freshly initialized project's `.saivage/config.json` by `saivage init`; and one is an example model identifier embedded in a JSDoc comment on `AgentContext.modelSpec` that would defeat the post-change source-sweep check. The strings span four subsystems (config schema, security cop, runtime supervisor, provider router, routing resolver) plus the CLI init flow plus the agent-context type declaration. The runtime strings do not even agree on which model to fall back to: orchestrator and router default to `"anthropic/claude-sonnet-4-20250514"`, supervisor and security default to `"github-copilot/gpt-5.4"`, and the routing resolver (twice) and CLI init (once) default to `"openai-codex/gpt-5.3-codex"`.

The operator's explicit constraint for this issue (quoted in [F04-hardcoded-default-models.md](../F04-hardcoded-default-models.md)) is:

> No model should be hard-coded. If no model is set in the config, the system must just fail to work and report the issue.

The bar is "remove the defaults entirely and make the bootstrap fail loudly when the config does not name a model for every role the runtime is about to use." The sweep check is the executable guard against regression; that guard is only credible if no model identifier â€” default, fallback, seed, or comment example â€” survives in production source.

Distinct behaviours that diverge today (unchanged from r2 except for the routing/allowed_models clarification below):

1. **Four different "default models" coexist.** A bare `saivage.json` produces an orchestrator model of `anthropic/claude-sonnet-4-20250514` (site #1) and supervisor/security models of `github-copilot/gpt-5.4` (sites #2, #3). Any worker role missing from `models` and missing from `models.default` resolves to *yet another* model: `openai-codex/gpt-5.3-codex` via the resolver fallback (sites #7, #8) when the resolver path is hit, or `anthropic/claude-sonnet-4-20250514` via `ModelRouter.resolveModelForRole` (site #6) when the router path is hit. Which one wins depends on which call site reaches the model first. Additionally, a freshly init'd project carries `openai-codex/gpt-5.3-codex` in `project.provider` (site #9) which short-circuits site #7's chain via the `project.provider` branch.
2. **Two of the constants are dead today but still encode the string.** The Zod default at [src/config.ts](src/config.ts#L84-L92) means `config.supervisor.model` is always a string, so the `?? DEFAULT_MODEL` in [src/runtime/supervisor.ts](src/runtime/supervisor.ts#L50) never fires. Same pattern for the security cop, same pattern for site #8 (the inline `?? "openai-codex/..."`). These dead defaults still encode model identifiers in source â€” operators rotating models would have to find and update them anyway to keep the codebase clean.
3. **Routing layer disagrees with router layer.** `ModelRoutingResolver` is the layered resolver (project routing â†’ runtime default â†’ `project.provider` â†’ hardcoded). `ModelRouter.resolveModelForRole` is a separate method on the legacy router with its own hardcoded fallback. 4. **The defaults encode vendor + product version**, not a capability tier. `gpt-5.4`, `gpt-5.3-codex`, `claude-sonnet-4-20250514` are vendor-routed product versions that the providers will EOL independent of any Saivage decision. When that happens, the affected subsystems silently start using whatever the provider routes that name to (or, if the provider drops the model, hard-fail in production with no traceability back to "this default was set in source two years ago").
5. **Fresh-init bypass.** Even after sites #1â€“#8 are removed, a project created via `saivage init` would still resolve to a model out of source-code because `project.provider` is seeded with site #9's string. The fix must remove the seed, not just the resolver-level fallbacks.
6. **`resolveSource` does not classify `allowed_models`-only rules as routing-derived.** `resolvePreferredModels` ([src/routing/resolver.ts](src/routing/resolver.ts#L228-L243)) returns `[...allowed]` for a rule that sets only `allowed_models` and has no `model` / `preferred_models`. ## Contract

The thing that needs a contract is "what model does role X use?" â€” and the answer today is "whatever survives the resolution chain, with up to four distinct hardcoded fallbacks at the bottom plus a hardcoded seed in `project.provider`." Under the operator directive, the contract becomes:

- **Inputs:** a `SaivageConfig` (parsed from `.saivage/saivage.json`), a `ProjectConfig` (parsed from `.saivage/config.json`), and a role name from the live roster (`AgentRole` âˆª `"supervisor" | "security" | "default"`).
- **Output:** a fully-qualified model spec string (`"<provider>/<model>"`).
- **Error mode:** if the config does not specify a model for the role and there is no `models.default`, no `routing.roles` entry (including no `allowed_models` set for the role), no `model_overrides` entry, and no operator-set `project.provider`, **bootstrap fails with a clear "no model configured for role X" error before any agent starts**. No silent fallback to any built-in string.
- **Lifecycle:** validation happens once at boot, after `loadConfig` returns and `ModelRoutingResolver` is built and before any agent/supervisor/cop is constructed. Lazy/runtime resolution then either trivially succeeds (config covers the role) or â€” if a role is reached that boot validation didn't consider â€” throws a typed `MissingModelForRoleError` that the caller propagates.
- **Conditional subsystems:** boot validation only requires `supervisor` and `security` model resolution when their `enabled` / `injectionScanner` flags are true. Bootstrap construction follows the same rule: `routing.resolve("supervisor")` and `routing.resolve("security")` are only called when the corresponding subsystem is enabled. A disabled subsystem does not require a model and does not trigger boot validation for that role.
- **Source classification:** the `ResolvedModelRoute["source"]` discriminant has four members post-F04: `"routing" | "legacy" | "runtime-default" | "project-default"`. A rule that sets *any* of `model`, `preferred_models`, `allowed_models`, or `profile` classifies as `"routing"`. The `"hardcoded-default"` member is removed; the corresponding branch in `resolveSource` is unreachable after the new `allowed_models` classification + the throwing `resolveLegacyModels` together cover every successful resolution path.
- **Fresh init:** `saivage init <path>` produces a config that fails `saivage start` with the same `MissingModelForRoleError`, naming each missing role and the exact `.saivage/saivage.json` path the operator must edit. There is no "init wrote me a model so it works out of the box" silent success path.

Roles that must be covered by the boot check:

- **Always required:** every role from the agent roster ([src/agents/types.ts](src/agents/types.ts#L20-L28) â†’ after F02, the unified roster). F04's validator iterates the roster and asks the resolver for each role without falling back to any string.
- **Conditionally required:** `supervisor` only when `config.supervisor.enabled` is `true`; `security` only when `config.security.injectionScanner` is `true`. Otherwise the subsystem is not constructed and asking for its model would falsely fail boot.
- **Always optional:** `models.default` is a convenience inherit-target; it does not need to be set if every other key is set explicitly.

## Constraints any solution must respect

1. **No backward compatibility.** No "fallback to old default if config missing" warning + continue. The operator was explicit: fail. No `@deprecated` markers, no transitional `model || OLD_DEFAULT` shims.
2. **Fail at boot, not on first use.** A misconfigured Saivage that runs for 4 hours and then crashes during supervisor cycle 1 because nobody configured `supervisor.model` is worse than today's hardcoded-default behaviour. The validator must run during `bootstrap()` before any agent is constructed.
3. **Conditional subsystems must be checked conditionally.** `supervisor.model` is only required when `supervisor.enabled === true`; `security.injectionModel` only when `security.injectionScanner === true`. Otherwise an operator who disables those subsystems would be forced to set fake values, and the bootstrap's `routing.resolve("supervisor")` / `routing.resolve("security")` calls would themselves fail. Bootstrap construction must also be conditional: the model resolution is invoked only when the subsystem is enabled.
4. **No `?? FALLBACK_CONSTANT` patterns survive in any of the nine runtime sites.** Sites 4, 5, 6, 7, 8 currently use `??` chains terminating in a string literal. After F04 those chains either terminate in a thrown error (sites 6, 7, 8) or collapse to the required override (sites 4, 5). Site 9 (the cli.ts seed) is deleted outright; no fallback replaces it. Site 10 (the JSDoc example) is rewritten to a non-model placeholder.
5. **System-boundary validation only.** Per workspace guideline, internal callers (`createPromptInjectionCop`, `new RuntimeSupervisor`, agent constructors) do not re-validate; they trust that boot validation has run. The only place that validates is the boot-time `validateModelCoverage` function.
6. **The error message must name (a) every missing role, (b) the config field the operator should set, (c) the file path of the active `saivage.json`.** No "Cannot read property 'model' of undefined" stack traces.
7. **Test fixtures stay literal.** Tests use concrete model strings (`"github-copilot/gpt-5.4"`) as realistic inputs; F04 does not need to introduce a `TEST_MODEL` helper or refactor those tests. The strings being deleted are *defaults*, not *test inputs*.
8. **No new docstrings/comments** on code not otherwise modifying. The JSDoc on `AgentContext.modelSpec` is being *edited* (model literal removed from the existing example), not *added*; this is consistent with the rule.
9. **Skills/memory subsystem is out of scope.** No skill loader is affected.
10. **`writeDefaultConfig`** at [src/config.ts](src/config.ts#L204-L237) seeds a starter `saivage.json` with `models: {}`. Under the operator directive, that is the correct seed â€” empty. An operator running `saivage init` and then `saivage start` will hit the validator's `MissingModelForRoleError` for every required role. The error names the file path and the field to set. No placeholder string is written; no "REPLACE-WITH-PROVIDER/MODEL" sentinel is introduced. The validator does not need a placeholder-detection rule because there is no placeholder.
11. **`saivage init` writes no model.** The CLI init seed at [src/server/cli.ts](src/server/cli.ts#L45) is deleted; the freshly written `.saivage/config.json` has no `provider` field. The `provider` field remains in `ProjectConfigSchema` as an optional operator-settable value (operators can hand-edit it, and the resolver's `project.provider` branch still honors it as legitimate operator config). The field is not removed from the schema in this change because removing it crosses into "what does the legacy non-routing config path look like" territory, which is independent of F04's "no source-level defaults" directive.
12. **Production-source sweep.** After the edits, the executable check is:

    ```bash
    rg -n 'github-copilot/gpt-5\.|anthropic/claude-sonnet-4-|openai-codex/gpt-5\.3-codex' src/ --type ts | grep -v '\.test\.ts'
    ```

    Expected output: zero matches. This includes the JSDoc example on `AgentContext.modelSpec` ([src/agents/types.ts](src/agents/types.ts#L49)) which is rewritten in the same change. The sweep contract is *strict* â€” comments, examples, and runtime defaults are all in scope â€” because narrowing the sweep to "runtime defaults only" would require a custom AST-aware checker just to skip JSDoc, which adds a lot of code for a one-off concern. Rewriting the one comment is a one-line edit; that is the cheaper resolution.
13. **`allowed_models`-only routing rules must keep working.** A `routing.roles.X` rule that sets only `allowed_models: ["provider/model"]` (no `model`, no `preferred_models`, no `profile`) resolves to a model via the existing `resolvePreferredModels` path ([src/routing/resolver.ts](src/routing/resolver.ts#L241)). The change must classify this case as `source: "routing"` (currently it falls through to `"hardcoded-default"`). Without this fix, removing the `"hardcoded-default"` branch would convert a valid operator-set rule into a thrown error.
