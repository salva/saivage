# F03 Librarian Agent Analysis R4 Review

Scope: strict factual and architectural audit of [SPEC/2026-05/rag-agent-integration/F03-librarian-agent/01-analysis-r4.md](SPEC/2026-05/rag-agent-integration/F03-librarian-agent/01-analysis-r4.md) against the current TypeScript source.

## Summary

R4 fixes the prior secret-path overclaim and is directionally sound on the dispatch model, prompt registration pattern, and a role-specific `Y†` extension. However, it still contains source-level false claims that would lead implementers to put the Librarian guard in the wrong place, rely on RAG tools that are not registered in the current source, use a nonexistent `TaskReport.notes` field for fallback routing, and encode the wrong memory storage territory in the roster entry.

## Confirmed Correct

- The `AccessCell` union is currently `"Y" | "Y†" | "-"`, and `canCall(role, op, kind)` gates only the matrix cell, not scope. `checkScope` receives `(role, op, kind, scope, scope_ref, ctx)` and applies the dagger path only after `cellFor(...) === "Y†"`; see [src/knowledge/permissions.ts](src/knowledge/permissions.ts#L29-L39), [src/knowledge/permissions.ts](src/knowledge/permissions.ts#L222-L244), and [src/knowledge/permissions.ts](src/knowledge/permissions.ts#L246-L292). A role-specific Librarian branch before the existing worker-stage branch is feasible and does not inherently break coder/researcher behavior.
- `RUN_LIBRARIAN_SCHEMA` plus a `DISPATCH_SCHEMA_BY_TOOL.run_librarian` entry follows the existing BaseAgent pattern. Dispatch schemas are explicit by tool name and roster entries are validated against that map at module load; see [src/agents/base.ts](src/agents/base.ts#L1001-L1130) and [src/agents/base.ts](src/agents/base.ts#L1132-L1151).
- A bootstrap `case "librarian"` follows the existing non-worker `inspector` shape. The switch constructs Manager, worker, and Inspector agents explicitly; see [src/server/bootstrap.ts](src/server/bootstrap.ts#L336-L396). Planner and Manager receive child spawners, while ordinary workers do not receive dispatch tools through their constructors; see [src/server/bootstrap.ts](src/server/bootstrap.ts#L481-L506), [src/agents/base.ts](src/agents/base.ts#L195-L200), and [src/agents/base.ts](src/agents/base.ts#L699-L703).
- The `IngestReport` correction is accurate. The source exposes only aggregate counters, including `chunksDroppedSecrets`, and no affected path list; see [src/rag/types.ts](src/rag/types.ts#L150-L159). R4 correctly drops the previous requirement to list affected paths and limits the memory write to count, collection id, caller-supplied context, and stats-derived timestamps.
- The non-RAG whitelist names are real: filesystem tools include `read_file`, `list_dir`, and `search_files`; knowledge skill tools include `list_skills`, `read_skill`, and `search_skills`; memory tools include `create_memory`, `update_memory`, `list_memories`, `get_memory`, and `search_memories`; `read_stash` is a BaseAgent synthetic tool. See [src/mcp/builtins.ts](src/mcp/builtins.ts#L401-L442), [src/mcp/knowledgeSkills.ts](src/mcp/knowledgeSkills.ts#L38-L132), [src/mcp/knowledgeMemory.ts](src/mcp/knowledgeMemory.ts#L32-L69), [src/mcp/knowledgeMemory.ts](src/mcp/knowledgeMemory.ts#L103-L128), and [src/agents/base.ts](src/agents/base.ts#L674-L697).

Adding `"librarian"` to `ToolFilterKind`, `ROSTER`, prompt keys, and knowledge roles is still required.

## Findings

### 1. The update-memory guard placement is factually wrong and unsafe as written.

R4 says the handler-side topic guard runs after `canCall`/`checkScope`, and that for `update_memory` the existing handler reads the prior record, applies the patch, and then guards the merged record. That is not what the current handler does. `create_memory` calls `gateRole`, extracts scope, calls `gateScope`, and then calls `createMemory`; `update_memory` only calls `gateRole(role, "create")` and then delegates directly to `updateMemory`; see [src/mcp/knowledgeMemory.ts](src/mcp/knowledgeMemory.ts#L193-L228). The lifecycle helper then finds the memory, builds the merged `updated` record, writes it, appends audit, rebuilds the index, and returns only `{ id, updated_at }`; see [src/knowledge/lifecycle.ts](src/knowledge/lifecycle.ts#L572-L596).

That means a handler-side guard placed after the current `updateMemory(...)` call would validate too late, after mutation. A correct F03 design must either preflight-read the existing record and build the effective merged record before calling `updateMemory`, or move the authorization hook into lifecycle before [src/knowledge/lifecycle.ts](src/knowledge/lifecycle.ts#L593-L595). Because `update_memory` currently cannot patch `topic` or `scope`, checking the existing record before mutation is equivalent for the current schema, but the analysis must say that explicitly and must not claim the current handler already performs the merge.

This also exposes a pre-existing scope nuance that R4 papers over: the current `update_memory` path does not call `checkScope` at all, despite coder/researcher rows being `Y†` for update; see [src/knowledge/permissions.ts](src/knowledge/permissions.ts#L76-L99) and [src/mcp/knowledgeMemory.ts](src/mcp/knowledgeMemory.ts#L216-L228). The Librarian design cannot use the current handler as proof that update writes are already scoped.

### 2. The proposed ACL error codes do not match the current permission types and handler envelope.

The `checkScope` snippet in §5 returns `code: "KNOWLEDGE_SCOPE_DENIED"`, while the current `ScopeCheckResult` only admits `code: "UNAUTHORIZED_SCOPE"`; see [src/knowledge/permissions.ts](src/knowledge/permissions.ts#L246-L280). The MCP handler then converts scope failures into `KnowledgeStoreError("UNAUTHORIZED_SCOPE", ...)` and serializes caught `KnowledgeStoreError` instances as `{ error: { code, message } }`; see [src/mcp/knowledgeMemory.ts](src/mcp/knowledgeMemory.ts#L171-L181) and [src/mcp/knowledgeMemory.ts](src/mcp/knowledgeMemory.ts#L297-L301).

Likewise, §1.4 names a `KNOWLEDGE_PERMISSION_DENIED` envelope that is not part of the current handler. The design may introduce a new error code, but then it must explicitly update the result type, helper, and tests. As a factual analysis of the current code, it should align with `UNAUTHORIZED_SCOPE` / `UNAUTHORIZED_ROLE` or define the required type changes.

### 3. The RAG whitelist names are not existing tool names in the current source.

R4 treats `rag_list`, `rag_stats`, `rag_query`, `rag_register`, `rag_ingest`, `rag_drop`, and `rag_admin` as concrete F02 tool names and builds the Librarian whitelist and fallback path around them. In the current source, no MCP RAG service or `ToolEntry` names for these tools exist; built-in service registration currently exposes filesystem, shell, git, web, skills, and memory services, not RAG; see [src/mcp/builtins.ts](src/mcp/builtins.ts#L1912-L1966). The current role filters also do not include any `rag_*` read tools in `READ_ONLY_TOOLS`; see [src/agents/tool-filters.ts](src/agents/tool-filters.ts#L12-L40).

This makes §1.5, §3.4, §4, §6, §7, and §8 overstate current-source reality. The document can depend on the F02 spec, but then it must phrase the RAG names as a prerequisite on F02 implementation, not as existing workspace tool names. If the audit target is the current source, the whitelist does not fully match registered tools.

### 4. The retrieval-miss fallback path relies on a nonexistent `TaskReport.notes` field.

§3.4 says workers record retrieval gaps in `TaskReport.notes`, “used by every worker today,” and surface that to Manager. The current `TaskReportSchema` has no `notes` field; it contains `summary`, checklist results with per-item optional `notes`, file/test/commit arrays, `issues_found`, truncation and failure metadata, and timestamps; see [src/types.ts](src/types.ts#L196-L215). Workers parse their final response into that schema and return it as the child dispatch result; see [src/agents/worker.ts](src/agents/worker.ts#L229-L238) and [src/runtime/dispatcher.ts](src/runtime/dispatcher.ts#L239-L249).

The fallback can remain architecturally coherent, but it must use an existing channel such as `summary`, a checklist result note, or `issues_found`, or else explicitly add and test a `TaskReport.notes` field. As written, the described path is not coherent with the current dispatch/report graph.

### 5. The roster convention uses the wrong memory storage path.

The proposed roster entry sets `writeTerritory` to `[".saivage/rag/", ".saivage/knowledge/memory/"]`. Current knowledge memory records are stored under `.saivage/memory/...`, not `.saivage/knowledge/memory/...`; `scopeDir` uses `kindDir = "memory"` and returns `.saivage/memory/project`, `.saivage/memory/stages/<id>`, or `.saivage/memory/sessions/<id>`; see [src/knowledge/lifecycle.ts](src/knowledge/lifecycle.ts#L110-L123). Conventions are warnings rather than hard enforcement, but they are still part of the roster contract and should point at real write territory; see [src/agents/conventions.ts](src/agents/conventions.ts#L20-L31).
