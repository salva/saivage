# F02 - Agent-facing RAG collection tools: review

The analysis is not ready to approve. It covers many of the requested headings, but several conclusions contradict the source-of-truth constraints and several implementation claims do not match the current RAG/MCP public surface.

## 1. Coverage Of Required Topics

The required operation inventory is incomplete. The topic file explicitly asks the analysis to enumerate registration, listing, stats, directory ingest, explicit text ingest, record deletion, query, collection drop, watcher toggle, and force reconcile ([saivage/SPEC/2026-05/rag-agent-integration/F02-collection-tools.md](saivage/SPEC/2026-05/rag-agent-integration/F02-collection-tools.md#L51-L52)). The tool inventory covers most of these but omits delete-a-record entirely ([saivage/SPEC/2026-05/rag-agent-integration/F02-collection-tools/01-analysis-r1.md](saivage/SPEC/2026-05/rag-agent-integration/F02-collection-tools/01-analysis-r1.md#L58-L73)). That leaves a required operation undecided, including its protected-dataset behavior and error mapping.

The current MCP model is discussed, but the analysis mixes correct high-level ideas with incorrect mechanics, so the coverage is not reliable enough for a design document. It also does not fully answer the topic's "which become tools and which are admin-only" question; it presents ten tools and then defers the actual split to the design phase ([saivage/SPEC/2026-05/rag-agent-integration/F02-collection-tools/01-analysis-r1.md](saivage/SPEC/2026-05/rag-agent-integration/F02-collection-tools/01-analysis-r1.md#L73-L76)). The source-of-truth already fixes the focused tool set under one service ([saivage/SPEC/2026-05/rag-agent-integration/F02-collection-tools.md](saivage/SPEC/2026-05/rag-agent-integration/F02-collection-tools.md#L78-L80)), so the functional analysis should make a constrained recommendation rather than leave the surface open.

Failure surface and telemetry coverage are present but too thin. The topic asks for drift, flood, watcher-unavailable, dataset-not-found, lock contention, and embedding provider rate-limit behavior ([saivage/SPEC/2026-05/rag-agent-integration/F02-collection-tools.md](saivage/SPEC/2026-05/rag-agent-integration/F02-collection-tools.md#L64-L65)); the draft maps most existing RAG errors but does not give a tool-visible story for watcher flood, RAG disabled, argument-schema failures, or register/drop races. Logging is described as if there is an existing metrics mechanism, but no such mechanism is identified in the cited runtime files.

## 2. Factual Accuracy Against The Cited Files

The tool-call context shape is wrong. The analysis says handlers receive `agentRole` ([saivage/SPEC/2026-05/rag-agent-integration/F02-collection-tools/01-analysis-r1.md](saivage/SPEC/2026-05/rag-agent-integration/F02-collection-tools/01-analysis-r1.md#L14-L18)), but `ToolCallContext` carries `role`, `agentId`, `projectRoot`, optional `stageId`, optional `channelId`, and optional `sessionId` ([saivage/src/mcp/toolContext.ts](saivage/src/mcp/toolContext.ts#L17-L34)). Any permission helper and schema examples need to use `ctx.role`, not `ctx.agentRole`.

The MCP registration model is misstated. Built-ins are not records shaped as `{ id, tools, handlers: Record<string, InProcessToolHandler> }`; `McpRuntime.registerInProcess` takes a service name, a `ToolEntry[]`, and one handler function ([saivage/src/mcp/runtime.ts](saivage/src/mcp/runtime.ts#L153-L162)), and `builtins.ts` registers services with one handler per service ([saivage/src/mcp/builtins.ts](saivage/src/mcp/builtins.ts#L1961-L1966)). This matters because the proposed file layout and handler dispatch should mirror the existing `skills` and `memory` services.

The error-surfacing description is also inaccurate. The runtime does not automatically map thrown typed errors by `name` into a stable agent-visible code. In-process handlers return `{ content, isError }`; when `isError` is true, `McpRuntime.callTool` wraps the content in a generic thrown `Error` ([saivage/src/mcp/runtime.ts](saivage/src/mcp/runtime.ts#L188-L203)). Existing knowledge handlers catch domain errors and explicitly return `{ error: { code, message } }` envelopes ([saivage/src/mcp/knowledgeSkills.ts](saivage/src/mcp/knowledgeSkills.ts#L169-L178), [saivage/src/mcp/knowledgeSkills.ts](saivage/src/mcp/knowledgeSkills.ts#L277-L278)). The RAG tool design must specify explicit handler-side conversion from each `RagError` to a tool envelope.

Several RAG type claims do not match the current public surface. `DatasetConfig.source` is a `RagSource` literal, not `{ kind: ... }` ([saivage/src/rag/types.ts](saivage/src/rag/types.ts#L39-L43)); the runtime config schema currently allows only `text-embedding-3-small`, not the large model ([saivage/src/config.ts](saivage/src/config.ts#L224-L228)); and records ingest uses `items`, not `records` ([saivage/src/rag/types.ts](saivage/src/rag/types.ts#L143-L147)). The draft schema in the registration section therefore cannot be implemented against the cited API as written ([saivage/SPEC/2026-05/rag-agent-integration/F02-collection-tools/01-analysis-r1.md](saivage/SPEC/2026-05/rag-agent-integration/F02-collection-tools/01-analysis-r1.md#L187-L193)).

The manager and dataset behavior are overclaimed. `Dataset` has `watch`, `unwatch`, and `reconcile` methods ([saivage/src/rag/dataset.ts](saivage/src/rag/dataset.ts#L149-L172)), but `RagManager` exposes only `list`, `get`, `register`, `ingest`, `query`, `stats`, `drop`, and `close` ([saivage/src/rag/manager.ts](saivage/src/rag/manager.ts#L42-L51)). Tool handlers that want watcher controls either need to call `manager.get(id)` and then `Dataset` methods or explicitly explain why watcher controls are not tools. Also, `manager.get` resolves datasets from `opts.datasets`, not from the registry ([saivage/src/rag/manager.ts](saivage/src/rag/manager.ts#L118-L120)), so the claim that runtime registrations are additive and survive restarts via the registry is not true without additional design work ([saivage/SPEC/2026-05/rag-agent-integration/F02-collection-tools/01-analysis-r1.md](saivage/SPEC/2026-05/rag-agent-integration/F02-collection-tools/01-analysis-r1.md#L339-L346)).

## 3. Architectural Soundness

The analysis violates the hard one-namespace constraint. The topic says all RAG-facing tools live under a single MCP service id and should aim for no more than eight distinct tools ([saivage/SPEC/2026-05/rag-agent-integration/F02-collection-tools.md](saivage/SPEC/2026-05/rag-agent-integration/F02-collection-tools.md#L19-L20)). The draft recommends or strongly foreshadows `rag.read`, `rag.write`, and `rag.admin` ([saivage/SPEC/2026-05/rag-agent-integration/F02-collection-tools/01-analysis-r1.md](saivage/SPEC/2026-05/rag-agent-integration/F02-collection-tools/01-analysis-r1.md#L89-L101)). That split can be analyzed as the required level-up alternative, but the analysis needs to reject it or explicitly mark it as conflicting with the source-of-truth constraint.

The permission model assumes wildcard namespace grants that the current per-agent filter does not have. `applyToolFilter` checks only the tool name, with static sets and predicates keyed by filter kind ([saivage/src/agents/tool-filters.ts](saivage/src/agents/tool-filters.ts#L12-L43)). A statement like "granting `rag.read.*`" is not compatible with the existing filter unless the analysis proposes a filter-model change, which would be scope creep for this topic. Within one service, the design should use concrete tool names such as `rag_query` or concrete MCP `ToolEntry.name` values and then add those names to the relevant filters.

The role story uses roles that are not in the current roster. `librarian` and `operator` are not `AgentRole` values in [saivage/src/agents/roster.ts](saivage/src/agents/roster.ts#L321-L356), while the topic only allows naming the future Librarian as a future caller. The analysis must separate current agent authorization from out-of-band operator access, and it must explain how an operator reaches the same tools when `ToolCallContext.role` is typed as an existing `AgentRole`.

Protected datasets are mostly respected but need a tighter contract. F01 says the `skills` and `memories` datasets remain runtime-managed and config-pinned, and F02 may expose them read-only ([saivage/SPEC/2026-05/rag-agent-integration/F01-skill-memory-on-rag/01-analysis-r1.md](saivage/SPEC/2026-05/rag-agent-integration/F01-skill-memory-on-rag/01-analysis-r1.md#L535-L536)). The draft blocks write/admin operations against those ids, which is correct, but it also gives future roles broad write/admin responsibilities before those roles exist. It should state the invariant independently of role names: no F02 collection tool may create, drop, ingest into, delete from, watch, or reconcile the protected datasets; only `list`, `stats`, and possibly raw `query` are allowed, and record-aware skill/memory reads remain owned by F01 tools.

## 4. Clean Code And Architecture-first Fit

There are no obvious backward-compatibility aliases or soft-deprecation tiers, which is good. The bigger clean-architecture problem is scope creep: the draft starts with ten tools, adds split services, mentions possible `ToolEntry` changes, invents metrics counters, and adds startup drift behavior ([saivage/SPEC/2026-05/rag-agent-integration/F02-collection-tools/01-analysis-r1.md](saivage/SPEC/2026-05/rag-agent-integration/F02-collection-tools/01-analysis-r1.md#L373-L381)). The topic asks for a small plain-JSON tool layer over the existing RAG public surface, not a broader MCP/runtime redesign.

The tool count is not actually bounded. The draft says the surface becomes compliant after watcher collapse and after `register`/`drop` are not exposed to agents ([saivage/SPEC/2026-05/rag-agent-integration/F02-collection-tools/01-analysis-r1.md](saivage/SPEC/2026-05/rag-agent-integration/F02-collection-tools/01-analysis-r1.md#L73-L76), [saivage/SPEC/2026-05/rag-agent-integration/F02-collection-tools/01-analysis-r1.md](saivage/SPEC/2026-05/rag-agent-integration/F02-collection-tools/01-analysis-r1.md#L91-L93)), but the source-of-truth focused proposal already includes `rag.register` and `rag.drop` in the single service ([saivage/SPEC/2026-05/rag-agent-integration/F02-collection-tools.md](saivage/SPEC/2026-05/rag-agent-integration/F02-collection-tools.md#L78-L80)). The analysis should settle on the bounded eight-tool shape or justify a smaller set, not carry multiple incompatible inventories forward.

## 5. Completeness

The error mapping needs to be comprehensive and implementable. It lists existing RAG error classes ([saivage/SPEC/2026-05/rag-agent-integration/F02-collection-tools/01-analysis-r1.md](saivage/SPEC/2026-05/rag-agent-integration/F02-collection-tools/01-analysis-r1.md#L350-L361)), but it does not map base `RagError`, disabled RAG, invalid arguments, JSON-schema validation, path guard failures from the tool layer, watcher-disabled errors from `Dataset.watch`, or watcher flood. The actual error hierarchy is finite and should be mapped from the exported classes in [saivage/src/rag/errors.ts](saivage/src/rag/errors.ts#L41-L125), plus F02-owned validation and authorization codes.

The concurrency story is incomplete. `proper-lockfile` is fail-fast with one stale retry before throwing `IngestLockedError` ([saivage/src/rag/lock.ts](saivage/src/rag/lock.ts#L3-L11), [saivage/src/rag/lock.ts](saivage/src/rag/lock.ts#L45-L53)); it is not a blocking queue with a configured retry budget. The analysis also needs to cover `register` versus `register`, `drop` versus `query`/`ingest`, manual `ingest` versus watcher reconcile, and what the tool returns when `Dataset.reconcile()` internally routes changed sources through `ingest` and hits a lock.

Telemetry is described as if structured metrics already exist in `src/runtime`, but the cited runtime folder does not expose a metrics/counter API, and the logging surface is a string-message buffer ([saivage/src/log.ts](saivage/src/log.ts#L1-L48)). The analysis can propose structured log messages using the existing logger or a small local wrapper, but it should not claim existing counters carry `ingests_total`, `query_count`, or similar metrics unless it points to the actual implementation.

## 6. Internal Consistency

The tool names are inconsistent across sections: the inventory says `watch_arm` and `watch_disarm`, the schema section says `watch.arm` and `watch.disarm`, the namespace section says a single `watch` tool may collapse arm/disarm/reconcile, and the file table allows split-service names. The analysis needs one canonical list matching the focused proposal and then every schema, permission rule, and error mapping must use that same list.

The result schemas do not line up with public RAG types. `add` promises `{ chunkId, embeddingsCacheHit }`, but the public ingest result is `IngestReport` with counters such as `chunksUpserted` and `chunksDroppedSecrets` ([saivage/src/rag/types.ts](saivage/src/rag/types.ts#L149-L159)). `reconcile` promises `{ scanned, changedPaths, removedPaths }`, but `Dataset.reconcile()` currently returns `Promise<void>` because it discards the controller result ([saivage/src/rag/dataset.ts](saivage/src/rag/dataset.ts#L170-L173)). `query` mentions `path` being present for sources `fs`, `code`, or `doc`, but `fs` is not a `RagSource` value ([saivage/src/rag/types.ts](saivage/src/rag/types.ts#L1-L4)).

Discovery and configuration do not align. The analysis rejects a new `description` field, so discovery is `rag.list` only ([saivage/SPEC/2026-05/rag-agent-integration/F02-collection-tools/01-analysis-r1.md](saivage/SPEC/2026-05/rag-agent-integration/F02-collection-tools/01-analysis-r1.md#L296-L304)), but then it claims runtime registrations survive restarts via the registry without explaining how strict `saivage.json` config and `RagManagerOptions.datasets` learn about those registered configs ([saivage/src/config.ts](saivage/src/config.ts#L217-L266), [saivage/src/rag/manager.ts](saivage/src/rag/manager.ts#L34-L47)). Either `rag.register` must persist a full config into the project configuration, or the analysis must admit that the current manager cannot support restart-surviving runtime registrations without a public-surface change.

## 7. Style Compliance

The document generally uses repo-root-style links and avoids decorative formatting. However, it is not fully self-contained because it repeatedly defers decisions to the later design or plan instead of making the analysis conclusion explicit. Phrases such as "the design proposals decide," "the plan likely picks," and "F02 chooses" make the document read like process notes rather than a standalone functional analysis. The next version should state the conclusion directly, avoid references to process sequencing, and keep all links current against the actual code.

7. Complete the failure, concurrency, and telemetry sections with implementable mappings for disabled RAG, invalid input, path guard failures, watcher-disabled/flood cases, provider failure without invented `retryAfterMs`, lock contention, register/drop races, and actual logging/metrics infrastructure.
8. Make the tool inventory, schemas, permissions, file list, and error table internally consistent with one canonical tool list and one discovery/config story.
