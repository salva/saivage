## High-Level Result

Changes are required before this analysis can be treated as factually correct. The broad direction is consistent with the source in several important places: the public RAG API is not extended, `RagManager.query(id, text, options?)` is the live signature, `IngestInput.kind === "records"` is the only public records-ingest shape, the ingest pipeline ignores caller-provided `metadata.source` for records, the eager-loader entry points can be preserved while changing their data source, and the lifecycle archive helper return shape is currently `{ archivedSkills, archivedMemories }`.

The blocking problems are narrower but material: the analysis claims byte-for-byte MCP output preservation while also removing `body_path` from `read_skill` records; it uses an invalid `QueryFilter` shape; its proposed recovery sweep relies on KNN query results as though they enumerate vector-store paths; it overstates the current `index.json` shape; it misstates current skill search scoring; and it does not account for the current `update_memory` ACL enforcement gap.

This supports the analysis' statement that skill chunks cannot be tagged `source="skill"` through the public records API.

## Required Changes

1. In sections 0.2, 0.3, 3.3, 4.1, and 10, reconcile the claim that every MCP output shape is preserved byte-for-byte with the proposed removal of `SkillRecord.body_path`. Current `SkillRecordSchema` requires `body_path`, and `readSkillById` returns `record: { ...found.record, ... }`, so the current `read_skill` output record includes `body_path` (`src/knowledge/types.ts:105-117`, `src/knowledge/lifecycle.ts:716-730`). Removing it is an MCP output-shape break, not just an internal schema break. Either preserve a `body_path` key in the returned record or change the analysis to explicitly document this as a deliberate `read_skill` output break and remove the byte-for-byte preservation claim.

2. In section 4.3, fix the proposed `QueryFilter` shape. The analysis says to build `{ in: { field: "path", values: pathsForIds } }`, but the public type is `{ in: Record<string, Array<string | number>> }`, so the shape would be `{ in: { path: pathsForIds } }` if the public API is used as-is (`src/rag/types.ts:114-124`). Keeping the current wording would be a public RAG API misuse.

3. In section 7.2, replace the boot divergence sweep's path enumeration mechanism. `manager.query(id, " ", { topK: 10000, filter: { pathGlob: "<kind>:*" } })` is a KNN query, not an authoritative vector-store scan; the sqlite-vec implementation fetches at most `topK` or `topK * POST_FILTER_OVERSHOOT` candidates and stops once `out.length >= topK` (`src/rag/store/sqlite-vec.ts:309-376`). The actual vector-store state currently has an internal `file_state` surface that enumerates stored paths and hashes (`src/rag/store/index.ts:37-41`, `src/rag/store/sqlite-vec.ts:432-460`). To satisfy the recovery requirement, the sweep must use actual store state, such as `Dataset.store.getFileState()` or an equivalent internal scan, not a broad semantic query plus `rag_sync`.

4. In sections 0.1, 6.1.b, and 10, correct the current `index.json` description and make the legacy-tree detection robust to both current shapes. The analysis says current `index.json` is `{ entries: IndexSummary[] }`, but fresh project initialization seeds `skills/project/index.json` as `{ skills: [] }` and `memory/project/index.json` as `{ memories: [], topic_map: {} }`; only `rebuildIndex` writes `{ entries }` later (`src/store/project.ts:154-171`, `src/knowledge/store.ts:309-330`). Since the no-migration legacy handling keys off legacy tree presence, the analysis must document this split or require the seeding code to be changed before relying on a single current shape.

5. In sections 0.2 and 10, correct the current skill search scoring summary. The analysis states `3*name + 2*triggers + 1*(description+body)`, but current `scoreSkillForSearch` gives `+3` when either name or trigger matches, `+2` for description, and `+1` for body snippet (`src/knowledge/loader.ts:117-135`). The memory scoring statement is correct (`src/knowledge/loader.ts:84-103`).

6. In sections 0.5 and 5.1, explicitly account for the current `update_memory` ACL enforcement gap before claiming ACL semantics are preserved. The permissions matrix marks coder/researcher `update-memory` as worker-stage restricted, and `checkScope` enforces that restriction for `Y-dagger` cells (`src/knowledge/permissions.ts:76-110`, `src/knowledge/permissions.ts:250-291`). The current `update_memory` handler calls `gateRole(role, "create")` but does not call `gateScope` before `updateMemory`, despite the comment saying it needs to fetch the record scope first (`src/mcp/knowledgeMemory.ts:216-228`). The analysis must say whether F01 preserves the actual current handler behavior or intentionally fixes it; if it fixes it, that is an ACL behavior change and should be documented.

7. In section 2 and section 6.1.d/e, include or explicitly default the required dataset `store` field when constructing protected dataset configs passed to RAG. `DatasetConfig` requires `store`, and `RagManagerOptions.datasets` is `ReadonlyArray<Omit<DatasetConfig, "projectId">>` (`src/rag/types.ts:39-45`, `src/rag/manager.ts:34-39`). The runtime config schema can default `store.kind`, but the in-memory entries pushed into `ragDatasets` before `manager.register(config)` must either include `store: { kind: "sqlite-vec" }` or be parsed through the config schema before use (`src/config.ts:217-267`).
