# Strict Review - F01 Skill/Memory on RAG Analysis R4

## Result

Changes requested. The document is not factually correct enough to approve. It mixes current-source observations with proposed F02 behavior that is not present in this workspace, and several claims would require changing or misusing the public RAG API in `src/rag/`.

## Blocking Findings

1. Section 2 uses a dataset `sources` sentinel that is not part of the public RAG API. `SourceRoot` is `{ root, include?, exclude? }`, and `DatasetConfig.sources` is `SourceRoot[]`, not `{ kind: "records" }[]` ([src/rag/types.ts](src/rag/types.ts#L28-L52)). The persisted config schema likewise requires source roots with `root` strings, while record ingest is modeled separately as `IngestInput.kind === "records"` ([src/config.ts](src/config.ts#L217-L269), [src/rag/types.ts](src/rag/types.ts#L143-L147)). Because the workspace rule says the public RAG API in `src/rag/` must not change, the analysis cannot introduce `sources: [{ kind: "records" }]` as a dataset config shape.

2. Sections 0.7, 2, and 4.3 misstate record-ingest metadata and make the proposed skill dataset infeasible as written. `ChunkMetadataInput` still includes `source`; it is not omitted by the type ([src/rag/types.ts](src/rag/types.ts#L60-L88)). However, the current record pipeline ignores `metadata.source` when building `metadataOverlay` and derives `source` as `"memory"` only when `metadataOverlay.scope === "memory"`; otherwise it calls `inferSource(path)`, which returns only `"doc"` or `"code"` ([src/rag/pipeline.ts](src/rag/pipeline.ts#L108-L122), [src/rag/pipeline.ts](src/rag/pipeline.ts#L137-L207)). A `knowledge.skills` ingest using `metadata.path = "skill:<id>"` and a query filter `eq("source", "skill")` would therefore not produce or find `source = "skill"` chunks under the current implementation.

3. Section 4.3 changes the public `RagManager.query` call shape. The actual API is `query(id: string, text: string, options?: QueryOptions): Promise<QueryHit[]>`, not `manager.query(id, { text, topK, filter })`, and the exported hit type is `QueryHit`, not `RagHit` ([src/rag/manager.ts](src/rag/manager.ts#L35-L48), [src/rag/query/pipeline.ts](src/rag/query/pipeline.ts#L18-L34)). This violates the explicit requirement that the public RAG API must not change.

4. Sections 0.4, 4.1, 4.2, and 10 do not preserve the current MCP read surface. Current `list_skills` accepts `scope`, `target_agent`, `include_archived`, and `include_superseded`; it has no `limit` or `after` input ([src/mcp/knowledgeSkills.ts](src/mcp/knowledgeSkills.ts#L110-L118), [src/mcp/knowledgeSkills.ts](src/mcp/knowledgeSkills.ts#L249-L257)). Current `list_memories` accepts `scope`, `topic_domain`, `include_archived`, and `older_than_days`; it also has no `limit` or `after` ([src/mcp/knowledgeMemory.ts](src/mcp/knowledgeMemory.ts#L103-L111), [src/mcp/knowledgeMemory.ts](src/mcp/knowledgeMemory.ts#L265-L273)). Current `get_memory` accepts either `id` or `topic`, not only `id` ([src/mcp/knowledgeMemory.ts](src/mcp/knowledgeMemory.ts#L116-L120), [src/mcp/knowledgeMemory.ts](src/mcp/knowledgeMemory.ts#L276-L283)). Current `read_skill` returns `{ record, body, redacted_spans }`, with the skill body read from `record.body_path`; it does not return a record JSON with inline `body` ([src/knowledge/lifecycle.ts](src/knowledge/lifecycle.ts#L716-L730)).

5. Sections 3.3, 4.1, and 9 understate schema fallout from deterministic built-in ids and inline skill bodies. The current knowledge schema requires `RecordBase.id` and `AuditEntry.record_id` to be UUIDs, and `SkillRecord` requires `body_path` ([src/knowledge/types.ts](src/knowledge/types.ts#L76-L114), [src/knowledge/types.ts](src/knowledge/types.ts#L154-L163)). Built-in skills currently synthesize UUIDs at load time ([src/knowledge/eagerLoader.ts](src/knowledge/eagerLoader.ts#L169-L212)). A `builtin:<name>` id and removal of `body_path` are not just loader tweaks; they require explicit schema and audit-contract changes.

6. Sections 0.1 and 0.3 mischaracterize the current JSON tree and eager-loading behavior. `index.json` is `{ entries: IndexSummary[] }`, not an `id -> relative path` map ([src/knowledge/store.ts](src/knowledge/store.ts#L309-L357)). Eager knowledge is not loaded once at process start by only checking `survive_compaction`; agent factories call `buildEagerBlock`, which calls `loadAllCandidates` and `resolveEagerRecords` per agent ([src/agents/planner.ts](src/agents/planner.ts#L30-L39), [src/agents/worker.ts](src/agents/worker.ts#L147-L160), [src/knowledge/eagerLoader.ts](src/knowledge/eagerLoader.ts#L236-L244), [src/knowledge/eagerLoader.ts](src/knowledge/eagerLoader.ts#L304-L313)). The resolver filters active records, target agents, skill triggers, memory opt-in via nonempty `target_agents`, and project-scoped survivor rules ([src/knowledge/loader.ts](src/knowledge/loader.ts#L233-L260), [src/knowledge/loader.ts](src/knowledge/loader.ts#L285-L300)). Survivors are also reinjected after compaction via `buildSurvivorBlock`, not merely held in a startup buffer ([src/agents/base.ts](src/agents/base.ts#L916-L954)).

7. Section 0.6 contains a wrong ACL summary. `data_agent` has skill read/list/search but no memory reads and no writes ([src/knowledge/permissions.ts](src/knowledge/permissions.ts#L112-L128)). The worker scope restriction applies to coder/researcher memory create/update cells, not to data_agent skill writes; `checkScope` enforces `scope === "stage"`, an active `ctx.stageId`, and `scope_ref === ctx.stageId` only for `Y-dagger` cells ([src/knowledge/permissions.ts](src/knowledge/permissions.ts#L84-L110), [src/knowledge/permissions.ts](src/knowledge/permissions.ts#L260-L290)). The handler error envelope is currently `UNAUTHORIZED_ROLE` / `UNAUTHORIZED_SCOPE`, not `KNOWLEDGE_PERMISSION_DENIED` ([src/mcp/knowledgeMemory.ts](src/mcp/knowledgeMemory.ts#L165-L180), [src/mcp/knowledgeMemory.ts](src/mcp/knowledgeMemory.ts#L298-L300)).

8. Sections 4.2, 4.3, and 5.2 do not capture current lifecycle/read semantics. `get_memory` can resolve by topic, searches session -> stage -> project, walks the supersession chain to the head, and returns null unless the head is active ([src/knowledge/lifecycle.ts](src/knowledge/lifecycle.ts#L216-L229), [src/knowledge/lifecycle.ts](src/knowledge/lifecycle.ts#L733-L760)). `listSkills` defaults out archived and superseded records but can include them via flags; `listMemories` defaults out archived and always filters superseded, plus supports `older_than_days` ([src/knowledge/lifecycle.ts](src/knowledge/lifecycle.ts#L766-L833)). `searchSkills` and `searchMemories` search only active records, use 500-character body snippets, and return 200-character windows around the match ([src/knowledge/lifecycle.ts](src/knowledge/lifecycle.ts#L838-L907)). Stage/session archival physically moves active records into an `archive/` subtree and deletes them from the live `records` directory, which is separate from merely setting `status` ([src/knowledge/lifecycle.ts](src/knowledge/lifecycle.ts#L917-L1017)).

9. Sections 5.1 and 7 describe write and recovery behavior that would change handler results and is not reconciled with existing contracts. Current write helpers return narrow shapes such as `{ id, status }`, `{ id, updated_at }`, or `{ new_id, old_id }` ([src/knowledge/lifecycle.ts](src/knowledge/lifecycle.ts#L247-L315), [src/knowledge/lifecycle.ts](src/knowledge/lifecycle.ts#L330-L399), [src/knowledge/lifecycle.ts](src/knowledge/lifecycle.ts#L512-L557), [src/knowledge/lifecycle.ts](src/knowledge/lifecycle.ts#L572-L630)). Adding `details.ragWarning` is a public knowledge-tool response change, despite the document repeatedly saying handler shape is invariant. Also, the proposed `rag_sync` sidecar detects divergence only from its own remembered hash state; it does not, by itself, prove the vector store actually still contains matching chunks. The existing vector store has its own chunk and `file_state` tables ([src/rag/store/sqlite-vec.ts](src/rag/store/sqlite-vec.ts#L151-L193), [src/rag/store/sqlite-vec.ts](src/rag/store/sqlite-vec.ts#L420-L461)), so the recovery design needs an explicit source-of-truth comparison against the RAG store or a statement that boot recovery is based on sidecar intent only.

10. Sections 4.4, 6, 7, and 9 rely on non-existent current services and error names. There is no bootstrap wiring for `RagManager`; bootstrap loads config and registers built-in services directly ([src/server/bootstrap.ts](src/server/bootstrap.ts#L8-L15), [src/server/bootstrap.ts](src/server/bootstrap.ts#L133-L151)). `registerBuiltinServices` has no `knowledge` option and currently registers the `skills` and `memory` handlers directly ([src/mcp/builtins.ts](src/mcp/builtins.ts#L1912-L1966)). `config.ts` exports `loadConfig` but no `saveSaivageConfig` or config writer ([src/config.ts](src/config.ts#L1-L3), [src/config.ts](src/config.ts#L335-L343)). The current RAG error classes are `DatasetNotFoundError`, `ProviderUnavailableError`, `EmbeddingDriftError`, `ConfigDriftError`, etc.; there are no `RAG_DISABLED`, `RAG_PROVIDER_UNAVAILABLE`, or `RAG_EMBEDDING_DRIFT` error codes in the source ([src/rag/errors.ts](src/rag/errors.ts#L3-L116)). When RAG is disabled, `createRagManager` returns a no-op manager whose methods throw `DatasetNotFoundError`, not `RAG_DISABLED` ([src/rag/manager.ts](src/rag/manager.ts#L54-L88)).

11. Section 8's tool-filter delta is directionally plausible but not complete enough for the stated behavior. Current `READ_ONLY_TOOLS` includes only filesystem/git read tools plus `list_skills` and `read_skill` ([src/agents/tool-filters.ts](src/agents/tool-filters.ts#L12-L15)). Because worker roles use the deny-list filter, many workers can already name the memory tools before handler ACLs reject or allow them ([src/agents/tool-filters.ts](src/agents/tool-filters.ts#L22-L40), [src/agents/roster.ts](src/agents/roster.ts#L74-L177)). The analysis needs to distinguish planner/reviewer/inspector/chat read-only exposure from worker deny-list exposure and state the actual handler error codes.

12. Section 11's one-shot legacy-tree deletion is not integrated with the current init path and misses existing project seeding. Project initialization currently seeds `.saivage/skills/project` and `.saivage/memory/project` with empty `index.json` and `audit.jsonl` files ([src/store/project.ts](src/store/project.ts#L145-L174)). The backout/install section must specify exactly where deletion happens, how it avoids destroying freshly seeded state before sidecar init succeeds, and how it handles existing built-in skill loading without a read-side compatibility shim.

## Completeness Gaps Against Existing Knowledge Tools

The analysis does not yet cover every behavior the JSON-tree tools provide today. It needs to preserve or deliberately replace: read-time redaction on skills and memories ([src/knowledge/lifecycle.ts](src/knowledge/lifecycle.ts#L716-L730), [src/knowledge/lifecycle.ts](src/knowledge/lifecycle.ts#L753-L760)); secret and blocked-path guards before writes ([src/knowledge/store.ts](src/knowledge/store.ts#L86-L155), [src/knowledge/lifecycle.ts](src/knowledge/lifecycle.ts#L273-L312), [src/knowledge/lifecycle.ts](src/knowledge/lifecycle.ts#L518-L557)); runtime-lock enforcement before mutations ([src/knowledge/lifecycle.ts](src/knowledge/lifecycle.ts#L45-L58), [src/knowledge/lifecycle.ts](src/knowledge/lifecycle.ts#L247-L251)); name/topic collision scope rules ([src/knowledge/lifecycle.ts](src/knowledge/lifecycle.ts#L273-L280), [src/knowledge/lifecycle.ts](src/knowledge/lifecycle.ts#L522-L528)); supersession scope-pair rules ([src/knowledge/lifecycle.ts](src/knowledge/lifecycle.ts#L231-L238), [src/knowledge/lifecycle.ts](src/knowledge/lifecycle.ts#L414-L481), [src/knowledge/lifecycle.ts](src/knowledge/lifecycle.ts#L647-L702)); and stage/session archival hooks ([src/knowledge/lifecycle.ts](src/knowledge/lifecycle.ts#L917-L1017)).

## Required Changes

1. In sections 0.4, 4.1, 4.2, 4.3, and 10, replace the claimed MCP read schemas and preserved response shapes with the actual schemas and results from `knowledgeSkills.ts`, `knowledgeMemory.ts`, and `lifecycle.ts`; explicitly include `target_agent`, `include_archived`, `include_superseded`, `topic_domain`, `older_than_days`, topic-based `get_memory`, redaction fields, and current sorting/filtering behavior.

2. In sections 0.7, 2, and 4.3, correct the RAG API description so it uses the existing `DatasetConfig`, `SourceRoot`, `IngestInput.records`, and `RagManager.query(id, text, options)` signatures without introducing `sources: [{ kind: "records" }]`, object-shaped query calls, or a `RagHit` type.

3. In sections 0.7, 2, 3.2, and 4.3, resolve the record-ingest source-tagging problem. The analysis must acknowledge that the current pipeline ignores `metadata.source` and cannot emit `source = "skill"` for records as written, then propose an implementation that preserves the public RAG type surface while making `skill` and `memory` chunks queryable.

4. In sections 0.1, 0.3, 3.3, 4.1, 9, and 10, correct the JSON-tree, built-in id, `index.json`, `body_path`, and schema claims. If deterministic `builtin:<name>` ids and inline skill bodies remain in scope, the analysis must explicitly call out the required changes to `RecordBaseSchema`, `AuditEntrySchema`, `SkillRecordSchema`, built-in loading, and existing read result contracts.

5. In sections 0.6 and 8, rewrite the ACL/tool-filter discussion using the actual matrix and filters: data_agent has no memory access and no writes; coder/researcher have stage-scoped memory create/update only; handler failures currently use `UNAUTHORIZED_ROLE` and `UNAUTHORIZED_SCOPE`; worker deny-list exposure differs from read-only exposure.

6. In sections 0.3, 5.2, 6.1, and 7, add a complete account of current eager loading and lifecycle behavior: per-agent eager block construction, target-agent filtering, trigger scoring, memory opt-in, project-scoped survivor reinjection after compaction, active-only search, list/archive/supersede differences, and stage/session archival moving records into `archive/` subtrees.

7. In sections 5.1, 7.1, and 7.2, reconcile RAG reingest failure handling with the existing knowledge-tool response contracts. If `details.ragWarning` remains, identify it as an intentional handler-surface change; otherwise define a response-preserving error/reporting path. Also make divergence detection verify actual vector-store state, not only a sidecar `rag_sync` table.

8. In sections 4.4, 6, 7, and 9, remove or clearly mark unsupported current-source claims about F02 services, `saveSaivageConfig`, protected `rag_register`/`rag_ingest`/`rag_drop`/`rag_admin` tools, `registerBuiltinServices(..., { knowledge })`, and `RAG_*` error-code names. Any dependency on future F02 work must be separated from factual current-state analysis.

9. In section 11, replace the legacy-tree deletion/backout plan with an implementable init sequence tied to the current project seeding path. It must describe when deletion occurs relative to successful sidecar initialization and protected dataset registration, and how it avoids migration shims while not leaving the runtime unable to load built-in or project knowledge.

10. Re-run the internal-consistency pass after the above fixes. The document currently says handler shape is invariant while also dropping `SkillRecord.body_path`, changing read/write response bodies, inventing RAG config shapes, and depending on services absent from bootstrap; those contradictions must be removed before approval.

VERDICT: CHANGES_REQUESTED
