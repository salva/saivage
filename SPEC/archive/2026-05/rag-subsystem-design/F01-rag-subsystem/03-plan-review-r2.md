## Per-axis findings

1. Scope and non-goals: aligned. The plan implements the focused in-process library under [03-plan-r2.md](saivage/SPEC/2026-05/rag-subsystem-design/F01-rag-subsystem/03-plan-r2.md#L5), with one sqlite-vec store, one OpenAI `text-embedding-3-small` provider, three chunkers, opt-in `rag.enabled`, and no disk/native load when disabled. That matches the approved Proposal A shape of one sqlite-vec store, one OpenAI provider, and a small chunker family [02-design-r2.md](saivage/SPEC/2026-05/rag-subsystem-design/F01-rag-subsystem/02-design-r2.md#L20), and the recommendation to implement Proposal A at [saivage/src/rag/](saivage/src/rag/) [02-design-r2.md](saivage/SPEC/2026-05/rag-subsystem-design/F01-rag-subsystem/02-design-r2.md#L566). Consumer integrations, local embeddings, MCP, reranking, hybrid search, cross-dataset query, background ingest, UI, and quantization are explicitly non-goals [03-plan-r2.md](saivage/SPEC/2026-05/rag-subsystem-design/F01-rag-subsystem/03-plan-r2.md#L9-L20), which satisfies the topic's requirement to stop short of consumer integration [F01-rag-subsystem.md](saivage/SPEC/2026-05/rag-subsystem-design/F01-rag-subsystem.md#L48-L50).

2. Batch ordering and completeness: aligned. The batch table is foundational-first: package/dependencies, public types/config, store, provider, chunkers, secrets, ingest, query, manager, e2e, then SPEC notes [03-plan-r2.md](saivage/SPEC/2026-05/rag-subsystem-design/F01-rag-subsystem/03-plan-r2.md#L43-L53). This satisfies the topic's ordering requirement that types, store abstraction, and config precede dataset-specific work [F01-rag-subsystem.md](saivage/SPEC/2026-05/rag-subsystem-design/F01-rag-subsystem.md#L48-L48). Each batch has files, validation tags, dependencies, rollback, and approximate tests, so an implementer has an executable sequence rather than a loose roadmap [03-plan-r2.md](saivage/SPEC/2026-05/rag-subsystem-design/F01-rag-subsystem/03-plan-r2.md#L57-L118), [03-plan-r2.md](saivage/SPEC/2026-05/rag-subsystem-design/F01-rag-subsystem/03-plan-r2.md#L285-L347).

3. Files outside [saivage/src/rag/](saivage/src/rag/): aligned for implementation-bearing changes. The plan limits non-rag source/config modifications to [saivage/package.json](saivage/package.json) and [saivage/src/config.ts](saivage/src/config.ts), and explicitly says [saivage/tsconfig.json](saivage/tsconfig.json), [saivage/eslint.config.js](saivage/eslint.config.js), and [saivage/vitest.config.ts](saivage/vitest.config.ts) require a new design decision instead of a midstream edit [03-plan-r2.md](saivage/SPEC/2026-05/rag-subsystem-design/F01-rag-subsystem/03-plan-r2.md#L5-L5). The batch summary reflects that B01 modifies only [saivage/package.json](saivage/package.json), B02 modifies only [saivage/src/config.ts](saivage/src/config.ts), and all later runtime/source batches have `none` in the modified-files column [03-plan-r2.md](saivage/SPEC/2026-05/rag-subsystem-design/F01-rag-subsystem/03-plan-r2.md#L43-L53). The e2e and SPEC additions are explicit planned test/docs batches, not hidden config edits [03-plan-r2.md](saivage/SPEC/2026-05/rag-subsystem-design/F01-rag-subsystem/03-plan-r2.md#L52-L53), [03-plan-r2.md](saivage/SPEC/2026-05/rag-subsystem-design/F01-rag-subsystem/03-plan-r2.md#L306-L345).

4. Validation commands: aligned. The validation tags map to concrete commands: `npm run typecheck`, `npm run lint`, `npx vitest run src/rag`, `npx vitest run tests/rag`, `npm test`, and a B10 manual smoke [03-plan-r2.md](saivage/SPEC/2026-05/rag-subsystem-design/F01-rag-subsystem/03-plan-r2.md#L25-L31), satisfying the topic's exact-validation requirement [F01-rag-subsystem.md](saivage/SPEC/2026-05/rag-subsystem-design/F01-rag-subsystem.md#L52-L52). The existing package already exposes `test`, `lint`, and `typecheck` scripts [saivage/package.json](saivage/package.json#L15-L23). B11 now has a concrete `T` + `L` validation row in both the batch table and detail section [03-plan-r2.md](saivage/SPEC/2026-05/rag-subsystem-design/F01-rag-subsystem/03-plan-r2.md#L53-L53), [03-plan-r2.md](saivage/SPEC/2026-05/rag-subsystem-design/F01-rag-subsystem/03-plan-r2.md#L345-L345).

5. Rollback strategy: aligned. The rollback model requires one commit per batch, standard `git revert`, opt-in runtime behavior through `rag.enabled`, no disk artifact when disabled, and operator rollback by removing `.saivage/rag/` plus the config slice [03-plan-r2.md](saivage/SPEC/2026-05/rag-subsystem-design/F01-rag-subsystem/03-plan-r2.md#L37-L37). This matches the approved design's lifecycle rule that removing the slice or setting `enabled: false` prevents manager construction and leaves on-disk artifacts operator-removable [02-design-r2.md](saivage/SPEC/2026-05/rag-subsystem-design/F01-rag-subsystem/02-design-r2.md#L774-L774), and it satisfies the topic's rollback requirement [F01-rag-subsystem.md](saivage/SPEC/2026-05/rag-subsystem-design/F01-rag-subsystem.md#L53-L53).

6. Acceptance criteria: aligned. The final criteria require typecheck, lint, full Vitest, isolated `src/rag` tests, isolated `tests/rag` tests, real OpenAI smoke ingest, drift refusal, disabled-mode inertness with no `.saivage/rag/` artifact or native-dep load, and secret-exclusion coverage [03-plan-r2.md](saivage/SPEC/2026-05/rag-subsystem-design/F01-rag-subsystem/03-plan-r2.md#L380-L390). Those criteria cover the approved design's primary failure responses for provider outages, embedding drift, corruption, ingest locking, and secret dropping [02-design-r2.md](saivage/SPEC/2026-05/rag-subsystem-design/F01-rag-subsystem/02-design-r2.md#L333-L339).

7. Architecture and complexity: aligned. The plan removes the alternate-vector-store implementation path from the implementation sequence: B01 halts for a new design decision on native-package failure and says not to substitute another vector store [03-plan-r2.md](saivage/SPEC/2026-05/rag-subsystem-design/F01-rag-subsystem/03-plan-r2.md#L111-L111), and the open-question checklist repeats that `sqlite-vec` failure must halt rather than introduce a source-build fallback or alternate store [03-plan-r2.md](saivage/SPEC/2026-05/rag-subsystem-design/F01-rag-subsystem/03-plan-r2.md#L372-L372). The plan pins the design-shaping details inline: deterministic `releaseFingerprint` derivation in B04 [03-plan-r2.md](saivage/SPEC/2026-05/rag-subsystem-design/F01-rag-subsystem/03-plan-r2.md#L195-L195), OpenAI provider batching at 96 inputs plus tests for that cap [03-plan-r2.md](saivage/SPEC/2026-05/rag-subsystem-design/F01-rag-subsystem/03-plan-r2.md#L193-L196), the per-input token cap at 7500 under the OpenAI 8191-token limit in B05 [03-plan-r2.md](saivage/SPEC/2026-05/rag-subsystem-design/F01-rag-subsystem/03-plan-r2.md#L219-L219), and the `<datasetId>.rebuild/` staging layout in B09 [03-plan-r2.md](saivage/SPEC/2026-05/rag-subsystem-design/F01-rag-subsystem/03-plan-r2.md#L292-L294). That is consistent with the analysis requirement to stamp provider/model/release fingerprint and refuse mixed spaces [01-analysis-r4.md](saivage/SPEC/2026-05/rag-subsystem-design/F01-rag-subsystem/01-analysis-r4.md#L268-L268), the OpenAI max-input analysis [01-analysis-r4.md](saivage/SPEC/2026-05/rag-subsystem-design/F01-rag-subsystem/01-analysis-r4.md#L172-L176), and the approved design's drift-refusal requirement [02-design-r2.md](saivage/SPEC/2026-05/rag-subsystem-design/F01-rag-subsystem/02-design-r2.md#L335-L335).
